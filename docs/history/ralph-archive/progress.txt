  - Service-level ON DELETE SET NULL pattern: SQLite cannot ALTER ADD FK so we
    cannot rely on cascade-set-null. Pattern: in delete_goal, run
    `UPDATE pomodoro_sessions SET goal_id=NULL WHERE goal_id=:gid` BEFORE the
    DELETE on the parent row. Service layer is responsible for referential
    integrity; the FK column is a plain nullable index.
  - Replacing a model in-place when the table name stays the same: keep the
    SQLAlchemy class declaration as the new schema (so create_all generates
    the right CREATE TABLE for fresh DBs) AND add idempotent ALTER TABLE
    statements in db_init.py for existing DBs that have the old schema.
    SQLite is permissive about NOT NULL on ALTER ADD when no rows exist.
    Old columns (e.g. document_id NOT NULL, target_date) remain in legacy DBs
    but are tolerated by the new model as long as it declares them nullable.
  - When migrating a table whose old schema is structurally incompatible
    (different NOT NULL columns), the cleanest dev path is: stop the backend,
    drop the table (verify 0 rows first), let the new model.create_all
    rebuild it. Idempotent ALTER ADD on its own does NOT remove a NOT NULL
    constraint, so writes from the new model will still hit "no column X" or
    "NOT NULL violation" errors against a legacy table.
  - Long-running uvicorn --reload backends accumulate stale prepared statements
    in their connection pool when the schema is mutated outside the process.
    After a manual DROP/CREATE on a watched table, the worker process must be
    restarted (kill + relaunch); --reload's file-watcher does not by itself
    invalidate the SQLAlchemy pool's cached statement metadata.
  - When deleting a test file because its imports are gone, also do it BEFORE
    running pytest -- otherwise collection errors mask the rest of the suite.
  - For type-dispatched compute methods (e.g. compute_progress switching on
    goal_type), keep the dispatcher trivial: a series of `if goal.goal_type
    == 'X': return await self._progress_X(goal)` lines + a final
    InvalidGoalType raise. Keeps each branch independently testable.
  - `WHERE created_at >= :start AND created_at <= :end` window queries on
    SQLite need _as_utc() normalization on both endpoints because aiosqlite
    returns DateTime columns tz-naive. The pomodoro started_at/completed_at
    pulled from the row are also tz-naive after a refresh; coerce them with
    `_as_utc()` before comparing.

    would need to have read the books -- i.e. a frontier model. That
    violates I-16 (local-first). The practical upper bound of HyDE on
    open-source small LLMs is ~+15pp; insufficient for HR@5=0.60 on
    datasets where the baseline is 0.25-0.40.
  - Cleaner architectural fit for the question-answer-divergence
    failure mode: cross-encoder reranking on top-50 RRF candidates
    (e.g. BAAI/bge-reranker-v2-m3). No LLM-knowledge dependency; ranks
    chunks by direct query-chunk semantic similarity. This should be
    promoted into its own AC-bearing follow-up story.
  - HyDE's lazy-import-of-llm pattern (`from app.services.llm import
    get_llm_service` inside the helper, with `# noqa: PLC0415`)
    avoids the circular import retriever <-> services that would
    happen if llm was imported at module level. Patch target for tests
    is `app.services.llm.get_llm_service` (where the symbol lives at
    import time), not the call-site module.
  - llama3.2:3b cold call ~3s, warm ~1s. mistral:7b ~2-3s. gemma4:8b
    sometimes returns empty completions (model-specific quirk; not
    investigated -- llama3.2:3b is the right default).
  - When the LLM is unavailable, HyDE must fail-soft. The helper
    catches all `Exception` and returns the original query so
    retrieval is at worst the no-hyde baseline. Without this, an
    Ollama outage would break /search entirely.
