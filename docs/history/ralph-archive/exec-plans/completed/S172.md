# Exec Plan: S172 -- Note nodes in Viz Knowledge and Learning Path graph modes

**Phase:** V3 Phase 2 - Note Intelligence and Synthesis
**Priority:** 12
**Status:** in-progress

## Goal

Add Note nodes to the Viz tab's Knowledge and Learning Path graph modes. The backend extends `get_graph_for_document` and `get_graph_for_documents` with an `include_notes` flag that queries Kuzu for Note nodes connected via WRITTEN_ABOUT or TAG_IS_CONCEPT edges to entities already in scope, plus LINKS_TO edges between notes. The frontend adds a "Notes" layer toggle to the Viz sidebar; note nodes render as indigo squares (custom WebGL program, similar to the existing hexagon program), sized by outgoing link count. Clicking a note node opens a `NotePreviewPanel` on the right side with the note's full content, tags, collections, and an "Open in Notes tab" button that fires the `luminary:navigate` event.

## Implementation Notes

### Backend: note query strategy

`get_note_nodes_for_entities` runs two Kuzu queries:

1. Fetch notes connected to entities in scope:
   ```
   MATCH (n:Note)-[:WRITTEN_ABOUT|:TAG_IS_CONCEPT]->(e:Entity)
   WHERE e.id IN [<entity_ids>]
   RETURN DISTINCT n.note_id, n.preview,
          count{(n)-[:WRITTEN_ABOUT|:TAG_IS_CONCEPT]->(:Entity)} AS outgoing_entity_count
   ```
   Kuzu does not support `IN` on a list parameter with a variable-length list in all versions; use the same `$eid0, $eid1, ...` placeholder pattern already used in `_get_co_occurrence_edges`.

2. Fetch LINKS_TO edges between the collected note_ids (only those present in scope):
   ```
   MATCH (n1:Note)-[l:LINKS_TO]->(n2:Note)
   WHERE n1.note_id IN [<note_ids>] AND n2.note_id IN [<note_ids>]
   RETURN n1.note_id, n2.note_id, l.link_type
   ```

The `outgoing_link_count` for node sizing uses the count of WRITTEN_ABOUT + TAG_IS_CONCEPT + LINKS_TO outgoing edges. Since Kuzu aggregate subquery syntax varies by version, compute it in Python from the edge sets already collected rather than inside the Cypher query.

Guard all `has_next()` calls before `get_next()` per I-3. Wrap entire method in try/except returning `([], [])` on Kuzu error (non-fatal, matches existing pattern).

### Frontend: square SDF shader

Copy `sigma-hexagon.ts` to `sigma-square.ts`. Replace the hexagon SDF fragment with a square SDF:
```glsl
float d = max(abs(p.x), abs(p.y)) - v_radius;
```
This renders axis-aligned squares. Register as `nodeProgramClasses: { square: NodeSquareProgram }` alongside the existing `hexagon` entry.

### Frontend: note node type guard

The existing `filteredGraph` memo in `Viz.tsx` filters `visibleNodes` by `activeTypes`. Note nodes have `type='note'` which is not in `ALL_ENTITY_TYPES`. Notes are controlled separately by `showNotes` toggle. The filter logic must be extended:

```typescript
const visibleNodes = data.nodes.filter((n) => {
  if (n.type === 'note') return showNotes
  if (DIAGRAM_NODE_TYPES.has(n.type)) return showDiagramNodes && activeTypes.has(n.type as EntityType)
  return activeTypes.has(n.type as EntityType)
})
```

The `showAllHidden` and `showEmpty` booleans are computed from `data.nodes` and `filteredGraph`. They remain correct because note nodes are not in `ALL_ENTITY_TYPES`, so they do not count toward the entity-type filter.

### Frontend: NotePreviewPanel positioning

The existing `selectedNode` popover is `position: fixed` at the click coordinates. `NotePreviewPanel` should be anchored to the right side of the graph canvas (`position: absolute`, `right: 0`, `top: 0`, `bottom: 0`, `width: 300px`), consistent with the story description ("right side panel"). This avoids the coordinate-calculation complexity of the floating popover pattern.

Use a separate state `selectedNoteId: string | null` (independent of `selectedNode`) so that clicking a note node sets `selectedNoteId` and clears `selectedNode`, and clicking an entity node sets `selectedNode` and clears `selectedNoteId`.

### Kuzu thread-safety

`get_note_nodes_for_entities` calls `_conn.execute()`. Per the existing `_lock` pattern documented in `patterns.md`, all `_conn.execute()` calls must be serialised via `self._lock`. The existing `get_graph_for_document` does NOT currently hold the lock (confirmed by reading the source). This is an existing gap in the codebase; do not introduce a regression by inconsistently locking only the new method. Acquire `self._lock` in the new helper and add a comment to track this.
