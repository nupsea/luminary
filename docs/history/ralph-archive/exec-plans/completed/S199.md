# Exec Plan: S199 -- Naming conventions: UPPER-CASE collections and lower-case tags with dash separators

**Phase:** V3 Phase 5 - Reading Experience and Chat Intelligence
**Priority:** 39
**Status:** in-progress

## Goal

Tags and collections currently accept any casing and separators, producing duplicates like 'Python', 'python', 'PYTHON' and collections named 'my notes', 'My Notes', 'MY_NOTES' as distinct entries. This story enforces a strict naming convention: collection names are UPPER-CASE with hyphen separators (e.g. 'MY-NOTES'), and tag slugs are lower-case with hyphen separators preserving '/' hierarchy (e.g. 'science/biology'). Normalization is applied at creation time (API endpoints and auto-generation) and back-filled via two migration endpoints that merge duplicates that collapse to the same normalized form. Frontend previews show the normalized form as the user types, so they see the final slug before committing.

## Cross-Story Risk

- **I-1 (No shared AsyncSession across asyncio.gather):** The migration endpoints do not use `asyncio.gather`. All work is done sequentially in a single session. Safe.
- **I-4 (FTS5 UNINDEXED column equality filtering):** `_sync_tag_index` uses `NoteTagIndexModel.tag_full == tag` for DELETE and load. `note_tag_index` is a regular table (not FTS5), so this is safe. The FTS5 invariant applies only to `notes_fts` and `chunks_fts` virtual tables.
- **I-5 (No module-level circular imports):** `app.services.naming` imports only `re` (stdlib). Importing it at module level in `collections.py`, `notes.py`, and `tags.py` introduces no circular dependency. No lazy import needed.
- **I-13 (Quality gates in order):** ruff -> pytest -> tsc. Fix ruff before running pytest.
- **S192 (DocumentReader notes panel / auto-collection):** `_make_collection_name` is called in `POST /collections/auto/{document_id}`. After this story, auto-collection names will switch from underscore-joined (e.g. 'DDIA_Book') to hyphen-joined UPPER-CASE (e.g. 'DDIA-BOOK'). Any existing auto-collections in user data keep their old names until `POST /collections/migrate-naming` is run. The migration endpoint handles back-fill. No regression to the S192 feature itself.
- **S162 / tag merge (tags.py `merge_tags`):** The merge endpoint (`POST /tags/merge`) calls `_sync_tag_index` with the merged tag list. After this story, `_sync_tag_index` normalizes slugs before storing. This is strictly additive -- merged tags will now be stored in normalized form. If both source and target happen to normalize to the same slug (edge case), the deduplication logic in `_sync_tag_index` already handles it via set operations.
- **S190 (TagManagement search):** `filterTagTree` in `tagUtils.ts` does substring matching on `tag.id`. After this story all stored ids are lowercase-hyphenated, so user searches in mixed case will still work via `.toLowerCase().includes(q)`. No regression.
