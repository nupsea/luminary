# Exec Plan: S197 -- Compare notes with book: auto-collection gap analysis

**Phase:** V3 Phase 5 - Reading Experience and Chat Intelligence
**Priority:** 37
**Status:** in-progress

## Goal

This story removes all manual note-selection friction from "Compare my notes with this book" gap analysis by wiring the existing `notes_gap_node` in `chat_graph.py` to automatically fetch notes from the document's auto-collection (introduced in S192) when no explicit note IDs are provided. It also adds a "Compare my notes" toolbar button to DocumentReader, a "Quick compare" default mode to GapDetectDialog (with an "Advanced" toggle for manual selection), and a per-gap "Take a note on this" action in GapResultCard that opens NoteEditorDialog pre-filled with the concept heading and auto-collection pre-assigned.

## Files to Touch

### Backend
- `backend/app/runtime/chat_graph.py` -- `notes_gap_node` function (lines 954-1048): replace the current `NoteModel.document_id == document_id` query with a two-step query: (1) find `NoteCollectionModel WHERE auto_document_id = document_id` to get the auto-collection, then (2) fetch `NoteCollectionMemberModel.note_id WHERE collection_id = auto_collection.id`; add count-based card messages for 0 notes and < 3 notes cases; lazy-import `NoteCollectionModel` and `NoteCollectionMemberModel` alongside existing `NoteModel` import
- `backend/tests/test_chat_graph_gap.py` -- new test cases: auto-fetches from auto-collection (success path), returns card message when auto-collection has 0 notes, returns card message when auto-collection has fewer than 3 notes

### Frontend
- `frontend/src/components/reader/DocumentReader.tsx` -- add "Compare my notes" button in the back-button header row (line ~2178-2186); button visible only when `noteCount >= 3`; uses `setChatPreload` + dispatches `luminary:navigate` with `tab: "chat"` after setting the pending message; reads `noteCount` from the existing `handleNoteCountKnown` callback already wired to `NotesReaderPanel`
- `frontend/src/components/GapDetectDialog.tsx` -- add `mode` state (`"quick" | "advanced"`); in "quick" mode, when a document is selected, auto-fetch the auto-collection via `GET /collections/by-document/{docId}` and then `GET /notes?collection_id={collId}` to populate `selectedNoteIds` automatically; add "Advanced" toggle button that reveals the manual note search and selection UI (existing behavior); keep `canAnalyze` guard unchanged
- `frontend/src/components/GapResultCard.tsx` -- add "Take a note on this" action button per gap item in the gaps list; clicking it dispatches `luminary:navigate` to `tab: "notes"` and opens NoteEditorDialog pre-filled with the concept as heading and the document's auto-collection pre-assigned; the action can be modeled as a `window.dispatchEvent(new CustomEvent("luminary:navigate", { detail: { tab: "notes", prefilledContent: "# <gap>", collectionId: <autoCollectionId> } }))` -- the Notes tab's luminary:navigate handler will need to support this detail shape

### Other
- `scripts/smoke/S197.sh` -- smoke test verifying: (1) `GET /collections/by-document/{doc_id}` returns 404 when no auto-collection exists; (2) `POST /collections/auto/{doc_id}` creates an auto-collection; (3) subsequent `GET /collections/by-document/{doc_id}` returns the auto-collection with note count; (4) the Chat SSE endpoint with intent `notes_gap` and a scoped document returns a `__card__` response

## Cross-Story Risk

- **S192 (DocumentReader notes panel -- auto-collections with `auto_document_id`):** This story depends directly on S192's `NoteCollectionModel.auto_document_id` field and the `GET /collections/by-document/{document_id}` endpoint. If S192 is not complete (it has `passes=true`), this story can proceed.
- **I-1 (Never share AsyncSession across asyncio.gather):** The modified `notes_gap_node` uses a single session for two sequential queries (collection lookup then member lookup). These are sequential within one `async with session`, not gathered -- I-1 is not at risk.
- **I-3 (Guard Kuzu `get_next()` with `has_next()`):** Not touched by this story.
- **I-5 (Lazy imports for circular dependencies):** `NoteCollectionModel` and `NoteCollectionMemberModel` are imported inside the `notes_gap_node` try block (already the established pattern in this function); this satisfies I-5.
- **I-10 (Loading, error, empty states for every frontend feature):** `GapDetectDialog` quick mode must show loading state while fetching auto-collection notes, error state if the fetch fails, and a "No notes yet" message if auto-collection has 0 notes.
- **I-11 (Cross-tab navigation via `luminary:navigate`):** The "Compare my notes" button in DocumentReader and the "Take a note on this" button in GapResultCard both dispatch `luminary:navigate` events per I-11. App.tsx's `onLuminaryNavigate` handler must handle the `prefilledContent` + `collectionId` detail fields added for the "Take a note on this" action.
- **test_chat_graph_gap.py mock pattern:** The existing `_mock_get_session_factory` returns one fixed `fetchall` result. The new auto-collection path needs two sequential `execute` calls with different results. Use `mock_session.execute = AsyncMock(side_effect=[collection_result, member_result])` pattern (see patterns.md "Two-pass LLM call mocking" -- same `side_effect` list pattern applies to AsyncMock execute calls).
