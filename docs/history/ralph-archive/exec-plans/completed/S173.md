# Exec Plan: S173 -- Collection health report: cohesion score, orphaned notes, coverage gaps, stale notes

**Phase:** V3 Phase 2 - Note Intelligence and Synthesis
**Priority:** 13
**Status:** in-progress

## Goal

Add a health dashboard for note collections that mirrors the deck health report (S160) but targets
note organization. Given a collection ID the backend computes: cohesion score (mean pairwise cosine
similarity of note vectors via LanceDB), orphaned notes (library-wide notes in no collection),
uncovered notes (collection notes with no flashcard coverage), stale notes (not edited in 90+ days),
and hotspot tags (top 5 tags). The frontend adds a gear icon to CollectionTree items that opens a
CollectionHealthPanel Sheet with all four sections and action buttons. The 'Generate Flashcards'
action pre-populates GenerateFlashcardsDialog with the uncovered note IDs. A new
NoteModel.archived column (bool, default False) lets users archive stale notes; GET /notes excludes
archived notes by default.

## Implementation Notes

### cohesion_score algorithm
Fetch all note vectors for the collection from `note_vectors_v2` LanceDB table using
`asyncio.to_thread`. Compute N x N pairwise cosine similarity (numpy dot product after L2
normalization). Return the mean of the upper triangle (excluding diagonal). With N < 6 return None.
Guard: if the LanceDB table has no rows for a given note_id (vector not yet computed), skip that
note from the set; if remaining < 6, still return None.

### uncovered_notes query
```python
# Notes in collection
member_note_ids: list[str] = (SELECT note_id FROM note_collection_members WHERE collection_id = :id)
# Covered note IDs: have a flashcard with source='note' and deck=collection.name
covered_ids: set[str] = (
    SELECT DISTINCT source_note_id FROM flashcards
    WHERE source='note' AND deck=:collection_name
    -- source_note_id is a column on FlashcardModel linking back to the originating note
)
```
Note: FlashcardModel does not have a direct `source_note_id` FK column as of the code read.
The deck=collection.name approach is the specified mechanism. Query: `SELECT id FROM flashcards WHERE source='note' AND deck=:name` -- collect these note IDs by cross-referencing the content or use source_excerpt. Actually the story says "whose id has no FlashcardModel row with source='note' and deck=collection.name". This implies the note_id is encoded in the flashcard row. Looking at FlashcardModel: there is no explicit note_id FK. The deck field is set to collection.name by FlashcardService when generating from a collection (S169). Therefore the correct query is: notes in collection whose id does NOT appear as a source_note_id -- but there is no source_note_id column. Use a coarser check: note is in this collection AND there exists no flashcard with source='note' AND deck=collection.name AND the note content appears in source_excerpt (too fragile). The simplest safe interpretation matching the story text: a note is "covered" if ANY flashcard exists with source='note' and deck=collection.name -- this is a collection-level coverage signal, not per-note. Reread the AC: "notes in this collection whose id has no FlashcardModel row with source='note' and deck=collection.name". This implies FlashcardModel should have a note_id or document_id column that can be matched back. FlashcardModel has document_id. For note-sourced cards document_id may not match. The story says to use this approach -- implement it as: add `note_id` FK to FlashcardModel as a nullable column, set it when generating note flashcards, OR use source_excerpt hashing. **Pragmatic approach**: use the content hash pattern. FlashcardModel already has `source_content_hash`. Generate cards with hash of note content. The uncovered query then checks if any flashcard row has `source='note'` and `deck=collection.name` and `source_content_hash = hash(note.content[:500])`. But this requires the generating service to have already set it.

**Simplest correct approach given current schema**: query `SELECT deck, COUNT(*) FROM flashcards WHERE source='note' GROUP BY deck` -- if deck=collection.name count > 0, at least some notes are covered. For per-note coverage we need a note_id linkage. **Decision**: add a nullable `note_id TEXT` column to FlashcardModel (and a matching ALTER TABLE migration) so note-sourced cards can be traced back. This is a schema addition needed to make AC 4 testable per-note. This addition is justified and additive. Update the service that generates note flashcards (notes.py generate_note_flashcards) to populate this column.

This impacts `backend/app/models.py` (FlashcardModel), `backend/app/db_init.py` (ALTER TABLE), and `backend/app/routers/notes.py` (set note_id when inserting). The CollectionHealthService uncovered_notes query then becomes: notes in collection WHERE id NOT IN (SELECT note_id FROM flashcards WHERE source='note' AND deck=:collection_name AND note_id IS NOT NULL).

### archived field in models.py
SQLite stores booleans as integers. Use `Mapped[bool]` with `mapped_column(Boolean, nullable=False, default=False, server_default="0")` to be consistent with FlashcardModel.fsrs_state pattern.

### Route ordering in collections.py
The new routes `GET /{id}/health` and `POST /{id}/health/archive-stale` must be registered AFTER the static routes (`GET /tree`) but BEFORE any other `/{id}/notes/{note_id}` patterns. Since all current routes use unique path segments (`/notes`, `/notes/{note_id}`), the `/health` and `/health/archive-stale` paths are safe to append in any order -- they don't conflict with existing paths.
