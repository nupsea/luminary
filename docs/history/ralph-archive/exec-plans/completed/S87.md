# S87 — Notes Focused Editor Dialog with Side-by-Side Write/Preview

## Goal

Replace the existing inline note editor in Notes.tsx (both list and card views) with a page-level NoteEditorDialog that shows Write and Preview panes side-by-side in a focused modal Dialog.

## Layers Affected

- Frontend only: API layer (no backend changes)

## Pre-Implementation Findings

Reading the source before writing any code revealed several facts that affect the plan:

1. **NoteCard is not a separate file.** It is a function component defined inline inside `frontend/src/pages/Notes.tsx` (line 338). There is no `frontend/src/components/NoteCard.tsx`. All changes happen in Notes.tsx and the new NoteEditorDialog.tsx.

2. **`editingNote` state already exists at page level** (line 460). It was wired up in S84 to replace the dead onClick stub. The list view `TableRow onClick` already calls `setEditingNote(note)`. What currently renders is a `NoteEditorTabs` block inline at page bottom (lines 741--767), not a Dialog.

3. **No shadcn Dialog component exists.** `frontend/src/components/ui/dialog.tsx` does not exist and `@radix-ui/react-dialog` is not listed in `frontend/package.json`. The implementation must: (a) install the package via `npm install`, and (b) create `frontend/src/components/ui/dialog.tsx` as a custom Radix-based Dialog before creating NoteEditorDialog.tsx.

4. **`NoteEditorTabs`** is defined in Notes.tsx (lines 249--326) and is used in two places: the NoteCard inline editor (lines 410--417) and the page-level inline editor (lines 753--762). Both usages are removed by S87.

5. **patchNote** API helper is already defined in Notes.tsx (lines 105--116). NoteEditorDialog will import or receive it via a prop -- since NoteEditorDialog is a separate file, it must either re-declare the API call or receive it as a prop. The cleanest approach: NoteEditorDialog makes the fetch call directly using the same pattern (no prop), consistent with how other components in this codebase embed their own fetch logic (ExplanationSheet, SearchDialog).

6. **MarkdownRenderer** is at `@/components/MarkdownRenderer` and accepts `children: string`. It works fine for real-time preview.

7. **Keyboard shortcut pattern:** SearchDialog (lines 93--99) uses `window.addEventListener("keydown", ...)` inside a `useEffect` guarded by `open`. This is the established pattern; replicate it for Ctrl+S / Cmd+S.

interface NoteEditorDialogProps {

### Step 3 — Create `frontend/src/components/NoteEditorDialog.tsx` [Frontend Component]

Full component specification:

**State:**
```typescript
const [content, setContent] = useState(note?.content ?? "")
const [saveError, setSaveError] = useState<string | null>(null)
```

**Effect 1 -- re-initialise on note change:**
```typescript
useEffect(() => {
  if (note) {
    setContent(note.content)
    setSaveError(null)
  }
}, [note?.id])
// dep is note?.id, not note object, to avoid re-initialising on every keystroke
```

**Effect 2 -- Ctrl+S / Cmd+S keyboard shortcut:**
```typescript
useEffect(() => {
  if (!note) return
  function onKeyDown(e: KeyboardEvent) {
    if ((e.ctrlKey || e.metaKey) && e.key === "s") {
      e.preventDefault()
      if (content !== note.content && !saveMut.isPending) {
        saveMut.mutate()
      }
    }
  }
  window.addEventListener("keydown", onKeyDown)
  return () => window.removeEventListener("keydown", onKeyDown)
}, [note, content, saveMut])
```

**Mutation:**
```typescript
const saveMut = useMutation({
  mutationFn: () => fetch(`http://localhost:8000/notes/${note!.id}`, {
    method: "PATCH",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ content }),
  }).then(async (res) => {
    if (!res.ok) throw new Error(`PATCH /notes/${note!.id} failed: ${res.status}`)
    return res.json() as Promise<Note>
  }),
  onSuccess: (updated: Note) => {
    onSaved(updated)
    onClose()
  },
  onError: () => {
    setSaveError("Failed to save. Please try again.")
  },
})
```

**Dialog title:** `note.content.slice(0, 60) || "Untitled note"`

**Layout (DialogContent body, two-column grid):**
```
div.grid.grid-cols-2.gap-0.h-full.overflow-hidden
  div.flex.flex-col.border-r.border-border          ← Write column
    div.px-3.py-2.text-xs.font-medium.border-b      ← "Write" label
    textarea.flex-1.resize-none.font-mono.text-sm.p-3.focus:outline-none.bg-transparent
  div.flex.flex-col                                  ← Preview column
    div.px-3.py-2.text-xs.font-medium.border-b      ← "Preview" label
    div.flex-1.overflow-auto.p-3                     ← MarkdownRenderer or empty state
```

The two-column grid fills the full DialogContent height. The Write textarea uses `flex-1` to fill the column vertically. No tab switching -- both panes visible simultaneously.

**Tags display (below Write pane, inside Write column, above DialogFooter):**
Since DialogFooter is outside the grid, tags and group_name are rendered inside the Write column below the textarea:
```
div.border-t.border-border.px-3.py-2.flex.flex-wrap.gap-1.text-xs.text-muted-foreground
  // note.tags.map -> Tag chip spans
  // note.group_name -> FolderOpen icon + name span
```

**DialogFooter:**
- Left: inline error text (`saveError && <p className="text-xs text-red-600 mr-auto">...`)
- Right: Cancel button (calls onClose), Save button (disabled when `content === note.content || saveMut.isPending`, shows `Loader2` when pending)

**Preview empty state:** `<p className="text-sm text-muted-foreground">Nothing to preview yet.</p>`

## Risk Flags

- Risk: `@radix-ui/react-dialog` is not installed -- the Dialog component cannot be built until this npm package is added. (Mitigation: Step 1 installs it before any component code is written. Verify `npm install` succeeds before proceeding to Step 2.)

- Risk: `NoteEditorTabs` removal from Notes.tsx is a surgical multi-location delete. If any reference is missed, `tsc --noEmit` will fail with "cannot find name NoteEditorTabs". (Mitigation: Grep for all occurrences of `NoteEditorTabs` before committing: there are exactly 2 usage sites and 1 definition site, all in Notes.tsx.)

- Risk: The `onUpdated` prop on NoteCard is removed. If any other call site passes `onUpdated`, TypeScript will error. (Mitigation: There is only one call site -- the card grid in Notes.tsx. Grep confirms no external import of NoteCard.)

- Risk: `saveMut` captured in the Ctrl+S `useEffect` closure. If `saveMut` is listed in `useEffect` deps and recreated on every render, the keydown handler will be added and removed on every keystroke, degrading performance. (Mitigation: Use a ref for the save callback, or rely on `useMutation` returning a stable `mutate` reference. TanStack Query v5 guarantees `mutate` is stable. Keep `note`, `content`, and `saveMut.isPending` in deps but not `saveMut` itself -- or restructure using `useCallback`.)

- Risk: Dialog z-index conflicts with other overlays (SearchDialog uses z-50). (Mitigation: Radix Dialog Portal renders into document.body by default; z-index is handled by Radix's stacking context, avoiding conflicts with fixed-position elements in the Notes page.)

- Risk: Two-column layout at narrow viewport widths. (Mitigation: `max-w-4xl` on DialogContent ensures adequate width. On very small screens the columns may be cramped, but the story does not require responsive breakpoints for the dialog. Acceptable as-is for desktop-first use.)

- Risk: Frontend three-states invariant (invariant 13). This story modifies the Notes page which already has loading/error/empty states. Those states remain untouched; the dialog is an overlay that does not affect them. No new data fetch path is added to the page. The dialog's own error state is the inline footer error per the AC. (Mitigation: Confirm the existing loading skeleton, error panel, and empty state are preserved after edits to Notes.tsx.)
