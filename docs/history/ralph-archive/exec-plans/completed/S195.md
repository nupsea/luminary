# Exec Plan: S195 -- Chat: Bloom-progressive recommendations with dedup and summary grounding

**Phase:** V3 Phase 5 - Reading Experience and Chat Intelligence
**Priority:** 35
**Status:** in-progress

## Goal

Replace the template-driven chat suggestion pills (S187) with LLM-generated questions that are grounded in the document's executive summary and top entities, progress through Bloom taxonomy levels based on engagement history, and never repeat a previously shown question. The change adds a `ChatSuggestionHistoryModel` to track shown/asked suggestions, refactors `GET /chat/suggestions` to call an LLM with dedup filtering, adds `POST /chat/suggestions/{id}/asked` to record clicked pills, and updates the `SuggestionPills` frontend component to use the new `{id, text}` response shape and call the asked endpoint on click.

## Cross-Story Risk

- **S187 (Chat suggestion pills, baseline):** Direct enhancement. The `SuggestionResponse` schema change from `{suggestions: string[]}` to `{suggestions: [{id, text}]}` is a breaking API change. All frontend consumers of `GET /chat/suggestions` must be updated simultaneously. Only `SuggestionPills` in `Chat.tsx` consumes this endpoint -- update is self-contained.
- **I-1 (AsyncSession sharing):** `generate_suggestions` and `mark_asked` each open their own session via `get_session_factory()`. No `asyncio.gather` with shared session.
- **I-2 (LanceDB sync wrapping):** Kuzu entity fetch uses `asyncio.to_thread` (already established pattern in existing `chat_meta.py`).
- **I-6 (Settings at module level):** `get_settings()` imported at module level in `suggestion_service.py` for `LITELLM_DEFAULT_MODEL`. Never lazy-imported.
- **I-7 (LLM call in non-SSE context):** `generate_suggestions` is a regular async function, not an SSE generator -- no rollback guard needed beyond normal session `async with` context manager.
- **I-13 (Quality gate order):** ruff before pytest before tsc. Enforced in step 9.
- **S196 (Chat scope clear):** Modifies `Chat.tsx` scope-change logic. If S196 is implemented after S195, it must preserve the `invalidateQueries` call added in this story.
