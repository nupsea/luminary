**Phase:** V3 Phase 6 - Polish and Bug Fixes

## Goal

Add a keyboard-accessible search overlay to PDFViewer.tsx that highlights all occurrences of a search term on the current page using the existing pdfjs TextLayer spans, tracks total match count across all pages via lazy per-page text caching, and allows next/prev navigation through matches -- including cross-page scrolling. ## Implementation Notes

### Highlight coexistence (AC 9)
`applyPdfHighlights` uses `<mark data-pdf-highlight>` and `span[data-hl-original]`. Search highlights must use a different attribute `data-search-highlight`. The clear functions must only clear their own marks. The ordering of effects determines which highlights appear on top: annotation highlights run in the `textLayerVersion + annotations + highlightsVisible + currentPage + sections` effect; search highlights run in a separate effect triggered by `textLayerVersion + searchQuery + searchMatchIndex`. Since both effects run after the same `textLayerVersion` bump, search highlights will always be applied after annotation highlights when the search bar is open -- this is correct since search is a transient overlay.

### Cross-page navigation (AC 5)
`searchMatches` is a flat array sorted by page then local index. `nextMatch` increments `searchMatchIndex`. When the new match's `page` differs from `currentPage`, call `goToPage(match.page)`. The page change triggers a new render cycle and a `textLayerVersion` bump, which triggers the search highlight effect for the new page. The active match index within that page is computed as the count of `searchMatches` with the same page and index less than `searchMatchIndex`.

### Lazy text extraction with cache (AC 8)
`pageTextCacheRef.current` is a `Map<number, string>` keyed by 1-based page number. Population is lazy: on query change, a `useEffect` fires, iterates pages 1..totalPages, checks cache, fetches missing pages via `pdfDoc.getPage(n).then(p => p.getTextContent())`, stores in cache. This effect is debounced by the query value (300ms) to avoid thrashing on each keystroke. Cancellation uses an `isCancelled` closure flag.

### Effect ordering (critical for AC 9 regression)
In React 18 strict mode, effects run in declaration order within the same component. The existing annotation highlight effect is declared at lines 488-507. The new search highlight effect must be declared AFTER it, so search marks are applied on top of annotation marks.

### Match count display (AC 3)
- Current-page matches: count of entries in `searchMatches` where `page === currentPage`
- Current-page active index: 1-based position among same-page matches (e.g. `2 of 5 on page`)
- Document total: sum of all entries in `searchMatches`
- While scanning: replace document total with "..." until all pages resolved

## Cross-Story Risk

- **S147 / S198** (annotation highlights via `applyPdfHighlights`): risk of regression if search highlight clearing accidentally removes `[data-pdf-highlight]` or `[data-hl-original]` marks. Mitigated by: using a distinct `data-search-highlight` attribute and only querying `mark[data-search-highlight]` in `clearSearchHighlights`.
- **S200** (PDF link handling via AnnotationLayer): no risk -- search operates only on the TextLayer div; the AnnotationLayer div is separate and untouched.
- **Invariant I-10**: search overlay must show "No matches" when query has no results (empty state). Search input shows "Scanning..." while document total is being computed (loading state for the total count).
- **Invariant I-13**: quality gates run in order ruff -> pytest -> tsc. No backend changes so ruff and pytest should be unaffected; tsc is the primary gate for this story.
