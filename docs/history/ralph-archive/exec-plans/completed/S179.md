# Exec Plan: S179 -- Context-aware flashcard generation: best-practices for non-fiction and technical books

**Phase:** V3 Phase 3 - UX Polish and Learner Focus
**Priority:** 19
**Status:** in-progress

## Goal

Rewrites the flashcard generation pipeline to avoid producing cards from illustrative digressions (analogies, narratives, transitions) in non-fiction and technical books. A pre-generation chunk classifier labels each candidate chunk; only concept/definition chunks (plus adjacent example/analogy elaborators) are submitted to the LLM. The system prompt gains a document genre hint and explicit quality rules. A new `chunk_classification` column on `FlashcardModel` records the classification decision. Embedding-based deduplication silently skips generated Q&A pairs with cosine similarity >= 0.85 to any existing card in the same deck.

## Key Design Decisions

### Chunk Classifier

The classifier is a lightweight heuristic function (no extra LLM call) operating on the raw chunk text. This is preferred for performance -- a single LLM call per chunk at scale would be prohibitively slow.

Classification rules (applied in order, first match wins):
1. **definition** -- chunk contains "is defined as", "refers to", "means that", "is the process of", or starts with a term followed by a colon
2. **concept** -- chunk contains "therefore", "as a result", "the key idea", "the principle", "this means", "the reason", or discusses causality/mechanism
3. **analogy** -- chunk contains "like a", "similar to", "imagine", "think of it as", "is like", "just as", "as if"
4. **example** -- chunk contains "for example", "for instance", "e.g.", "consider this", "such as", "to illustrate"
5. **transition** -- chunk is short (< 80 chars) OR contains "in the next", "in the following", "as we saw", "moving on"
6. **narrative** -- default fallback for any chunk not matching the above

Adjacent elaborator logic: for each concept/definition chunk at index i, also include chunks at i-1 and i+1 if their classification is example or analogy.

### Genre Inference

`_infer_genre(doc: DocumentModel) -> str` inspects `doc.content_type` and `doc.title.lower()`. Returns one of: `"technical"`, `"academic"`, `"narrative"`, `"non-fiction"`. Rules:
- content_type == "book" and title keywords match (programming, systems, distributed, database, algorithm, machine learning, software, engineering, computer) -> `"technical"`
- content_type == "book" -> `"non-fiction"` (default for books)
- content_type in ("pdf", "web") -> `"academic"`
- fallback -> `"narrative"`

### Genre-Aware System Prompt

A new `CONTEXT_AWARE_FLASHCARD_SYSTEM` prompt replaces `FLASHCARD_SYSTEM` in the `generate()` method. It is built with a `_build_genre_system_prompt(genre: str) -> str` function that interpolates the genre hint and adds quality rules:
- No questions whose answer is a proper noun from an analogy or story used to explain a concept
- Questions must be answerable without having read the specific analogy
- Prefer "Explain X in your own words" and "What is the practical implication of Y" over "According to the text, what does Z do"
- Frame questions around WHY or HOW, not WHAT was mentioned in an illustration

### Embedding-Based Dedup

`_compute_cosine(v1: list[float], v2: list[float]) -> float` using numpy dot product (vectors are already L2-normalised by `EmbeddingService.encode` with `normalize_embeddings=True`). For each generated card, embed the question text via `await asyncio.to_thread(get_embedding_service().encode, [question])`. Fetch all existing flashcard questions for the same deck from SQLite. Embed them in a single batch. Skip insertion if any cosine >= 0.85.

Note: the embedder uses `bge-small-en-v1.5` (384-dim, L2-normalised). Cosine similarity for normalised vectors is simply the dot product. No LanceDB table is needed -- the dedup runs purely in-memory against questions fetched from SQLite.

The `chunk_classification` field on the inserted `FlashcardModel` is set to the classification label of the source chunk (e.g. "concept", "definition"). For the context-path (user-selected text) and note-sourced paths, set it to `None` -- these paths bypass the classifier.

## Cross-Story Risk

- **S169** (collection-based flashcard gen): that path uses `source_content_hash` dedup. It calls `FLASHCARD_SYSTEM` (the static constant) directly in `generate_from_collection()`. The new genre-aware prompt is wired only into `generate()` -- the main doc-chunk path. `generate_from_collection()` and other `generate_from_*` methods are not modified in this story. No regression risk.
- **S137** (technical flashcards / Bloom's taxonomy): `generate_technical()` uses `TECH_FLASHCARD_SYSTEM` separately and is not touched. No regression risk.
- **S173/S174** (note_id FK, note-sourced cards): `generate_from_notes()` is not modified. The new `chunk_classification` column is nullable; existing rows get NULL via the ALTER TABLE migration. No regression risk.
- The `_dedup_check()` function fetches existing cards by deck name. The `generate()` method currently does not receive a `deck` parameter -- it always creates cards with `deck="default"`. Verify the router call site and confirm `deck` is hardcoded or passed through before implementing dedup. If deck is not passed, scope dedup to all cards for the document instead.
