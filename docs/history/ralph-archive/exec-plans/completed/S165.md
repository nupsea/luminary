# Exec Plan: S165 -- Hierarchical tag browser UI: tree sidebar, tag management panel, NoteEditorDialog autocomplete

**Phase:** V3 Phase 1 - Note Organization and Knowledge Graph
**Priority:** 6
**Status:** in-progress

## Goal

S162 built the hierarchical tag backend (NoteTagIndexModel, CanonicalTagModel, prefix-search API, PUT /tags/{id}, DELETE /tags/{id}). This story builds the matching frontend: a TagTree component in the Notes sidebar replacing the flat tag button list, a TagManagementPanel popover for rename/re-parent/merge, and a TagAutocomplete component that replaces the flat text input in NoteEditorDialog. It also adds the missing POST /tags/merge backend endpoint and updates note card tag display to show breadcrumb-style hierarchy. The result is a visually hierarchical tag system that guides users toward the controlled vocabulary established in S162.

## Cross-Story Risk

- **S162** (backend tag system): S165 adds POST /tags/merge which modifies CanonicalTagModel rows and NoteTagIndexModel indirectly via _sync_tag_index. If _sync_tag_index in notes.py is not importable from tags.py, a refactor to a shared service module (e.g. `services/tag_service.py`) will be required. Read notes.py before implementing merge to verify the import path.
- **S164** (Collections UI): NoteEditorDialog gains a new TagAutocomplete section replacing the flat tag input. The Collections section added in S164 must be preserved and not displaced. Carefully integrate without removing the collections checkbox section.
- **S163** (Notes as Kuzu nodes): notes modified during merge (tag array replacement via PATCH equivalent) may trigger NoteGraphService upsert_note_node asyncio.create_task. The merge endpoint writes directly via ORM and calls _sync_tag_index but does NOT go through the notes router -- no create_task fires. If Kuzu graph accuracy for merged tags matters, a future story should trigger graph re-sync post-merge.
