# Exec Plan: S163 -- Notes as Kuzu graph nodes: NOTE type, WRITTEN_ABOUT entity edges, TAG_IS_CONCEPT bridge

**Phase:** V3 Phase 1 - Note Organization and Knowledge Graph
**Priority:** 3
**Status:** in-progress

## Goal

Add persistent Kuzu graph integration for notes. This story introduces a Note node type in
Kuzu, three relationship tables (WRITTEN_ABOUT, TAG_IS_CONCEPT, DERIVED_FROM), and a
NoteGraphService that wires them together. On every note save, GLiNER extracts entities from
the note content; matches are linked via WRITTEN_ABOUT edges. Tags that match existing Entity
names get TAG_IS_CONCEPT edges. Notes with a source document get DERIVED_FROM edges. The
chat_graph notes_node is enriched to append Kuzu entity names to the top-3 note context
snippets, enabling graph-backed note discovery without full-text scan.

## Implementation Notes

### Kuzu Upsert Pattern for Note Nodes
Kuzu does not have native MERGE. Use the same pattern as `upsert_entity()` in graph.py:
check existence with MATCH + has_next(), then SET or CREATE accordingly.

### GLiNER Integration in NoteGraphService
Import lazily inside method body (I-5): `from app.services.ner import get_entity_extractor  # noqa: PLC0415`
The extractor's `.extract()` method takes `chunks: list[dict]` where each dict has `{id, document_id, text}`.
For note_graph, wrap the note content as a single-chunk list: `[{"id": note_id, "document_id": document_id or "", "text": content}]`.
Use `content_type="unknown"` so tech-specific entity types are excluded (matches note use case).

### asyncio.to_thread Requirement (I-2)
All Kuzu `_conn.execute()` calls are synchronous. The entire body of each async method in
NoteGraphService must be wrapped in `await asyncio.to_thread(...)` or use a private sync
helper called via to_thread. The GLiNER `extract()` call is also synchronous and must
similarly be executed via to_thread (it already uses `loop.run_in_executor` in other services
-- align with that pattern using `asyncio.to_thread`).

### Fire-and-Forget Wrapper in notes.py
The _upsert_note_graph async helper must be a module-level function (not a lambda) so its
exception is logged. Pattern from _embed_and_store_note: wrap body in try/except Exception
and log with logger.warning. The lazy import of NoteGraphService goes inside this function
body.

### DELETE Handler: Synchronous Graph Delete
The story spec says "called synchronously in DELETE /notes/{id} handler". This means
`await delete_note_node(note_id)` runs inline before the vector delete, not as a
create_task. This matches the pattern where vector delete is also inline in the DELETE handler.

### chat_graph.py Enrichment
Keep the enrichment non-blocking to failure: wrap get_entities_for_note in try/except.
Append to the note line only if entities list is non-empty. The enrichment runs for the
top 3 (not all k=5) results to bound the number of Kuzu lookups per query.
