### Step 2 — [Frontend] Add API helper functions to Study.tsx

Below the existing `generateFlashcards` function, add:

```typescript
async function fetchEntityPairs(documentId: string): Promise<EntityPairsResponse> {
  const res = await fetch(
    `${API_BASE}/flashcards/entity-pairs?document_id=${encodeURIComponent(documentId)}`
  )
  if (!res.ok) return { pairs: [] }
  return res.json() as Promise<EntityPairsResponse>
}

async function generateFromGraph(req: {
  document_id: string
  k: number
}): Promise<Flashcard[]> {
  const res = await fetch(`${API_BASE}/flashcards/generate-from-graph`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(req),
  })
  if (!res.ok) throw new GenerateError(res.status, "Failed to generate entity flashcards")
  return res.json() as Promise<Flashcard[]>
}
```

## Risk Flags

- **Risk: `ScoredChunk` constructor** -- The dataclass in `app/types.py` requires `section_heading`
  and `page` as positional fields. The test stubs must supply them or pytest will raise a
  `TypeError`. Mitigation: read `ScoredChunk` definition carefully before writing stubs; use
  keyword arguments.

- **Risk: Lazy import patch location for `get_graph_service` and `get_retriever`** -- Both are
  imported via `from app.services.graph import get_graph_service` and
  `from app.services.retriever import get_retriever` inside the method body of
  `generate_from_graph`. Per patterns.md, patch at the source module:
  `patch("app.services.graph.get_graph_service", ...)` and
  `patch("app.services.retriever.get_retriever", ...)`. If a test returns the real service
  instead of the stub (detectable by unexpected Kuzu file access), switch to patching at
  `app.services.flashcard` after verifying the import resolves there post-inlining.

- **Risk: Route ordering** -- `GET /flashcards/entity-pairs` and `POST /flashcards/generate-from-graph`
  are already registered before the parametric `/{document_id}` routes in the router (confirmed
  in flashcards.py lines 211 and 238 vs. 267). No change needed.

- **Risk: TSC strict mode** -- Adding `onGenerateFromGraph: (k: number) => void` to
  `GeneratePanelProps` will cause a TypeScript error at the single call site in `Study.tsx`
  until the prop is passed. The call site (around line 699) must be updated in the same edit.

- **Risk: Frontend three-state invariant** -- Invariant 13 requires loading, error, and empty
  states for every UI section. The entity-pair preview section must explicitly render all three.
  The empty state message must distinguish between "no pairs yet" (Kuzu graph is sparse) and
  an error state, so users understand what action to take.

- **Risk: `pairsLoading` and `pairsError` both false during initial render before `mode` is
  set to `"entities"`** -- The `useQuery` call is `enabled: mode === "entities"`, so when
  `mode === "text"` the query is in an idle state (`isLoading=false`, `isError=false`,
  `data=undefined`). The rendered JSX for `mode === "entities"` is only shown when
  `mode === "entities"`, so the idle state is never visible. No extra guard needed.
