# S111 - In-document annotation and text highlighting with persistent storage

## Goal

Add persistent text-highlight annotations to DocumentReader: users select text in any section preview, choose a color from a popover, optionally type a note, save the annotation, and see highlights reconstructed on reload; a Highlights tab in the left panel lists all annotations for the document with delete-with-confirmation.

---

- **Types** — new AnnotationModel ORM class in `backend/app/models.py`

- **API** — new `backend/app/routers/annotations.py`; registered in `backend/app/main.py`; `backend/app/db_init.py` updated
- **Frontend** — `FloatingToolbar.tsx`, `DocumentReader.tsx`, new `AnnotationPopover.tsx` component

## Implementation Steps

### Step 1 — [Types] Add AnnotationModel to models.py

Add the following class at the end of `backend/app/models.py`, after `ReadingProgressModel`:

```python
class AnnotationModel(Base):
    __tablename__ = "annotations"

    id: Mapped[str] = mapped_column(String, primary_key=True)
    document_id: Mapped[str] = mapped_column(String, nullable=False, index=True)
    section_id: Mapped[str] = mapped_column(String, nullable=False)
    # chunk_id is nullable: sections do not map 1:1 to chunks
    chunk_id: Mapped[str | None] = mapped_column(String, nullable=True)
    selected_text: Mapped[str] = mapped_column(Text, nullable=False)
    start_offset: Mapped[int] = mapped_column(Integer, nullable=False)
    end_offset: Mapped[int] = mapped_column(Integer, nullable=False)
    # yellow|green|blue|pink
    color: Mapped[str] = mapped_column(String(20), nullable=False, default="yellow")
    note_text: Mapped[str | None] = mapped_column(Text, nullable=True)
    created_at: Mapped[datetime] = mapped_column(
        DateTime, default=lambda: datetime.now(UTC)
    )
```

Fields match the AC exactly. `section_id` is non-nullable because highlights must anchor to a section to be reconstructable.

### Step 3 — [API] Create backend/app/routers/annotations.py

New file. Pattern mirrors `backend/app/routers/reading.py` (session from `get_session_factory`, no Depends injection, just `get_db` for simple endpoints).

**Pydantic schemas:**

```python
class AnnotationCreateRequest(BaseModel):
    document_id: str
    section_id: str
    chunk_id: str | None = None
    selected_text: str
    start_offset: int
    end_offset: int
    color: Literal["yellow", "green", "blue", "pink"] = "yellow"
    note_text: str | None = None

class AnnotationResponse(BaseModel):
    id: str
    document_id: str
    section_id: str
    chunk_id: str | None
    selected_text: str
    start_offset: int
    end_offset: int
    color: str
    note_text: str | None
    created_at: datetime
    model_config = {"from_attributes": True}
```

**Endpoints:**

- `POST /annotations` — create, HTTP 201, returns `AnnotationResponse`
- `GET /annotations` — query param `document_id: str` (required), returns `list[AnnotationResponse]` ordered by `created_at` asc
- `DELETE /annotations/{annotation_id}` — HTTP 204, 404 if not found

Structured logging at INFO for every mutation. No LLM, no FTS, no vector store — pure SQLite ORM.

Example POST handler:

```python
@router.post("", response_model=AnnotationResponse, status_code=201)
async def create_annotation(
    req: AnnotationCreateRequest,
    session: AsyncSession = Depends(get_db),
) -> AnnotationResponse:
    annotation = AnnotationModel(
        id=str(uuid.uuid4()),
        document_id=req.document_id,
        section_id=req.section_id,
        chunk_id=req.chunk_id,
        selected_text=req.selected_text,
        start_offset=req.start_offset,
        end_offset=req.end_offset,
        color=req.color,
        note_text=req.note_text,
        created_at=datetime.now(UTC),
    )
    session.add(annotation)
    await session.commit()
    await session.refresh(annotation)
    logger.info("Created annotation", extra={"annotation_id": annotation.id})
    return AnnotationResponse.model_validate(annotation)
```

### Step 6 — [Frontend] Update FloatingToolbar.tsx

Add a `onHighlight` optional callback prop. When the user clicks the Highlight button, pass back the selected text AND the character offsets within the containing section preview element. The toolbar already has `selectedText`; the offset computation needs the selection range.

**Interface change:**

```typescript
interface FloatingToolbarProps {
  containerRef: React.RefObject<HTMLElement | null>
  onExplain: (text: string, mode: ExplainMode) => void
  onHighlight?: (info: HighlightInfo) => void
}

export interface HighlightInfo {
  text: string
  sectionId: string       // data-section-id of the nearest ancestor li
  startOffset: number     // character offset within the section preview text
  endOffset: number
}
```

**Offset computation in handleMouseUp:** After computing position from the range rect, also extract the `data-section-id` from the nearest `[data-section-id]` ancestor of `range.startContainer`, and compute `startOffset`/`endOffset` as character positions within the preview `<p>` element text content. Store this alongside `selectedText` in a `highlightInfo` ref.

**Highlight button in toolbar JSX:** Add a new button before or after the Explain buttons:

```tsx
{onHighlight && (
  <button
    onMouseDown={(e) => {
      e.preventDefault()
      if (pendingHighlightInfo) onHighlight(pendingHighlightInfo)
      setPosition(null)
      setSelectedText("")
    }}
    className="rounded px-2.5 py-1 text-xs font-medium text-foreground hover:bg-accent"
  >
    Highlight
  </button>
)}
```

`pendingHighlightInfo` is a `useRef<HighlightInfo | null>` updated in `handleMouseUp`.

**Offset computation detail:** In `handleMouseUp`, after establishing the range:

```typescript
// Walk up from startContainer to find [data-section-id] li
let node: Node | null = range.startContainer
let sectionId = ""
while (node) {
  if (node instanceof HTMLElement && node.dataset["sectionId"]) {
    sectionId = node.dataset["sectionId"]
    break
  }
  node = node.parentElement
}
// Find preview <p> text within that li
const previewEl = (node as HTMLElement | null)?.querySelector("p.text-xs")
const previewText = previewEl?.textContent ?? ""
// Compute offset of selection within the preview text using indexOf
const selText = selection.toString().trim()
const startOffset = previewText.indexOf(selText)
const endOffset = startOffset >= 0 ? startOffset + selText.length : -1
```

If `sectionId` is empty or `startOffset < 0`, the Highlight button is hidden (set `pendingHighlightInfo` to null).

**8e. Highlight reconstruction in section preview rendering:**

Each section `<li>` currently renders its preview as:

```tsx

  {section.preview}
</p>
```

Replace with a `<SectionPreviewWithHighlights>` sub-component (defined inline in DocumentReader.tsx or imported from a small helper) that:

1. Takes `preview: string`, `annotations: AnnotationItem[]`, `sectionId: string`
2. Filters annotations to those matching `sectionId`
3. Sorts by `start_offset`
4. Splits `preview` into text-before, highlighted-span, text-after segments using character slicing

## Risk Flags

- **Risk: Offset mismatch on clamped text.** The section preview is `line-clamp-2` in CSS, which visually hides text after ~2 lines but does NOT truncate the DOM `textContent`. Browser `window.getSelection()` returns the full character offset within the un-clamped text. The `preview` field in the DB is the full preview text (not truncated). So offsets will be correct as long as we compute them against the full `textContent` of the preview `<p>`, not a visually-truncated substring. **Mitigation**: in `SectionPreviewWithHighlights`, always render against `section.preview` (the full DB string), not a trimmed version. The line-clamp class applies only to visual display.
- **Risk: Multiple overlapping annotations on the same text span.** Two annotations could have overlapping offsets. The naive segment-split algorithm breaks when ranges overlap. **Mitigation**: sort annotations by `start_offset` and handle overlap by always preferring the first annotation's color for the overlapping bytes. Alternatively, for MVP, render only non-overlapping annotations (skip any annotation that starts before the previous one's `end_offset`). Document this limitation in a code comment.
- **Risk: left panel tab toggle breaks IntersectionObserver.** The reading-progress IntersectionObserver mounts via `useReadingProgress` which looks for `[data-section-id]` elements. When the Highlights tab is shown, no section `<li>` elements are in the DOM, so no intersections fire. This is correct behavior (user is not reading sections). **Mitigation**: no change needed; the observer's `observe` call in the effect runs once on mount; the re-render that hides sections from the DOM will be handled naturally by the browser removing those nodes from the observed set.
- **Risk: FloatingToolbar position vs AnnotationPopover overlap.** The FloatingToolbar currently positions itself using `containerRef` relative coords. The AnnotationPopover must be rendered AFTER the FloatingToolbar is dismissed (i.e., after the user clicks Highlight and `setPosition(null)` is called in FloatingToolbar). DocumentReader tracks `pendingHighlight` state separately. **Mitigation**: in FloatingToolbar, the Highlight button's `onMouseDown` calls `e.preventDefault()`, fires `onHighlight()`, then calls `setPosition(null)` to dismiss itself before DocumentReader renders the popover. The popover is rendered by DocumentReader at a fixed position (e.g., center of the sectionListRef container) rather than at the selection rect (which is already gone).
- **Risk: AnnotationPopover positioned correctly.** When DocumentReader renders `<AnnotationPopover>` inside the `sectionListRef` div (which has `position: relative`), it must use `position: absolute` with appropriate top/left. Since the selection is cleared by the time the popover renders, store the FloatingToolbar's last `position` (top, left) in DocumentReader state alongside `pendingHighlight`, and use those coords for the popover position.
- **Risk: Delete endpoint does not cascade.** `AnnotationModel` has no FK constraint on `document_id` or `section_id` (consistent with other tables in this codebase -- TD-010 documents that FK CASCADE is not enforced). When a document is deleted, `DELETE /documents/{id}` in `documents.py` manually deletes related rows. **Mitigation**: add a `DELETE FROM annotations WHERE document_id = :doc_id` step in `documents.py`'s delete handler (search for the existing pattern of manual cascade deletes in that file and add annotations to the same block). Flag this in a code comment.
