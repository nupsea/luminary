# S109 ‚Äî Proactive Suggested Explorations in Chat from Kuzu Entity Graph

## Goal

Expose a `GET /chat/explorations` endpoint that returns up to 5 entity-pair exploration suggestions derived from Kuzu RELATED_TO edges for a document, and render them as a dismissible chip row in the Chat tab that pre-fills and auto-submits the chat input when clicked.

## Design Answers

### 1. How does get_related_entity_pairs_for_document work?

Two-step Kuzu query:

Step 1 ‚Äî collect entity IDs that are MENTIONED_IN this document:
```
MATCH (e:Entity)-[:MENTIONED_IN]->(d:Document {id: $did})
RETURN e.id
```

Step 2 ‚Äî find RELATED_TO edges where both endpoints are in that set, ordered by confidence descending, capped at `limit` rows:
```
MATCH (a:Entity)-[r:RELATED_TO]->(b:Entity)
WHERE a.id IN [$eid0, $eid1, ...]
  AND b.id IN [$eid0, $eid1, ...]
RETURN a.name, b.name, r.relation_label, r.confidence
ORDER BY r.confidence DESC
LIMIT $limit
```

Returns `list[tuple[str, str, str, float]]` ‚Äî (name_a, name_b, relation_label, confidence).

If the entity set is empty OR the RELATED_TO query returns no rows, return `[]`.

### 2. How is ExplorationSuggestion.text constructed from entity pairs?

If `relation_label` is non-empty: `"What is the {relation_label} between {name_a} and {name_b}?"`
Otherwise: `"How is {name_a} related to {name_b}?"`

Both names are title-cased before insertion (entity names are stored lowercase by the disambiguator).

`entity_names` field = `[name_a, name_b]` (lowercase as stored; display is handled client-side).

### 3. Where exactly in Chat.tsx is the chip row added?

The existing `ChatSuggestions` pills row renders at line 732-738 (messages.length === 0 && activeDocumentId). The new `ExploreConnectionsChips` renders immediately below `ChatSuggestions`, with its own condition:

```
messages.length === 0 && scope === "single" && effectiveDocId
```

This respects AC5 (hidden when chat is non-empty) and AC6 (omitted gracefully when no entities; the query returns empty array, component returns null).

The component fetches `GET /chat/explorations?document_id={effectiveDocId}`.

### 4. How do existing suggestion pills work?

`ChatSuggestions` (line 129-171) uses `useQuery` with key `["due-pills", activeDocumentId]`, renders skeleton while loading, returns null on error. Click calls `onSuggest(text)` which calls `sendMessage(text)` ‚Äî this directly submits the message (pre-fill + immediate send). The new chip row follows the same pattern: clicking calls `sendMessage(text)` on the parent.

## Implementation Steps

### Step 1 ‚Äî [Service] Add `get_related_entity_pairs_for_document` to `KuzuService`

In `backend/app/services/graph.py`, add a new public method to `KuzuService` between `get_entities_for_documents` and `get_graph_for_document`:

```python
def get_related_entity_pairs_for_document(
    self, document_id: str, limit: int = 5
) -> list[tuple[str, str, str, float]]:
    """Return entity pairs connected by RELATED_TO edges within a document.

    Only returns pairs where BOTH entities are mentioned in the given document.
    Ordered by confidence descending. Returns empty list on any error or when
    the document has no graph entities.

    Returns:
        List of (name_a, name_b, relation_label, confidence) tuples.
    """
    try:
        # Step 1: collect entity IDs in this document
        id_result = self._conn.execute(
            "MATCH (e:Entity)-[:MENTIONED_IN]->(d:Document {id: $did})"
            " RETURN e.id",
            {"did": document_id},
        )
        entity_ids: list[str] = []
        while id_result.has_next():
            row = id_result.get_next()
            if row[0]:
                entity_ids.append(row[0])

        if not entity_ids:
            return []

        # Step 2: RELATED_TO edges between those entities
        placeholders = ", ".join(f"$eid{i}" for i in range(len(entity_ids)))
        params: dict = {f"eid{i}": eid for i, eid in enumerate(entity_ids)}
        params["lim"] = limit
        rel_result = self._conn.execute(
            f"MATCH (a:Entity)-[r:RELATED_TO]->(b:Entity)"
            f" WHERE a.id IN [{placeholders}] AND b.id IN [{placeholders}]"
            f" RETURN a.name, b.name, r.relation_label, r.confidence"
            f" ORDER BY r.confidence DESC"
            f" LIMIT $lim",
            params,
        )
        pairs: list[tuple[str, str, str, float]] = []
        while rel_result.has_next():
            row = rel_result.get_next()
            name_a, name_b, label, conf = row[0], row[1], row[2], row[3]
            if name_a and name_b:
                pairs.append((name_a, name_b, label or "", float(conf or 0.0)))
        return pairs
    except Exception:
        logger.debug(
            "get_related_entity_pairs_for_document failed, returning empty",
            exc_info=True,
        )
        return []
```

### Step 2 ‚Äî [API] Add `ExplorationSuggestion` model and endpoint to `chat_meta.py`

Add to `backend/app/routers/chat_meta.py`:

```python
class ExplorationSuggestion(BaseModel):
    text: str
    entity_names: list[str]


@router.get("/explorations", response_model=list[ExplorationSuggestion])
async def get_explorations(
    document_id: str = Query(..., description="Document ID to query RELATED_TO edges for"),
) -> list[ExplorationSuggestion]:
    """Return up to 5 entity-pair exploration suggestions for a document.

    Suggestions are derived from Kuzu RELATED_TO edges where both entities
    are mentioned in the given document. Returns empty list if no graph
    entities exist for this document.
    """
    svc = get_graph_service()
    pairs = svc.get_related_entity_pairs_for_document(document_id, limit=5)
    results: list[ExplorationSuggestion] = []
    for name_a, name_b, relation_label, _ in pairs:
        title_a = name_a.title()
        title_b = name_b.title()
        if relation_label:
            text = f"What is the {relation_label} between {title_a} and {title_b}?"
        else:
            text = f"How is {title_a} related to {title_b}?"
        results.append(ExplorationSuggestion(text=text, entity_names=[name_a, name_b]))
    return results
```

### Step 3 ‚Äî [Frontend] Add `ExploreConnectionsChips` component to `Chat.tsx`

Add after the `ChatSuggestions` component definition (around line 171), before `ConfusionBanner`:

```typescript
interface ExplorationSuggestion {
  text: string
  entity_names: string[]
}

interface ExploreConnectionsChipsProps {
  documentId: string
  onSuggest: (text: string) => void
}

function ExploreConnectionsChips({ documentId, onSuggest }: ExploreConnectionsChipsProps) {
  const { data, isLoading, isError } = useQuery({
    queryKey: ["explorations", documentId],
    queryFn: async () => {
      const res = await fetch(
        `${API_BASE}/chat/explorations?document_id=${encodeURIComponent(documentId)}`
      )
      if (!res.ok) throw new Error("Failed to fetch explorations")
      return res.json() as Promise<ExplorationSuggestion[]>
    },
    staleTime: 120_000,
  })

  if (isError) return null
  if (isLoading) {
    return (
      <div className="flex flex-wrap gap-2 border-t border-border px-6 py-2">
        <div className="h-7 w-40 animate-pulse rounded-full bg-muted" />
        <div className="h-7 w-36 animate-pulse rounded-full bg-muted" />
      </div>
    )
  }
  if (!data || data.length === 0) return null

  return (
    <div className="flex flex-wrap gap-2 border-t border-border px-6 py-2">
      <span className="self-center text-xs text-muted-foreground">Explore connections:</span>
      {data.map((s) => (
        <button
          key={s.text}
          onClick={() => onSuggest(s.text)}
          className="truncate max-w-[240px] rounded-full border border-blue-200 bg-blue-50 px-3 py-1.5 text-xs text-blue-700 hover:bg-blue-100 transition-colors dark:border-blue-800 dark:bg-blue-950 dark:text-blue-300 dark:hover:bg-blue-900"
        >
          {s.text}
        </button>
      ))}
    </div>
  )
}
```

## Risk Flags

- Risk: RELATED_TO edges may be sparse in practice (GLiNER extracts entities, but RELATED_TO edges require a separate relation extraction step not currently wired in ingestion). The endpoint returns `[]` gracefully, the chip row is omitted, and AC6 explicitly covers this. No phantom data path ‚Äî the code handles empty correctly.
  - Mitigation: AC6 explicitly tests empty-graph graceful omission. The test suite seeds synthetic RELATED_TO edges via `KuzuService` methods to verify the happy path independently of ingestion.

- Risk: Kuzu `LIMIT $lim` with a named parameter may not be supported in all Kuzu versions (some versions require integer literals in LIMIT clauses). Check the existing Kuzu version in pyproject.toml before implementing; if named params fail, use Python string formatting for the LIMIT value (safe because it is an integer constant, not user input).
  - Mitigation: The `count_for_document` method already uses parametrized queries but not LIMIT. Use `f"... LIMIT {limit}"` (integer, not user string) if named params fail.

- Risk: `relation_label` may be `None` (Kuzu returns None for unset STRING fields). The implementation guards with `label or ""` before string interpolation.
  - Mitigation: Explicit `or ""` guard in the service method and conditional text construction in the router.

- Risk: Chip row condition in Chat.tsx uses `effectiveDocId` (line 507: `selectedDocId ?? activeDocumentId`). Must ensure the condition is `scope === "single" && effectiveDocId` to avoid showing graph-exploration chips in "All my content" scope where there is no single focused document.
  - Mitigation: Condition explicitly checks both `scope === "single"` and `effectiveDocId` is truthy.
