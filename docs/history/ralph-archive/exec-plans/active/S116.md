# S116 -- FSRS Fragility Heatmap on DocumentReader Section List

## Goal

Add a `GET /study/section-heatmap?document_id=X` endpoint that returns per-section FSRS fragility scores, and render those scores as color-coded left borders on DocumentReader section list items with a hover tooltip showing due card count and average retention percentage.

## Layers Affected

- **Types** (new Pydantic response schema in study.py)
- **Service** (pure helper function `_compute_section_heatmap` added in study.py)
- **API** (new `GET /study/section-heatmap` endpoint in `backend/app/routers/study.py`)
- **Frontend** (DocumentReader section list rendering in `frontend/src/components/reader/DocumentReader.tsx`, heatmap query invalidation in `frontend/src/pages/Study.tsx`)

## Implementation Steps

### Step 1 -- [Types/API] Add Pydantic schemas to study.py

Add two new Pydantic models after the existing `StrugglingCardItem` schema (around line 218):

```python

    section_id: str
    fragility_score: float | None
    due_card_count: int
    avg_retention_pct: float | None


class SectionHeatmapResponse(BaseModel):
    heatmap: dict[str, SectionHeatmapItem]
```

`fragility_score` is `None` for sections with no cards. `due_card_count` and `avg_retention_pct` are included so the frontend tooltip can display them without a second fetch.

### Step 2 -- [Service] Add pure helper `_compute_section_heatmap`

Add a pure function (no I/O, no DB, all inputs explicit) to `study.py`, placed above the endpoint definitions:

```python
def _compute_section_heatmap(
    cards: list[FlashcardModel],
    chunk_to_section: dict[str, str | None],
    now: datetime,
) -> dict[str, SectionHeatmapItem]:
    """Aggregate FSRS retrievability per section.

    fragility_score = 1 - avg_retrievability where retrievability = exp(-t/S).
    Sections with no cards are absent from the returned dict.
    Pure function -- no I/O.
    """
    ...
```

Implementation logic:

1. Group cards by `section_id` using `chunk_to_section[card.chunk_id]`. Skip cards where `chunk_id` is None or section_id resolves to None (those cards have no section association).
2. For each section group, compute per-card retrievability:
   - If `card.last_review` is None or `card.fsrs_stability <= 0`: retrievability = 0.0 (never reviewed or uninitialized)
   - Otherwise: `days_since = (now - card.last_review.replace(tzinfo=UTC)).total_seconds() / 86400`; `retrievability = math.exp(-days_since / card.fsrs_stability)`
3. `avg_retrievability = sum(retrievabilities) / len(retrievabilities)`
4. `fragility_score = round(1.0 - avg_retrievability, 4)` -- clamped to [0.0, 1.0]
5. `due_card_count = sum(1 for c in group if c.due_date and c.due_date.replace(tzinfo=UTC) <= now)`
   - Note: `due_date` is stored as naive datetime (SQLite); wrap with `.replace(tzinfo=UTC)` before comparing to `now`
6. `avg_retention_pct = round(avg_retrievability * 100, 1)` -- percentage for tooltip display
7. Return `{section_id: SectionHeatmapItem(...)}` for all sections that have at least one card

### Step 3 -- [API] Add `GET /study/section-heatmap` endpoint

Add the endpoint to `study.py` after `get_struggling_cards`:

```python
@router.get("/section-heatmap", response_model=SectionHeatmapResponse)
async def get_section_heatmap(
    document_id: str,
    session: AsyncSession = Depends(get_db),
) -> SectionHeatmapResponse:
    """Return per-section fragility scores for a document.

    fragility_score ranges from 0.0 (well-retained) to 1.0 (completely forgotten).
    Sections with no flashcards are absent from the heatmap dict.
    """
```

Implementation:
1. Fetch all `FlashcardModel` rows for `document_id` (no state filter -- include new/learning/review/relearning cards; all are relevant to fragility).
2. If no cards: return `SectionHeatmapResponse(heatmap={})`.
3. Collect `chunk_ids = [c.chunk_id for c in cards if c.chunk_id]`.
4. Single query joining `ChunkModel` to get `section_id` for each `chunk_id`:
   ```python
   chunk_stmt = select(ChunkModel.id, ChunkModel.section_id).where(
       ChunkModel.id.in_(chunk_ids)
   )
   ```
5. Build `chunk_to_section: dict[str, str | None]`.
6. Call `_compute_section_heatmap(cards, chunk_to_section, datetime.now(UTC))`.
7. Return `SectionHeatmapResponse(heatmap=result)`.
8. Log at INFO: `"section-heatmap: document_id=%s sections_with_cards=%d"`.

Important: `document_id` is a required query parameter, not a path parameter. The endpoint path is `/study/section-heatmap` with `document_id: str` as a `Query` dependency. This matches the AC: `GET /study/section-heatmap?document_id=X`.

function fragilityBorderClass(score: number | null): string {
  if (score === null) return ""
  if (score <= 0.3) return "border-l-4 border-l-green-500"
  if (score <= 0.6) return "border-l-4 border-l-yellow-500"
  return "border-l-4 border-l-red-500"
}

## Risk Flags

- Risk: AC7 states "card with retrievability 0.2 -> fragility_score <= 0.3" which is mathematically wrong (1 - 0.2 = 0.8, not <= 0.3). The intent is likely "card with retrievability 0.2 -> fragility_score is correctly computed as 0.8" or the AC means "high-retrievability card -> low fragility_score". Mitigation: implement the formula as `fragility_score = 1 - avg_retrievability` which is correct and defensible. For AC7 test, use a card with high retrievability (~0.794) to assert fragility_score <= 0.3. Document the AC discrepancy in the test comment.

- Risk: `due_date` in `FlashcardModel` is stored as naive datetime (no timezone, SQLite limitation). The comparison `card.due_date <= now` must use `card.due_date.replace(tzinfo=UTC)` to avoid comparison errors with timezone-aware `datetime.now(UTC)`. This pattern is already established in `_get_due_for_session()`. Mitigation: replicate the same `.replace(tzinfo=UTC)` pattern in `_compute_section_heatmap`.

- Risk: A flashcard may have `chunk_id=None` (generated from section scope directly or from gap-bridge cards). These cards cannot be resolved to a section and will be silently skipped in the heatmap. This is correct behavior per AC3 (sections with no resolvable cards return null). Mitigation: skip cards where `chunk_id is None` when building `chunk_to_section`.

- Risk: `FlashcardModel.fsrs_stability` defaults to `0.0` for new/unreviewed cards. Division by zero in `exp(-t/S)` is avoided by treating `stability <= 0` as retrievability=0.0 (completely forgotten). Mitigation: add explicit guard in `_compute_section_heatmap`: `if c.fsrs_stability <= 0: retrievability = 0.0`.

- Risk: The `title` attribute tooltip (HTML native) is plain text and lacks styling. The AC says "hovering shows a tooltip" -- a native browser tooltip satisfies this without requiring a Tooltip component. Mitigation: use `title` attribute as planned; document this decision in code comment. If styled tooltip is required in review, upgrade to shadcn Tooltip in a follow-up.

- Risk: DocumentReader.tsx already has 950 lines. Adding the heatmap query and rendering logic adds ~30-40 lines. File remains manageable. Mitigation: keep changes tight; helper function `fragilityBorderClass` is 5 lines.

- Risk: Study.tsx `onExit` invalidates `["section-heatmap", activeDocumentId]` but the DocumentReader renders for a potentially different `documentId` (passed as prop). The Study page and DocumentReader operate on the same `activeDocumentId` from the Zustand store; they are never open simultaneously (Study tab replaces Learning tab). The heatmap invalidation is advisory -- the user must navigate to Learning tab and open DocumentReader after studying. TanStack Query's `staleTime: 60_000` will serve stale data for up to 60 seconds and refresh on next mount, which is acceptable. Mitigation: the invalidation in `onExit` ensures a fresh fetch the next time DocumentReader mounts for that document.

## Notes

- No new npm packages needed. `cn()`, `useQuery`, `useQueryClient` are all already imported in DocumentReader.tsx.
- No new Python packages needed. `math` is already imported in study.py.
- The `_compute_section_heatmap` pure function follows the pattern established by `_compute_gaps` and `_build_session_plan` in the same file (pure, no I/O, explicit parameters -- see core-beliefs.md Belief 19).
- The endpoint path `/study/section-heatmap` is consistent with the existing `/study/gaps/{document_id}`, `/study/stats/{document_id}`, `/study/struggling` pattern in the same router.
- Three frontend states for the heatmap query: loading (no border shows while heatmap is undefined, section renders normally), error (fetch silently returns `{}` -- sections without heatmap data show no border, which is acceptable because heatmap is an enhancement not a primary feature), empty (no cards -> `heatmap={}` -> no borders). The empty/error silent fallback is appropriate for an overlay feature; the document is still readable without it.
