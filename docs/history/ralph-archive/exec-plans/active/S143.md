# S143 — Learning objectives tracking: chapter progress rings, coverage auto-update after FSRS review, and study shortcuts

## Goal

Build the coverage-tracking layer on top of the S132 learning objectives: fire a background computation after every FSRS review that marks objectives as covered when flashcard stability is sufficient, expose progress APIs on documents, and surface progress rings and a "Study" shortcut in the UI.

## Key Architectural Decisions

### 1. section_id join path for flashcards

`FlashcardModel` has no `section_id` column directly. The section must be resolved via:
`FlashcardModel.chunk_id -> ChunkModel.id -> ChunkModel.section_id`

The `ObjectiveTrackerService.update_coverage()` must perform this join:
```sql
SELECT f.id, f.fsrs_stability, c.section_id
FROM flashcards f
LEFT JOIN chunks c ON f.chunk_id = c.id
WHERE f.document_id = :doc_id
  AND (f.flashcard_type != 'definition' OR f.flashcard_type IS NULL)
  AND f.fsrs_stability IS NOT NULL
```

Then group by `section_id` and compute `avg(fsrs_stability)`.

### 2. Coverage threshold

`avg(fsrs_stability) >= 10.0` days across non-definition flashcards for that section.
If a section has zero qualifying flashcards, the objective stays `covered=False` (no phantom coverage).

### 3. Fire-and-forget pattern

Follow the established pattern from `backend/app/routers/notes.py` (module-level `_background_tasks` set):
```python
# In flashcards.py router module-level:
_background_tasks: set[asyncio.Task] = set()

# In review_flashcard() endpoint, after service.schedule():
from app.services.objective_tracker import get_objective_tracker_service  # noqa: PLC0415
if card.document_id:
    tracker = get_objective_tracker_service()
    task = asyncio.create_task(tracker.update_coverage(card.document_id))
    _background_tasks.add(task)
    task.add_done_callback(_background_tasks.discard)
```
Do NOT await the task. The review endpoint returns immediately.

### 4. Progress endpoint response schema

```python
class ChapterProgressItem(BaseModel):
    section_id: str
    heading: str
    total_objectives: int
    covered_objectives: int
    progress_pct: float  # 0.0-100.0

class DocumentProgressResponse(BaseModel):
    document_id: str
    total_objectives: int
    covered_objectives: int
    progress_pct: float  # 0.0-100.0
    by_chapter: list[ChapterProgressItem]
```

Both endpoints (`GET /documents/{id}/progress` and `POST /documents/{id}/refresh_progress`) share this schema.
`POST /refresh_progress` calls `update_coverage(document_id)` synchronously (awaited, not fire-and-forget), then returns the updated progress.

### 5. DocumentListItem extension

Add `objective_progress_pct: float | None` to `DocumentListItem`.
Value is `None` (not `0.0`) when the document has zero `LearningObjectiveModel` rows — this is the "absent ring" case.
Value is `0.0` when objectives exist but none are covered.
The list endpoint adds a correlated scalar subquery for `total_objectives` count; the Python post-processing layer computes the ratio.

### 6. Frontend: progress ring

Use a small SVG ring (not the shadcn Progress bar, which is linear). A 12px ring for per-chapter headers; a 24px ring for library cards.
Pure React functional component `<ProgressRing pct={n} size={24} />` in `DocumentCard.tsx`.
Ring absent (render `--` text) when `objective_progress_pct === null`.
Ring shows gray background arc + colored foreground arc.

### 7. ChapterGoalsPanel upgrade

Current state: objectives are a flat list for the whole document, rendered as dashes.
S143 changes: the panel is now section-scoped per chapter. The Chapter Goals panel opens when the user clicks a section header's progress ring.

Design choice: keep the existing `ChapterGoalsPanel` component and add props:
- `sectionId: string | null` — when set, filter objectives to that section
- `onStudyClick: (sectionId: string) => void` — navigate to Study tab filtered to section + bloom_level >= 2

Objectives render as checkbox rows (checked = `covered=true`, read-only visual).
"Study" button appears beside each uncovered objective, calling `onStudyClick(obj.section_id)`.

### 8. Study tab navigation filter

The Study tab (`Study.tsx`) currently accepts `documentId` via Zustand store.
S143 adds `studySectionFilter: { sectionId: string; bloomLevelMin: number } | null` to the Zustand store.
When this filter is set, the Study tab's flashcard fetch is filtered: `GET /flashcards/{document_id}?section_id={sid}&bloom_level_min=2`.

**Note**: The `GET /flashcards/{document_id}` endpoint does not currently support query filtering by `section_id` or `bloom_level`. This endpoint must be extended to accept optional `?section_id=&bloom_level_min=` query params. This is a small extension to `flashcards.py`.

### 9. Bloom filter in flashcards list endpoint

`GET /flashcards/{document_id}?section_id=<sid>&bloom_level_min=<n>` extension:
- `section_id` resolves via chunk join (same as above)
- `bloom_level_min` filters `FlashcardModel.bloom_level >= n` (null cards excluded when filter is set)

## Implementation Steps

### Step 1 [Service] — Create `ObjectiveTrackerService`

File: `backend/app/services/objective_tracker.py`

```python
class ObjectiveTrackerService:
    async def update_coverage(self, document_id: str) -> None:
        """Mark LearningObjectiveModel rows covered based on avg flashcard stability.

        Coverage criterion: avg(fsrs_stability) of non-definition flashcards in
        the same section_id >= 10 days. No-op for sections with zero qualifying cards.
        """

    async def get_progress(self, document_id: str) -> dict:
        """Return DocumentProgressResponse dict.

        Always returns by_chapter grouped by section_id, sorted by SectionModel.section_order.
        Returns zeros (not 404) when no objectives exist for the document.
        """

def get_objective_tracker_service() -> ObjectiveTrackerService:
    ...  # singleton pattern matching get_fsrs_service()
```

Implementation notes:
- Use `get_session_factory()()` (async context manager) to open its own session — do NOT accept session as parameter, because the service is called as a background task after the request's session is committed and possibly closed.
- Import from `app.models` inside the method to avoid circular imports at module load time (same pattern as `learning_objective_extractor.py`).
- `update_coverage` must handle the case where the document has no flashcards gracefully (objectives stay uncovered).
- `get_progress` must join `LearningObjectiveModel` -> `SectionModel` (by `section_id`) to fetch section headings for `by_chapter`.

## Risk Flags

- **Risk: `FlashcardModel` has no direct `section_id` — must join through `ChunkModel`**
  Mitigation: use `outerjoin(ChunkModel, FlashcardModel.chunk_id == ChunkModel.id)` as seen in `fsrs_service.py` `get_struggling_cards()`. Cards with null `chunk_id` or null `section_id` after join are excluded from the stability average (they cannot be matched to an objective).

- **Risk: Coverage trigger runs in background after session commit; session used by `update_coverage` must be a new session**
  Mitigation: `ObjectiveTrackerService` opens its own session via `get_session_factory()()` — it never receives a session parameter. This matches the pattern used by `LearningObjectiveExtractorService.store_all()`.

- **Risk: `list_flashcards` endpoint now needs a chunk join for the `section_id` filter; current implementation returns raw ORM rows with no join**
  Mitigation: When `section_id` filter is active, add an inner join to `ChunkModel` and filter on `ChunkModel.section_id`. When no filter, preserve the existing no-join path to avoid performance regression.

- **Risk: `GET /documents/{id}/progress` returns `{"by_chapter": []}` for fiction books (no objectives); frontend must not crash on empty list**
  Mitigation: Addressed by the empty state criterion. The `ChapterGoalsPanel` returns null when `total_objectives === 0`.

- **Risk: `DocumentListItem.objective_progress_pct` is a new field; existing frontend code does not destructure it, so TypeScript compile will fail if the backend returns it but types.ts is not updated first**
  Mitigation: Update `types.ts` in Step 7 before making any frontend component changes. The field is nullable so no breaking change to existing code that reads the list.

- **Risk: Study tab `studySectionFilter` persists in Zustand across tab navigations; user may arrive at Study with a stale filter**
  Mitigation: Clear filter on component unmount and provide visible "X" button to clear manually.

- **Risk: The `POST /documents/{id}/refresh_progress` endpoint awaits `update_coverage` synchronously; for large documents with many flashcards this could be slow (up to several hundred rows scanned)**
  Mitigation: The operation is pure SQLite aggregation (no LLM), so even 1000-row scans complete in <100ms. No async mitigation needed.

- **Risk: `objective_progress_pct` in `DocumentListItem` requires two additional correlated subqueries per document in `list_documents()`**
  Mitigation: Keep the existing scalar subquery pattern used for `flashcard_count`, `chunk_count`, etc. SQLite handles correlated scalar subqueries efficiently for typical library sizes (<500 docs). No index changes needed (LearningObjectiveModel already has `document_id` indexed).
