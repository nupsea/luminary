# S134 — Vision LLM Image Analysis: Diagram Classification, Description Embedding, and Retrieval Integration

## Goal

Add an `image_analyze` enrichment job type that runs a vision LLM (llava:13b via LiteLLM) on each extracted image, stores the description and image_type on ImageModel, embeds descriptions into a new LanceDB `image_vectors` table, indexes descriptions in a new `images_fts` FTS5 table, and surfaces matching image thumbnails in Chat answers when image similarity is high enough.

---

- **Config** — add `VISION_MODEL` setting (default `ollama/llava:13b`)
- **Repo** — add `image_vectors` LanceDB table methods to LanceDBService; add `images_fts` FTS5 DDL to db_init.py
- **Service** — new `image_enricher.py` with `ImageEnricherService` and `image_analyze_handler`
- **Runtime** — register `image_analyze` handler in `main.py` lifespan; enqueue `image_analyze` job after `image_extract` completes in `image_extractor.py`
- **API** — no new endpoints (existing `/images/{id}/raw` and `/documents/{id}/images` already serve description/image_type via `ImageItem`)
- **Frontend** — Chat.tsx: parse `image_ids` from the SSE `done` payload; render thumbnail strip below answer text

### Step 5 — [Service] Create image_enricher.py

Create `backend/app/services/image_enricher.py`. Key design decisions:

**Pre-filter (pure function, no LLM call):**
```python
def _is_decorative(pil_image) -> bool:
    """Return True if image has <5 unique colors or aspect ratio >10:1."""
    img_rgb = pil_image.convert("RGB")
    pixels = list(img_rgb.getdata())
    unique_colors = len(set(pixels))
    if unique_colors < 5:
        return True
    w, h = pil_image.size
    ratio = max(w, h) / max(min(w, h), 1)
    return ratio > 10.0
```

**Vision LLM call via LiteLLM:**
```python
import base64, litellm

async def _call_vision_llm(image_path: Path, settings) -> dict:
    with open(image_path, "rb") as f:
        b64 = base64.b64encode(f.read()).decode()
    prompt = (
        "Task 1: Classify this image as one of: architecture_diagram, "
        "sequence_diagram, er_diagram, flowchart, code_screenshot, table, "
        "chart, photo, decorative, other. "
        "Task 2: List every labeled component visible and any connections between them. "
        "Task 3: Extract any visible text verbatim. "
        'Respond in JSON: {"image_type": "...", "components": [...], "text": "...", "description": "..."}'
    )
    response = await litellm.acompletion(
        model=settings.VISION_MODEL,
        messages=[{
            "role": "user",
            "content": [
                {"type": "text", "text": prompt},
                {"type": "image_url", "image_url": {"url": f"data:image/png;base64,{b64}"}},
            ],
        }],
        temperature=0.0,
    )
    raw = response.choices[0].message.content
    # Parse JSON; fall back to {"image_type": "other", "description": raw, "components": [], "text": ""}
    ...
```

**Offline degradation:** catch `litellm.ServiceUnavailableError` and re-raise — the worker will catch it and mark the job `failed`. Log a warning. Document remains accessible.

**`ImageEnricherService.enrich(document_id)` method:**
1. Load all `ImageModel` rows for `document_id` where `description IS NULL`
2. For each image:
   a. Open PIL Image from absolute path
   b. Call `_is_decorative()` — if True, update `image_type='decorative'` in DB; skip LLM
   c. Call `_call_vision_llm()` — parse JSON response
   d. Update `ImageModel.description` and `ImageModel.image_type` in SQLite via UPDATE
   e. Embed description using `get_embedding_service().encode([description])[0]`
   f. Call `get_lancedb_service().upsert_image_vector(image_id, document_id, description, vector)`
   g. Write to `images_fts` via raw SQL: `INSERT INTO images_fts(image_id, document_id, body) VALUES (:image_id, :document_id, :body)`
3. One SQLAlchemy session per batch; use `asyncio.gather` with `Semaphore(3)` across images to avoid OOM on large documents

**`image_analyze_handler(document_id, job_id)` coroutine** — thin wrapper that instantiates `ImageEnricherService` and calls `enrich(document_id)`. All imports are lazy (`# noqa: PLC0415`). Follows the same pattern as `image_extract_handler`.

### Step 8 — [Service] Extend retriever.py to search image_vectors and images_fts

Add two new private methods to `HybridRetriever`:

**`image_vector_search(query_vector, document_ids, k, threshold=0.5) -> list[str]`**
- Calls `get_lancedb_service().search_image_vectors(query_vector, document_ids, k, threshold)`
- Returns list of image_ids

**`image_keyword_search(query, document_ids, k) -> list[str]`** (async)
- Sanitizes query via existing `_sanitize_fts_query()`
- Queries `images_fts` with `MATCH :query` and optional `document_id IN (...)` filter
- Returns list of image_ids

Modify `retrieve()` to call both image search methods and include results. Return signature changes: create `RetrievalResult` dataclass or return a tuple. Given the existing callers of `retrieve()` use `list[ScoredChunk]`, the cleanest approach is:

Add `image_ids: list[str]` to `ChatState` (done in Step 4), then extend the return type of `retrieve()` to return `tuple[list[ScoredChunk], list[str]]` (chunks, image_ids). This is a breaking change to `HybridRetriever.retrieve()`.

All callers of `retrieve()` must be updated:
- `backend/app/runtime/chat_graph.py` — `search_node`, `comparative_node`, `augment_node` call `retrieve()`
- `backend/app/services/gap_detector.py` — calls `retrieve()`
- `backend/app/services/note_search.py` (if it calls retrieve)
- Any test files that call `retrieve()` directly

Check actual callers before implementing.

Alternative (less disruptive): keep `retrieve()` returning `list[ScoredChunk]` and add a separate `retrieve_with_images()` method that returns the tuple. Only `search_node` in chat_graph.py calls the new method. This is preferred to minimize blast radius.

Use `retrieve_with_images()` approach:

```python
async def retrieve_with_images(
    self,
    query: str,
    document_ids: list[str] | None,
    k: int,
) -> tuple[list[ScoredChunk], list[str]]:
    """Hybrid retrieval returning chunks and matched image_ids."""
    chunks = await self.retrieve(query, document_ids, k)
    # Image search runs in parallel with chunk retrieval (after embedder already ran for chunks)
    query_vector = get_embedding_service().encode([query])[0]
    image_ids_vec = self.image_vector_search(query_vector, document_ids, k=5, threshold=0.5)
    image_ids_fts = await self.image_keyword_search(query, document_ids, k=5)
    # Deduplicate, preserving vector-match order first
    seen: set[str] = set()
    image_ids: list[str] = []
    for iid in image_ids_vec + image_ids_fts:
        if iid not in seen:
            seen.add(iid)
            image_ids.append(iid)
    return chunks, image_ids
```

## DB Schema Changes

### images_fts (new SQLite FTS5 virtual table)
```sql
CREATE VIRTUAL TABLE IF NOT EXISTS images_fts
USING fts5(
    body,
    image_id UNINDEXED,
    document_id UNINDEXED
)
```
- `body` is the indexed description text
- `image_id` and `document_id` are UNINDEXED (metadata only)
- Rowid-based deletion per the FTS5 UNINDEXED column pattern from MEMORY.md

### image_vectors_v1 (new LanceDB table)
```
image_id      string (merge key)
document_id   string
description   string
vector        float32[384]
```
- Dimension 384 matches the existing bge-small-en-v1.5 embedder
- Table name includes `_v1` suffix following the established LanceDB versioning pattern

### images (existing SQLite table — no schema changes)
`image_type` and `description` columns already exist in `ImageModel` (added in S133 with nullable=True). No ALTER TABLE needed.

## Risk Flags

- **Risk: litellm vision call format for Ollama llava.** LiteLLM's multimodal message format with base64 inline images (`data:image/png;base64,...`) must match what the Ollama llava model accepts. The story description specifies this format explicitly. Verify against the LiteLLM docs for `ollama/llava:13b`. If base64 inline fails, try `image_url` with a localhost URL pointing to the `/images/{id}/raw` endpoint. Mitigation: integration test with skipif guard catches this; unit tests mock the call entirely.

- **Risk: retrieve() caller count.** Adding `retrieve_with_images()` as a new method avoids breaking existing callers. Confirm that `gap_detector.py`, `note_search.py`, and all other callers use `retrieve()` (not `retrieve_with_images()`). Read each caller before modifying. Do not change `retrieve()` return signature.

- **Risk: FTS5 rowid vs content_hash collision.** The `images_fts` table uses rowid for delete (per the established pattern). The `image_id` is stored as UNINDEXED. For deletion on document delete, a new code path must be added to `routers/documents.py` delete endpoint: `DELETE FROM images_fts WHERE rowid IN (SELECT rowid FROM images_fts WHERE image_id = :iid)` for each image. This must be done or orphaned FTS5 rows accumulate. Mark as a required change even though the delete flow is not an AC.

- **Risk: Memory usage for large PDFs.** Opening all images as PIL and base64-encoding them for the LLM can exhaust memory on PDFs with 50+ images. The `Semaphore(3)` in `enrich()` limits concurrent processing but images are still opened sequentially. Mitigation: process one at a time in a loop (not `asyncio.gather`) if memory is a concern; start with `Semaphore(3)` and the integration test will reveal OOM.

- **Risk: Vector dimension mismatch.** The embedder is `bge-small-en-v1.5` producing 384-dim vectors. The `image_vectors_v1` schema must use `pa.list_(pa.float32(), 384)`. If a future story changes the embedder, this table schema will be versioned out (bump to `_v2`). Do not hardcode 1024 — the ARCHITECTURE.md docs are outdated on this point; the source of truth is `vector_store.py` line 23 which shows 384.

- **Risk: ChatState TypedDict exhaustiveness.** Adding `image_ids: list[str]` to ChatState requires updating every site that constructs the initial state dict. Search for all `ChatState(` and dict-literal state initializations in `chat_graph.py` and `qa.py`. Missing field causes a TypedDict KeyError at runtime.

- **Risk: Frontend TypeScript strict mode.** `image_ids` in the SSE payload is `string[] | undefined`. The parse must default to `[]` before assigning to `ChatMessage.image_ids`. Ensure `tsc --noEmit` passes after the Chat.tsx changes.

- **Risk: images_fts delete on document deletion.** The existing document delete path in `routers/documents.py` deletes ImageModel rows manually (no FK cascade per TD-010). After S134 it must also clear `images_fts` rows and `image_vectors` LanceDB rows for deleted images. Add this to the implementation scope — failure to do so causes retrieval to return stale image references.
