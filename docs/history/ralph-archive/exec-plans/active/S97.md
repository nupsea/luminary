## Risk Flags

- Risk: `asyncio.Semaphore` + `asyncio.gather` error propagation — `litellm.ServiceUnavailableError` raised inside `_generate_one` propagates out of `gather` and is re-raised at the `generate_from_gaps` call site. Pattern verified against `patterns.md` (`litellm.ServiceUnavailableError must be caught at the asyncio.gather call site`). The service does NOT catch it internally; it propagates to the router which maps it to 503. (mitigation: tested via `test_from_gaps_api_503_ollama_offline`)

- Risk: `datetime.utcnow()` vs `datetime.now(UTC)` — the existing `generate()` and `generate_from_notes()` methods still use `datetime.utcnow()` (TD-002 in tech-debt-tracker). `generate_from_gaps()` correctly uses `datetime.now(UTC)`. No regression introduced. (mitigation: no change to existing methods)

- Risk: `deck` column migration — existing databases without the `deck` column are handled by the additive migration in `db_init.py` which is wrapped in a try/except. Cards inserted before the migration will have the SQLite default `'default'`. (mitigation: migration already in place and tested via in-memory DB in tests)

- Risk: Frontend `GapResultCard` uses `document_id?: string` from props and `data.document_id?: string` from the gap card data, preferring `data.document_id ?? documentId ?? ""`. This is correct: the chat-injected card carries `document_id` from the scope of the chat session. No change needed. (mitigation: existing component logic)

- Risk: TypeScript strict mode — `GapResultCard.tsx` uses `(await res.json()) as { detail?: string }` and `(await res.json()) as { created: number }` which are type assertions, not `any`. `tsc --noEmit` must pass. (mitigation: assertions are explicit; no `any` usage)
