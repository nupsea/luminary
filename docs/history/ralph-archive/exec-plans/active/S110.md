## Database Schema

Exact SQL for the `reading_progress` table (created via SQLAlchemy ORM, not raw DDL):

```sql
CREATE TABLE reading_progress (
    id          TEXT PRIMARY KEY,        -- UUID
    document_id TEXT NOT NULL,
    section_id  TEXT NOT NULL,
    first_seen_at  DATETIME NOT NULL,
    last_seen_at   DATETIME NOT NULL,
    view_count  INTEGER NOT NULL DEFAULT 1
);
CREATE UNIQUE INDEX IF NOT EXISTS uq_reading_progress_doc_section
    ON reading_progress (document_id, section_id);
```

The unique index enables upsert semantics: `INSERT OR IGNORE` for first visit, then `UPDATE ... SET last_seen_at=:now, view_count=view_count+1 WHERE document_id=:doc_id AND section_id=:sec_id` for repeat visits. This pair of statements runs inside a single session transaction (no SQLite UPSERT ON CONFLICT is needed because aiosqlite supports it, but the two-statement approach is consistent with the existing patterns used in `notes_fts`).

**Cascade-delete**: `reading_progress` rows must be deleted when the parent document is deleted. This is added to the manual-delete sequences in `delete_document` and `bulk_delete_documents` in `documents.py` (FK pragma pattern consistent with existing code — see TD-010).

### Step 3 — [API] Create `backend/app/routers/reading.py`

New file. Pattern follows `notes.py`.

```python
import logging
import uuid
from datetime import UTC, datetime

from fastapi import APIRouter, HTTPException
from pydantic import BaseModel
from sqlalchemy import func, select

from app.database import get_session_factory
from app.models import DocumentModel, ReadingProgressModel, SectionModel

logger = logging.getLogger(__name__)

router = APIRouter(prefix="/reading", tags=["reading"])


class ReadingProgressRequest(BaseModel):
    document_id: str
    section_id: str


class ReadingProgressResponse(BaseModel):
    document_id: str
    section_id: str
    view_count: int
    first_seen_at: datetime
    last_seen_at: datetime


@router.post("/progress", response_model=ReadingProgressResponse, status_code=200)
async def upsert_reading_progress(body: ReadingProgressRequest) -> ReadingProgressResponse:
    """Mark a section as seen. Increments view_count on repeat visits."""
    async with get_session_factory()() as session:
        # Validate document exists
        doc = (await session.execute(
            select(DocumentModel.id).where(DocumentModel.id == body.document_id)
        )).scalar_one_or_none()
        if doc is None:
            raise HTTPException(status_code=404, detail="Document not found")

        now = datetime.now(UTC)

        existing = (await session.execute(
            select(ReadingProgressModel).where(
                ReadingProgressModel.document_id == body.document_id,
                ReadingProgressModel.section_id == body.section_id,
            )
        )).scalar_one_or_none()

        if existing is None:
            row = ReadingProgressModel(
                id=str(uuid.uuid4()),
                document_id=body.document_id,
                section_id=body.section_id,
                first_seen_at=now,
                last_seen_at=now,
                view_count=1,
            )
            session.add(row)
            await session.commit()
            await session.refresh(row)
            result = row
        else:
            existing.last_seen_at = now
            existing.view_count += 1
            await session.commit()
            await session.refresh(existing)
            result = existing

    logger.info(
        "Reading progress upsert",
        extra={
            "document_id": body.document_id,
            "section_id": body.section_id,
            "view_count": result.view_count,
        },
    )
    return ReadingProgressResponse(
        document_id=result.document_id,
        section_id=result.section_id,
        view_count=result.view_count,
        first_seen_at=result.first_seen_at,
        last_seen_at=result.last_seen_at,
    )
```

### Step 7 — [Frontend] IntersectionObserver in `DocumentReader.tsx`

Add a `useReadingProgress` hook inline at the top of the component file (before the `DocumentReader` export). This hook:

1. Takes `documentId: string` and `sections: SectionItem[]` as parameters.
2. Uses `useRef<Map<string, ReturnType<typeof setTimeout>>>` to track per-section dwell timers.
3. Creates an `IntersectionObserver` with `threshold: 0.5` (section half-visible = "in viewport").
4. On `intersecting = true` for a section element: starts a 3-second `setTimeout`. If it fires, calls `POST /reading/progress`.
5. On `intersecting = false`: clears the timer without firing.
6. Observes each `<li>` element using a `data-section-id` attribute. Uses `useEffect` to set up and tear down the observer when `documentId` or `sections` length changes.
7. Returns nothing (side-effect only hook).

Implementation sketch:

```typescript
const API_BASE = "http://localhost:8000"  // already defined at file top

async function postReadingProgress(documentId: string, sectionId: string): Promise<void> {
  await fetch(`${API_BASE}/reading/progress`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ document_id: documentId, section_id: sectionId }),
  })
}

function useReadingProgress(documentId: string, sections: SectionItem[]) {
  const timers = useRef<Map<string, ReturnType<typeof setTimeout>>>(new Map())

  useEffect(() => {
    if (sections.length === 0) return

    const observer = new IntersectionObserver(
      (entries) => {
        for (const entry of entries) {
          const sectionId = (entry.target as HTMLElement).dataset["sectionId"]
          if (!sectionId) continue

          if (entry.isIntersecting) {
            if (!timers.current.has(sectionId)) {
              const t = setTimeout(() => {
                timers.current.delete(sectionId)
                void postReadingProgress(documentId, sectionId)
              }, 3000)
              timers.current.set(sectionId, t)
            }
          } else {
            const t = timers.current.get(sectionId)
            if (t !== undefined) {
              clearTimeout(t)
              timers.current.delete(sectionId)
            }
          }
        }
      },
      { threshold: 0.5 },
    )

    // Observe all section elements via data-section-id attribute
    const elements = document.querySelectorAll("[data-section-id]")
    for (const el of elements) observer.observe(el)

    return () => {
      observer.disconnect()
      for (const t of timers.current.values()) clearTimeout(t)
      timers.current.clear()
    }
  }, [documentId, sections.length])  // re-run only when sections change
}
```

In `DocumentReader`, call the hook after `doc` is confirmed non-null:


## Risk Flags

- **Risk: `reading_progress_pct` in `list_documents()` adds 2 extra scalar subqueries per document** (one for `section_count`, one for `read_section_count`). The existing list endpoint already runs 5 correlated subqueries per row. Adding 2 more increases to 7. For < 100 documents this is acceptable. For larger libraries, these should be aggregated into a single JOIN. Mitigation: document as a future optimisation in tech-debt-tracker if this causes visible latency.

- **Risk: `document.querySelectorAll("[data-section-id]")` inside `useEffect` may miss elements if React has not committed the DOM yet**. The effect runs after render (post-commit), so all `<li data-section-id>` elements will exist. However, if sections are re-rendered asynchronously (e.g. pagination within the reader), the observer will not pick up new elements. Mitigation: re-run the effect when `sections.length` changes (already in the dependency array).

- **Risk: IntersectionObserver fires on initial render immediately if the first section is already visible** — a 3-second timer will trigger a progress POST within 3 seconds of opening the document. This is the intended behaviour per the story description ("sections they scroll past and dwell on >= 3 seconds get marked as read").

- **Risk: `reading_progress_pct` field added to `DocumentListItem` requires all existing usages of that type to compile.** Both `DocumentCard` and `DocumentRow` receive `DocumentListItem` as a prop. Adding the field to the interface is not breaking — TypeScript will error only if we forget to include it. The backend always returns it (computed as `0.0` when no sections read), so the field is always present in the JSON response. Mitigation: add the field to both the TypeScript interface and the backend response model in the same commit.

- **Risk: Cascade-delete coverage**. TD-010 notes that SQLite FK pragma is not globally enforced. The plan manually adds `ReadingProgressModel` to the delete sequences in both `delete_document` and `bulk_delete_documents`. If a new delete path is added in a future story without updating it, orphaned rows will accumulate. Mitigation: add a comment above the model in `models.py` noting that new delete paths must include this table.

- **Risk: No `section_count` guard** — if a document has 0 sections (e.g. parsing failed), `reading_progress_pct` must return `0.0` without a division-by-zero error. The plan handles this with `(read_count / section_count) if section_count > 0 else 0.0`. Verify this guard is in both `get_document()` and `list_documents()`.
