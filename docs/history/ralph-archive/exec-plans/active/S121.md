## Risk Flags

- Risk: `asyncio.create_subprocess_exec` for ffmpeg is called inside an async node that itself runs inside LangGraph's async executor. The existing audio tests mock this at the `asyncio.create_subprocess_exec` level and pass. No regression expected. (Mitigation: existing test_transcribe_node_video_with_ffmpeg covers this path.)
- Risk: The `/tmp/{doc_id}_audio.wav` temp file is deleted after transcription. If the OS reclaims `/tmp` aggressively (some containers), this path may not exist. (Mitigation: `wav_path.unlink(missing_ok=True)` is already used, which is safe. If the file was never created, unlink is still a no-op due to `missing_ok=True`.)
- Risk: `GET /documents/{id}/status` currently hardcodes the error string, meaning any ffmpeg-specific error message is silently dropped. The fix (Step 1) is a one-line change with no schema migration needed.
- Risk: The frontend `UploadDialog` already reads `data.error_message` from the status response (line 205 in `UploadDialog.tsx`). After Step 1, the ffmpeg message will propagate automatically. No frontend change required.
