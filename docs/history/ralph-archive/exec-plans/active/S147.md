# S147 — Text Selection Action Bar

## Goal

Replace the existing FloatingToolbar (Explain + Highlight only) in DocumentReader with a richer SelectionActionBar that fires on text selection across both the section list and the PDF text layer, offering five actions: Explain, Add to Note, Create Flashcard, Ask in Chat, and Highlight.

## Layers Affected

- Frontend (components layer): new SelectionActionBar, new NoteCreationDialog (thin wrapper; distinct from the existing NoteEditorDialog which edits existing notes), new DocumentFlashcardDialog (distinct from the notes-scoped GenerateFlashcardsDialog)
- Frontend (store layer): new `chatPreload` field added to Zustand store
- Frontend (pages layer): Chat.tsx reads chatPreload from store on mount
- No backend changes required

## Files to Create or Modify

### New files

- `frontend/src/components/reader/SelectionActionBar.tsx` — the core new component; typed Props interface; listens on the container ref for mouseup/touchend; positions 8px above selection via getBoundingClientRect; Escape + outside-mousedown dismiss; fires 5 actions
- `frontend/src/components/reader/NoteCreationDialog.tsx` — modal Dialog that wraps a simple textarea pre-filled with a blockquote of the selected text and a source attribution line; calls POST /notes on save; distinct from NoteEditorDialog (which edits existing notes and has tag suggest flows)
- `frontend/src/components/reader/DocumentFlashcardDialog.tsx` — modal Dialog that generates flashcards for a document section; accepts `documentId`, `sectionId | undefined`, `context: string`; calls POST /flashcards/{documentId}; distinct from GenerateFlashcardsDialog (which is note-scoped)
- `scripts/smoke/S147.sh` — smoke test: POST /annotations returns 2xx

### Modified files

- `frontend/src/store.ts` — add `chatPreload: { text: string; documentId: string | null } | null` field + `setChatPreload` setter + `clearChatPreload` setter
- `frontend/src/components/reader/DocumentReader.tsx` — replace `<FloatingToolbar>` + `<AnnotationPopover>` usage with `<SelectionActionBar>`; add state for NoteCreationDialog and DocumentFlashcardDialog; attach the single containerRef that spans both section list and PDF viewer; pass sourceRef resolver
- `frontend/src/pages/Chat.tsx` — on mount, read `chatPreload` from store, populate `input` state with `chatPreload.text`, call `clearChatPreload`; focus textarea

### Step 2 — [Component] Create SelectionActionBar.tsx

File: `frontend/src/components/reader/SelectionActionBar.tsx`

Props interface (TypeScript strict, no `any`):
```typescript
export interface SourceRef {
  sectionId: string | undefined  // undefined when selection is from PDF page with no section mapping
  documentId: string
  documentTitle: string
}

export interface SelectionActionBarProps {
  containerRef: React.RefObject<HTMLElement | null>
  documentId: string
  documentTitle: string
  /** Called with the section_id of the nearest [data-section-id] ancestor, or undefined for PDF text layer */
  resolveSourceRef: (startContainer: Node) => SourceRef
  onExplain: (text: string, mode: ExplainMode) => void
  onAddToNote: (text: string, sourceRef: SourceRef) => void
  onCreateFlashcard: (text: string, sourceRef: SourceRef) => void
  onAskInChat: (text: string, sourceRef: SourceRef) => void
  onHighlight: (text: string, sourceRef: SourceRef) => void
}
```

Internal state:
- `position: { top: number; left: number } | null`
- `selectedText: string`
- `pendingSourceRef: SourceRef | null`
- `toolbarRef: React.RefObject<HTMLDivElement>`

Selection detection (in a `useEffect` on `containerRef`):
1. Attach `mouseup` listener to `containerRef.current`; also attach `touchend` listener for mobile
2. On mouseup/touchend: wrap in `setTimeout(() => ..., 10)` (same pattern as existing FloatingToolbar) to allow selection to settle
3. Call `window.getSelection()`, get `toString().trim()`
4. If empty: reset state and return
5. Call `getRangeAt(0).getBoundingClientRect()` for range rect; call `containerRef.current.getBoundingClientRect()` for container rect
6. Position: `top = rect.top - containerRect.top - 8 - TOOLBAR_HEIGHT`, `left = rect.left - containerRect.left + rect.width / 2`
   - Note: use `8` not `44` (story says "8px above"); TOOLBAR_HEIGHT is approximately 36px (one row of buttons at py-1)
   - Final: `top = rect.top - containerRect.top - 44`, using `44` to match: 8px gap + ~36px bar height; this keeps the bar fully visible above the selection, not just 8px gap to bottom of bar
   - Implementation note: the bar must appear **above** the selection; position the bar's bottom edge 8px above the selection top. Using CSS `transform: translateY(-100%)` on the bar and setting `top = rect.top - containerRect.top - 8` achieves this cleanly. See Risk 2.
7. Resolve `sourceRef` from `selection.getRangeAt(0).startContainer`
8. Set position, selectedText, pendingSourceRef

Dismiss logic:
- `document.addEventListener("keydown", ...)`: on `Escape` key, reset state
- `document.addEventListener("mousedown", ...)`: if `!toolbarRef.current?.contains(e.target)` reset state

Button click handlers (each calls `e.preventDefault()` to avoid clearing selection before the handler fires; then resets bar state):
- **Explain**: calls `onExplain(selectedText, "plain")`
- **Add to Note**: calls `onAddToNote(selectedText, pendingSourceRef)`
- **Create Flashcard**: calls `onCreateFlashcard(selectedText, pendingSourceRef)`
- **Ask in Chat**: calls `onAskInChat(selectedText, pendingSourceRef)`
- **Highlight**: calls `onHighlight(selectedText, pendingSourceRef)` — only enabled when `pendingSourceRef.sectionId` is defined (PDF pages without section mapping cannot get offset-based highlights); button still visible but shows a tooltip "Not available in PDF view without section mapping" when sectionId is undefined

Positioning: `absolute z-50`, `transform: translateX(-50%) translateY(-100%)` (centered horizontally, appears above selection), `marginTop: -8px` gap. Container must be `position: relative`.

### Step 5 — [Component] Modify DocumentReader.tsx

File: `frontend/src/components/reader/DocumentReader.tsx`

Changes:

a) **Extend containerRef to cover PDF viewer**: The current `sectionListRef` only covers the section list div. The story requires selections from PDFViewer to also trigger the bar. The fix: use a **new outer containerRef** that wraps both the section list div and the hidden PDF viewer div. Rename existing `sectionListRef` to `outerLeftPanelRef` and attach it to the left-panel `div.flex.w-3/5` container instead. This single ref covers both `leftTab === "sections"` and `leftTab === "pdfview"` subtrees.

   - The left panel container div (`div.flex.w-3/5.flex-col.overflow-hidden.border-r`) does NOT need to be `position: relative` itself; instead, make the inner content area that holds SelectionActionBar `relative`. Specifically, the div with `ref={sectionListRef}` is already `relative`. For the PDF view, the `<PDFViewer>` container will need to be wrapped in a `relative` div, or the parent left-panel div must become `position: relative`.

   - Best approach: keep a `readerContainerRef` on the outermost `div.flex.w-3/5` left panel, and inside `SelectionActionBar`, compute position relative to this container. Also ensure this container has `position: relative`.

b) **Replace FloatingToolbar + AnnotationPopover**:
   - Remove imports: `FloatingToolbar`, `AnnotationPopover`, types `ExplainMode` and `HighlightInfo` from FloatingToolbar
   - Import `ExplainMode` from `ExplanationSheet` or redefine locally (it is currently exported from FloatingToolbar -- this must be re-exported or moved)
   - Import `SelectionActionBar`, `NoteCreationDialog`, `DocumentFlashcardDialog`
   - Remove state: `pendingHighlight: HighlightInfo | null`
   - Add state:
     - `selectionNoteOpen: boolean`
     - `selectionNoteText: string`
     - `selectionNoteSourceRef: SourceRef | null`
     - `selectionFlashcardOpen: boolean`
     - `selectionFlashcardText: string`
     - `selectionFlashcardSourceRef: SourceRef | null`
   - Add store selectors: `setChatPreload` from `useAppStore`
   - Implement `resolveSourceRef(startContainer: Node): SourceRef`:
     - Walk up DOM from `startContainer` looking for `[data-section-id]` element
     - If found: return `{ sectionId: el.dataset.sectionId, documentId, documentTitle: doc.title }`
     - If not found (PDF text layer): return `{ sectionId: undefined, documentId, documentTitle: doc.title }`

c) **Implement action handlers**:
   - `onExplain(text, mode)`: existing `handleExplain` logic -- open ExplanationSheet
   - `onAddToNote(text, sourceRef)`: set `selectionNoteText`, `selectionNoteSourceRef`, `selectionNoteOpen = true`
   - `onCreateFlashcard(text, sourceRef)`: set `selectionFlashcardText`, `selectionFlashcardSourceRef`, `selectionFlashcardOpen = true`

   - `onHighlight(text, sourceRef)`: if `sourceRef.sectionId` is undefined, do nothing (PDF without section); otherwise POST /annotations directly: `{ document_id: documentId, section_id: sourceRef.sectionId, chunk_id: null, selected_text: text, start_offset: 0, end_offset: text.length, color: "yellow", note_text: null }`, then invalidate annotations query

d) **Placement**: `<SelectionActionBar>` goes inside the `readerContainerRef` div (the left 3/5 panel), rendered once (not inside the conditional leftTab branches).

e) **NoteCreationDialog and DocumentFlashcardDialog** rendered at bottom of the DocumentReader JSX tree, outside the two-panel layout, similar to existing FeynmanDialog and ExplanationSheet.

f) **FloatingToolbar**: Remove from DocumentReader entirely. The FloatingToolbar component file remains in place (it may be used elsewhere); only the import and usage in DocumentReader.tsx is removed.

## Risk Flags

- **Risk 1: ExplainMode type import chain breaks**. `ExplainMode` is currently exported from `FloatingToolbar.tsx` and imported by both `DocumentReader.tsx` and `ExplanationSheet.tsx`. When FloatingToolbar is removed from DocumentReader, the import path changes. Mitigation: Do NOT delete FloatingToolbar.tsx. Move `ExplainMode` export to a shared location, OR keep importing it from FloatingToolbar. The cleanest path is to keep the type in FloatingToolbar (it still exists in the codebase) and import only the type `type { ExplainMode }` in SelectionActionBar and DocumentReader. ExplanationSheet already imports from FloatingToolbar and should not be changed.

- **Risk 2: Bar positioning — translateY vs absolute top offset**. The current FloatingToolbar uses `top = rect.top - containerRect.top - 44` which hardcodes 44px (approximate bar height + 8px gap). This works but is fragile if the bar is taller than expected. A cleaner approach: position the div at `top = rect.top - containerRect.top - 8` and add `style={{ transform: 'translateX(-50%) translateY(-100%)', marginBottom: '8px' }}` (this requires CSS adjustment). The 5-button bar is wider than the 3-button FloatingToolbar; measure the rendered height. Fallback: keep the same `top - 44` pattern and add a comment. Implementer should measure the bar at 5 buttons and adjust the constant accordingly.

- **Risk 3: Outer containerRef for PDF layer selections**. The PDF text layer renders inside PDFViewer which is inside `div.hidden` when `leftTab !== "pdfview"`. Attaching the mouseup listener to the outer left panel div ensures capture from both branches. However, the `position: relative` is needed on the container for absolute positioning of SelectionActionBar to work correctly. The outer left panel div (currently `div.flex.w-3/5.flex-col.overflow-hidden.border-r`) must have `relative` added to its className. Verify this does not break the existing AnnotationPopover layout (which was positioned relative to `sectionListRef`).

- **Risk 4: Highlight color for direct POST from SelectionActionBar**. The AC says `color: '#fef08a'`. The existing `AnnotationItem` type defines `color` as `"yellow" | "green" | "blue" | "pink"`. The existing POST /annotations endpoint schema expects one of those string values. Using `#fef08a` will fail validation. Options: (a) use `"yellow"` as the color (which maps to yellow-200 in the highlight reconstruction logic); (b) add `"selection"` as a fifth color option. The story description says `color='#fef08a'` but the backend already has `"yellow"` mapped to the same visual. Resolution: use `"yellow"` as the color value and add a code comment explaining the story's hex corresponds to the "yellow" annotation color. If the AC is tested strictly for `#fef08a`, the backend type and frontend type must both be updated to support hex colors. The safer path (no backend changes required per the story) is to use `"yellow"`.

- **Risk 5: NoteCreationDialog vs NoteEditorDialog naming confusion**. The story says "NoteEditorDialog: New modal dialog wrapper" but `NoteEditorDialog.tsx` already exists and is a different component (edits existing notes). This plan creates `NoteCreationDialog.tsx` instead to avoid collision and naming confusion. The implementer must verify the story's intent: the pre-filled dialog described in the AC is creation (POST /notes), not editing (PATCH /notes).

- **Risk 6: DocumentFlashcardDialog endpoint verification**. Before implementing Step 4, read `backend/app/api/flashcards.py` to confirm `POST /flashcards/{documentId}` accepts `context` and `section_id` fields. If the endpoint does not yet accept `context` as a pre-fill hint (it may only accept `count` and `scope`), the dialog should generate flashcards and show them, passing `context` as a `section_id`-scoped request without relying on the `context` param. Fallback: generate with `section_id` from `sourceRef.sectionId` if present.

- **Risk 7: Double-mount guard not needed here**. Unlike the SSE LibraryOverview (S84), SelectionActionBar uses `addEventListener` with cleanup via `removeEventListener` in the useEffect return. React strict mode double-mount is handled correctly by the cleanup function.

- **Risk 8: touchend for mobile**. The story mentions `touchend`; the existing FloatingToolbar only handles `mouseup`. Implement `touchend` listener identically to `mouseup` for completeness. On `touchend`, the selection state is updated after the browser processes the touch; add the same `setTimeout(..., 10)` delay.
