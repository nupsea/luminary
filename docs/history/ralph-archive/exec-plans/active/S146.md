# S146 -- PDF.js Native PDF Viewer with Text Layer, Page Navigation, and TOC Panel

## Goal

Add a native PDF rendering tab to DocumentReader that renders actual PDF pages via PDF.js (pdfjs-dist) with a text layer, virtual page rendering, page navigation, zoom control, and a TOC panel built from SectionModel data; also add two new backend endpoints (`GET /documents/{id}/file` and `GET /documents/{id}/pdf-meta`) and populate `ChunkModel.pdf_page_number` during PDF ingestion.

---

- **Types** -- ChunkModel gains `pdf_page_number` (nullable int); new Pydantic response model `PDFMetaResponse`
- **Config** -- no changes
- **Repo** -- no changes (migration is additive DDL in db_init.py)
- **Service** -- `_chunk_book()` and `_chunk_tech_book()` in `ingestion.py` must populate `pdf_page_number` from the parsed section's `page_start`; PDF-format detection needed for page number population
- **Runtime** -- no LangGraph graph changes
- **API** -- two new endpoints in `documents.py`: `GET /{document_id}/file` and `GET /{document_id}/pdf-meta`

### Step 3: [Service] Populate `pdf_page_number` in `ingestion.py`

File: `/Users/sethurama/DEV/LM/learning-mate/backend/app/workflows/ingestion.py`

The story requires `pdf_page_number` to be populated for chunks from PDF documents during ingestion. The page number should be 1-based (matching PDF page numbering conventions) and derived from the parsed section's `page_start` field, which is already populated by the PDF parser via `fitz`.

**Key insight from reading the code**: The ingestion workflow calls `_chunk_book()` (for content_type "book") and `_chunk_tech_book()` (for "tech_book", "tech_article"). The document format (pdf/txt/docx) is tracked in `state["format"]` and in `DocumentModel.format`. The sections carry `page_start` from the parser.

**Change in `_chunk_book()`**: When creating each `ChunkModel`, add `pdf_page_number` based on whether the document format is "pdf":

```python
# Inside the loop over sections in _chunk_book()
# Before creating ChunkModel, derive pdf_page_number
fmt = state.get("format", "").lower()
chunk_pdf_page = s.get("page_start", 0) if fmt == "pdf" else None
# page_start from the parser is 0-based (fitz page index + 1 in parser).
# Check parser.py: flush_section passes `next_page - 1` as page_end and
# `page_num + 1` as `current_page_start`; so page_start is already 1-based.
# If page_start is 0 for PDFs, convert 0 to 1 (first page).
if fmt == "pdf" and chunk_pdf_page == 0:
    chunk_pdf_page = 1

chunk_models.append(ChunkModel(
    id=chunk_id,
    document_id=doc_id,
    section_id=section_model.id,
    text=enriched_text,
    token_count=len(enriched_text.split()),
    page_number=s.get("page_start", 0),
    speaker=None,
    chunk_index=chunk_idx,
    pdf_page_number=chunk_pdf_page,  # NEW
))
```

**Verify parser page numbering**: In `parser.py` `_parse_pdf` fallback, the call is `flush_section(line_text, 1, page_num + 1)` where `page_num` is 0-indexed. So `page_start` is 1-indexed. In `book_parser.py` and `universal_parser.py`, the page_start convention should be verified -- those parsers set page numbers via `fitz` page indexes and should also produce 1-based values after `+ 1` conversion. The `page_start` stored in sections is already 1-based across all code paths.

**Change in `_chunk_tech_book()`**: Same pattern -- derive `chunk_pdf_page` from `state.get("format")` and `s.get("page_start", 0)`, set `pdf_page_number=chunk_pdf_page` on each ChunkModel. The existing ChunkModel constructor in this function already has `page_number=s.get("page_start", 0)`.

**Other content type paths** (conversation, notes, code, audio/video): These call `chunk_node` directly and create ChunkModel with `page_number=0` or `page_number=meta.get("start_line", 0)`. Leave `pdf_page_number=None` (default) for all non-book, non-tech-book paths. These content types are never PDF-format documents with meaningful page numbers.

**The `_chunk_book` function signature change**: `state` is already passed as the first arg, so `state.get("format")` is accessible. No signature changes are needed.

### Step 4: [API] Add `PDFMetaResponse` Pydantic model and two endpoints in `documents.py`

**Add `GET /{document_id}/file` endpoint** (after the existing `/{document_id}/video` endpoint for symmetry with audio/video):

```python
@router.get("/{document_id}/file")
async def serve_document_file(document_id: str) -> FileResponse:
    """Serve the raw uploaded file for a document.

    Used by the PDF.js viewer (S146) to stream PDF bytes.
    Returns 404 if the document does not exist in SQLite or the file is not
    found on disk. Returns the file with Content-Type: application/pdf for
    PDF documents.
    """
    async with get_session_factory()() as session:
        result = await session.execute(
            select(DocumentModel).where(DocumentModel.id == document_id)
        )
        doc = result.scalar_one_or_none()
    if doc is None:
        raise HTTPException(status_code=404, detail="Document not found")
    fp = Path(doc.file_path)
    if not fp.exists():
        raise HTTPException(status_code=404, detail="Document file not found on disk")
    # Serve with correct MIME type based on format
    mime_map = {
        "pdf": "application/pdf",
        "docx": "application/vnd.openxmlformats-officedocument.wordprocessingml.document",
        "txt": "text/plain",
        "md": "text/markdown",
        "epub": "application/epub+zip",
    }
    mime = mime_map.get(doc.format.lower(), "application/octet-stream")
    return FileResponse(str(fp), media_type=mime)
```

Note: no content_type restriction -- the endpoint serves any document's raw file, not just PDFs. The frontend only calls it for PDF documents. This is the correct design: the endpoint is generic, the caller is specific.

**Add `GET /{document_id}/pdf-meta` endpoint**:

```python
@router.get("/{document_id}/pdf-meta", response_model=PDFMetaResponse)
async def get_pdf_meta(document_id: str) -> PDFMetaResponse:
    """Return PDF metadata: page count and whether a TOC (sections) exists.

    Returns 404 if document not found.
    Returns 400 if the document is not a PDF (format != 'pdf').
    page_count: from DocumentModel.page_count (set during ingestion).
    has_toc: True when at least one SectionModel row exists for this document.
    """
    async with get_session_factory()() as session:
        result = await session.execute(
            select(DocumentModel).where(DocumentModel.id == document_id)
        )
        doc = result.scalar_one_or_none()
        if doc is None:
            raise HTTPException(status_code=404, detail="Document not found")
        if doc.format.lower() != "pdf":
            raise HTTPException(
                status_code=400,
                detail=f"Document is not a PDF (format={doc.format})",
            )
        section_count_result = await session.execute(
            select(func.count(SectionModel.id))
            .where(SectionModel.document_id == document_id)
        )
        section_count = section_count_result.scalar_one() or 0

    return PDFMetaResponse(
        page_count=doc.page_count,
        has_toc=section_count > 0,
    )
```

`func` is already imported. `SectionModel` is already imported.

### Step 7: [Frontend] Create `PDFViewer.tsx`

File: `/Users/sethurama/DEV/LM/learning-mate/frontend/src/components/reader/PDFViewer.tsx`

This is the most substantial new file. Key design decisions from the story spec:

**Architecture:**
- `pdfjsLib.getDocument({ url })` loads the PDF
- `workerSrc` set once on the `pdfjsLib.GlobalWorkerOptions` using `PDFJS_WORKER_URL`
- Only renders the visible page + one page ahead (virtual rendering -- not all pages upfront)
- Canvas + text layer overlay per rendered page
- TOC panel on the left shows SectionModel headings; clicking a section jumps to its `page_start`
- Zoom control: 50%-200% range; persisted in component state
- Keyboard arrow navigation (left/right arrows advance pages) via useEffect

**Component interface:**
```tsx
interface PDFViewerProps {
  documentId: string
  sections: SectionItem[]  // from DocumentReader's existing doc.sections
}
```

**State:**
```ts
const [pdfDoc, setPdfDoc] = useState<PDFDocumentProxy | null>(null)
const [currentPage, setCurrentPage] = useState(1)
const [totalPages, setTotalPages] = useState(0)
const [zoom, setZoom] = useState(1.0)  // 1.0 = 100%
const [loadStatus, setLoadStatus] = useState<"loading" | "error" | "ready">("loading")
const [pageInput, setPageInput] = useState("1")
```

**useEffect to load PDF:**
```ts
useEffect(() => {
  let cancelled = false
  setLoadStatus("loading")

  // Set worker once
  pdfjsLib.GlobalWorkerOptions.workerSrc = PDFJS_WORKER_URL

  const task = pdfjsLib.getDocument({ url: `${API_BASE}/documents/${documentId}/file` })
  task.promise.then((doc) => {
    if (cancelled) return
    setPdfDoc(doc)
    setTotalPages(doc.numPages)
    setLoadStatus("ready")
  }).catch(() => {
    if (!cancelled) setLoadStatus("error")
  })

  return () => {
    cancelled = true
    task.destroy()
  }
}, [documentId])
```

**Page rendering useEffect** (renders current page and prefetches current+1):
```ts
// canvasRef for current page, nextCanvasRef for next page
useEffect(() => {
  if (!pdfDoc || loadStatus !== "ready") return

  let cancelled = false

  async function renderPage(pageNum: number, canvas: HTMLCanvasElement) {
    const page = await pdfDoc!.getPage(pageNum)
    const viewport = page.getViewport({ scale: zoom })
    canvas.width = viewport.width
    canvas.height = viewport.height
    const ctx = canvas.getContext("2d")
    if (!ctx || cancelled) return
    await page.render({ canvasContext: ctx, viewport }).promise
    if (cancelled) return
    // Text layer: create a div overlay
    // ... (see text layer setup below)
  }

  void renderPage(currentPage, canvasRef.current!)
  if (currentPage < totalPages) {
    void renderPage(currentPage + 1, nextCanvasRef.current!)
  }

  return () => { cancelled = true }
}, [pdfDoc, currentPage, zoom, totalPages, loadStatus])
```

**Text layer**: Use `pdfjsLib.renderTextLayer()` with a div overlay positioned absolutely over the canvas. The text layer enables text selection (required by S147). Style the text layer div with `position: absolute; top: 0; left: 0; overflow: hidden; opacity: 0.2; line-height: 1.0;` -- standard PDF.js text layer approach.

**Navigation handlers:**
```tsx
function goToPage(n: number) {
  const clamped = Math.max(1, Math.min(n, totalPages))
  setCurrentPage(clamped)
  setPageInput(String(clamped))
}
function handlePrev() { goToPage(currentPage - 1) }
function handleNext() { goToPage(currentPage + 1) }
function handlePageInputCommit() {
  const n = parseInt(pageInput, 10)
  if (!isNaN(n)) goToPage(n)
}
```

**Keyboard navigation** via `useEffect`:
```ts
useEffect(() => {
  function onKey(e: KeyboardEvent) {
    if (e.key === "ArrowRight") handleNext()
    if (e.key === "ArrowLeft") handlePrev()
  }
  window.addEventListener("keydown", onKey)
  return () => window.removeEventListener("keydown", onKey)
}, [currentPage, totalPages])  // deps ensure stale closures do not persist
```

**TOC panel**: Left column showing `sections` prop. Clicking a section calls `goToPage(section.page_start)`. Sections with `page_start === 0` still show but jump to page 1 (fallback).

**Zoom control**: `<input type="range" min={50} max={200} step={10} value={zoom * 100} onChange={...}>` plus a display label showing the percentage.

**Three states (invariant -- mandatory):**
- **Loading**: `<Skeleton className="h-full w-full" />` (use `Skeleton` component, not `Loader2`) while `loadStatus === "loading"`
- **Error**: `<p className="text-sm text-destructive">Could not load PDF file. The document may not be available.</p>` when `loadStatus === "error"`
- **Empty**: N/A per story spec -- "PDF View" tab is hidden for non-PDF documents; the empty state is never reached inside `PDFViewer.tsx`

**Full component layout sketch**:
```tsx
// Outer: flex row
// Left: TOC panel (1/4 width) - section list with scroll
// Right: main viewer (3/4 width) - toolbar at top (nav + zoom), canvas below
```

**Import style**: Use `import * as pdfjsLib from "pdfjs-dist"` with `// @ts-expect-error -- pdfjs-dist types are bundled in package` if TypeScript resolution issues arise. PDF.js v4 ships with `pdf.d.ts` so types should be available without a separate `@types` package.

**Type for PDFDocumentProxy**: `import type { PDFDocumentProxy } from "pdfjs-dist"` -- available in pdfjs-dist v4.

## Risk Flags

- **Risk: pdfjs-dist v4 import style in Vite**. PDF.js v4 uses ES module exports. The import `import * as pdfjsLib from "pdfjs-dist"` should work with Vite 7 in ESM mode. If Vite's tree-shaking causes issues with the worker build, use a dynamic import: `const pdfjsLib = await import("pdfjs-dist")`. The CDN worker URL sidesteps the Vite worker bundling issue entirely. Mitigation: test `npm run build` after installing pdfjs-dist before writing any component code to catch bundler issues early.

- **Risk: pdfjs-dist types in TypeScript strict mode**. PDF.js v4 bundles its own TypeScript declarations. The `PDFDocumentProxy`, `PDFPageProxy`, `RenderTask` types are available. However, `renderTextLayer` API changed in v4 -- verify the current API signature from the package's `d.ts` before implementing the text layer. Mitigation: keep the text layer implementation minimal (canvas-only rendering still satisfies the story's AC); the text layer is needed for S147 selection events but S146 only says "text layer for selection", not that it must be fully interactive.

- **Risk: `page_start` values from the parser may be inconsistent**. The fallback PDF parser in `parser.py` uses `flush_section(line_text, 1, page_num + 1)` which makes `page_start` 1-based. However, `book_parser.py` and `universal_parser.py` may use different conventions. Read both files before implementing Step 3 to confirm the actual values. If `page_start` can be 0 for the first page, apply the `if fmt == "pdf" and chunk_pdf_page == 0: chunk_pdf_page = 1` guard. Mitigation: the test in Step 9 uses explicitly set `page_start` values [1..5] and asserts the exact set, so any off-by-one in the production code path will be caught.

- **Risk: `_chunk_book` does not currently receive `state` as first argument**. Inspecting the actual function signature: `async def _chunk_book(state: IngestionState, pd: dict | None, doc_id: str) -> IngestionState`. The `state` dict contains `"format"`. So `state.get("format", "")` is safe. No signature change needed.

- **Risk: `_chunk_tech_book` receives `state` but `format` is in `state["format"]`**. Same pattern -- `state.get("format", "").lower()` gives the file format. Verified from the function signature: `async def _chunk_tech_book(state: IngestionState, pd: dict | None, doc_id: str)`.

- **Risk: `SectionModel.page_start` values may be 0 for all sections in text-based PDFs** where the parser did not extract meaningful page numbers. For example, the book parser for `.txt` files always sets `page_start=0`. Only PDFs processed through `_parse_pdf` will have non-zero `page_start`. The `pdf_page_number` will be 1 (the `== 0` fallback) for all chunks in these cases, which is acceptable (the PDF viewer will scroll to page 1 for any TOC click). Mitigation: document this limitation with a code comment.

- **Risk: Tab state extensibility in DocumentReader**. The current `leftTab` state type is `"sections" | "highlights"`. Adding `"pdfview"` requires updating the type and all related conditional logic. Ensure that when `doc.format !== "pdf"` and `leftTab === "pdfview"`, the UI falls back gracefully to "sections". Add a `useEffect` guard:
  ```ts
  useEffect(() => {
    if (leftTab === "pdfview" && doc?.format !== "pdf") {
      setLeftTab("sections")
    }
  }, [leftTab, doc?.format])
  ```

- **Risk: Keyboard event listener conflict**. The `keydown` listener in `PDFViewer` captures all arrow key events globally, which can conflict with other inputs in `DocumentReader` (note editor text area, annotation input, etc.). Mitigation: check `e.target` in the keydown handler and skip if the event target is an input, textarea, or contenteditable element:
  ```ts
  if (
    e.target instanceof HTMLInputElement ||
    e.target instanceof HTMLTextAreaElement ||
    (e.target as HTMLElement).isContentEditable
  ) return
  ```

- **Risk: `func.count(SectionModel.id)` import in `get_pdf_meta`**. `func` is already imported in `documents.py` from `sqlalchemy`. `SectionModel` is already imported. No new imports needed for the `get_pdf_meta` endpoint.

- **Risk: `DocumentModel.page_count` may be 0 for PDFs ingested before this story**. The `page_count` field is set in `_parse_pdf` via `len(doc)` (fitz document length). For PDFs that were already ingested, `page_count` is already populated correctly. For re-ingested documents, the value will be refreshed. For historical documents, `page_count=0` in `PDFMetaResponse` is technically correct -- the PDF viewer will load the document and determine `totalPages` from pdfjs itself anyway (it calls `pdfDoc.numPages`). The `/pdf-meta` endpoint's `page_count` is informational only.

- **Risk: TypeScript `noUnusedLocals` / `noUnusedParameters`**. In `PDFViewer.tsx`, ensure all declared state variables and refs are actually used. TypeScript strict mode will reject unused locals. Pay attention to `nextCanvasRef` (pre-rendering one page ahead) -- if this ends up hidden (rendered but not visible), TypeScript still requires it to be used. The pre-rendering logic must actually assign to and use both canvas refs.

- **Risk: TD-003 (bundle size)**. Adding `pdfjs-dist` will significantly increase bundle size -- pdfjs-dist is ~1.5MB minified. This worsens the existing TD-003 tech debt. Mitigation: the `PDFViewer` component should be lazy-loaded via `React.lazy()` in `DocumentReader.tsx` to avoid adding to the initial bundle. Document this in the commit message and update `tech-debt-tracker.md` to note the expected bundle size increase.

  ```tsx
  // In DocumentReader.tsx -- lazy import PDFViewer
  const PDFViewer = React.lazy(() =>
    import("./PDFViewer").then((m) => ({ default: m.PDFViewer }))
  )
  // Wrap in <Suspense fallback={<Skeleton className="h-full w-full" />}>
  ```

## Key Invariants Verified

1. **uv only** -- no pip references in any step; frontend uses `npm install` (correct for JS packages).
2. **Layer order** -- Types (models.py) -> Repo (db_init.py migration) -> Service (ingestion.py) -> API (documents.py). No reverse imports. `PDFMetaResponse` Pydantic model lives in the router file (API layer), not in types.py, per the existing codebase pattern.

4. **Pydantic at every boundary** -- `PDFMetaResponse(BaseModel)` for the `/pdf-meta` endpoint. The `/file` endpoint returns `FileResponse` (binary stream, no Pydantic needed). All existing endpoint inputs remain Pydantic-typed.
5. **Tests required** -- `test_pdf_page_number.py` covers 6 test cases: chunk page number population, non-PDF null, /file 200, /file 404, /pdf-meta 200, /pdf-meta 400.
6. **Three frontend states** -- PDFViewer loading (Skeleton), error (inline text), empty (N/A -- tab hidden for non-PDF). DocumentReader's new "pdfviewvisited" pattern ensures Skeleton shows correctly while loading.
