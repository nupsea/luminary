### Step 4 -- [Repo] Extend `backend/app/services/graph.py` for PREREQUISITE_OF with source_section_id

**4a.** Update the `_create_schema` PREREQUISITE_OF statement to add `source_section_id`:

```python
"CREATE REL TABLE IF NOT EXISTS PREREQUISITE_OF("
"FROM Entity TO Entity, document_id STRING, confidence FLOAT, source_section_id STRING)",
```

IMPORTANT: Kuzu uses `CREATE REL TABLE IF NOT EXISTS` -- the `IF NOT EXISTS` guard means
existing databases do NOT pick up the new column. Wrap the schema change defensively:

- The `add_prerequisite` method (S117) must keep its existing signature to avoid breaking
  the regex-based pipeline.
- Add a new method `add_prerequisite_with_section(dependent_id, prerequisite_id,
  document_id, confidence, source_section_id)` that tries the 3-property CREATE first,
  falls back to the 2-property form (for old DBs missing `source_section_id`).

```python
def add_prerequisite_with_section(
    self,
    dependent_id: str,
    prerequisite_id: str,
    document_id: str,
    confidence: float,
    source_section_id: str,
) -> None:
    """Create PREREQUISITE_OF with source_section_id, fallback for old DBs."""
    result = self._conn.execute(
        "MATCH (a:Entity {id: $dep})-[r:PREREQUISITE_OF]->(b:Entity {id: $pre})"
        " WHERE r.document_id = $did RETURN r.confidence",
        {"dep": dependent_id, "pre": prerequisite_id, "did": document_id},
    )
    if not result.has_next():
        try:
            self._conn.execute(
                "MATCH (a:Entity {id: $dep}), (b:Entity {id: $pre})"
                " CREATE (a)-[:PREREQUISITE_OF {document_id: $did,"
                " confidence: $conf, source_section_id: $sid}]->(b)",
                {
                    "dep": dependent_id, "pre": prerequisite_id,
                    "did": document_id, "conf": confidence, "sid": source_section_id,
                },
            )
        except Exception:
            # Old DB without source_section_id column -- fall back to 2-property form
            self._conn.execute(
                "MATCH (a:Entity {id: $dep}), (b:Entity {id: $pre})"
                " CREATE (a)-[:PREREQUISITE_OF {document_id: $did, confidence: $conf}]->(b)",
                {"dep": dependent_id, "pre": prerequisite_id,
                 "did": document_id, "conf": confidence},
            )
```

**4b.** Add `get_entry_point_concepts(document_id, limit=3) -> list[dict]`:

Returns concepts that have no PREREQUISITE_OF outgoing edges (i.e. have no unsatisfied
prerequisites themselves -- they are the "root" concepts reachable without prerequisites).
Actually, "entry point" here means concepts that are the deepest roots: they are NOT
pointed to by any PREREQUISITE_OF edge in the document (i.e. nothing requires them as
prerequisites). These are the concepts with no incoming edges (they require nothing).

Wait -- the PREREQUISITE_OF direction is: `(dependent)-[:PREREQUISITE_OF]->(prerequisite)`.
Entry-point concepts have no OUTGOING PREREQUISITE_OF edges, meaning nothing is listed as
their prerequisite. They are safe starting concepts.

```python
def get_entry_point_concepts(self, document_id: str, limit: int = 10) -> list[str]:
    """Return entity names that have no outgoing PREREQUISITE_OF edges for this document.

    These are concepts with no listed prerequisites -- valid starting points.
    Returns at most `limit` names, ordered by MENTIONED_IN count desc.
    """
    try:
        # All concepts with MENTIONED_IN this doc
        all_result = self._conn.execute(
            "MATCH (e:Entity)-[r:MENTIONED_IN]->(d:Document {id: $did})"
            " RETURN e.id, e.name, r.count",
            {"did": document_id},
        )
        all_entities: dict[str, tuple[str, int]] = {}  # id -> (name, count)
        while all_result.has_next():
            row = all_result.get_next()
            all_entities[row[0]] = (row[1], int(row[2] or 1))

        # Concepts with outgoing PREREQUISITE_OF edges (they have prerequisites)
        dep_result = self._conn.execute(
            "MATCH (a:Entity)-[r:PREREQUISITE_OF]->(b:Entity)"
            " WHERE r.document_id = $did RETURN DISTINCT a.id",
            {"did": document_id},
        )
        has_prereqs: set[str] = set()
        while dep_result.has_next():
            has_prereqs.add(dep_result.get_next()[0])

        # Entry points: in the doc, have no outgoing prereq edges
        entry = [
            (eid, name, count)
            for eid, (name, count) in all_entities.items()
            if eid not in has_prereqs
        ]
        # But only include if they ARE pointed to by someone (i.e. someone depends on them)
        # so we skip concepts that have no PREREQUISITE_OF involvement at all
        referenced_result = self._conn.execute(
            "MATCH (a:Entity)-[r:PREREQUISITE_OF]->(b:Entity)"
            " WHERE r.document_id = $did RETURN DISTINCT b.id",
            {"did": document_id},
        )
        referenced: set[str] = set()
        while referenced_result.has_next():
            referenced.add(referenced_result.get_next()[0])

        entry_in_graph = [(eid, name, count) for eid, name, count in entry if eid in referenced]
        entry_in_graph.sort(key=lambda x: x[2], reverse=True)
        return [name for _, name, _ in entry_in_graph[:limit]]
    except Exception:
        logger.debug("get_entry_point_concepts failed", exc_info=True)
        return []
```

### Step 5 -- [Service] Create `backend/app/services/prereq_extractor.py`

New LLM-based service following the `reference_enricher.py` pattern:

**Service structure:**

```python
"""PrereqExtractorService: LLM-based prerequisite pair extraction from section summaries.

Registered as enrichment job_type='prerequisites'.
Reads SectionSummaryModel rows for a document, calls LLM per section,
writes PREREQUISITE_OF edges to Kuzu.
Sets SectionModel.difficulty_estimate = prerequisite chain depth.
"""

import json, logging, re, uuid
import litellm
from sqlalchemy import select, update
from app.config import get_settings
from app.database import get_session_factory
from app.models import SectionModel, SectionSummaryModel
from app.services.graph import get_graph_service

logger = logging.getLogger(__name__)

_SYSTEM_PROMPT = (
    "You are a technical curriculum analyst."
    " For the provided section summary, list concepts this section requires"
    " the reader to already understand."
    " Be specific. Use the exact concept names from the text."
    " Output a JSON array only, no prose:"
    ' [{"requires": "concept name", "required_by": "concept name", "confidence": 0.0}].'
    " Only include prerequisites with confidence >= 0.7."
    " Return [] if no prerequisites apply."
)
```

Key methods:

**`extract(section_summary_content, section_id, document_id) -> list[dict]`**:
- Calls `litellm.acompletion` with the system prompt and section content
- Parses JSON array `[{requires, required_by, confidence}]`
- Filters to confidence >= 0.7
- Returns [] on any failure (non-fatal)
- Raises `litellm.ServiceUnavailableError` (propagates to worker)

**`_parse_prereqs(raw_text) -> list[dict]`** -- pure function:
- Finds `[...]` block, parses JSON, validates each item has `requires`, `required_by`, `confidence`
- Returns list of valid dicts

**`_resolve_entity_id(name, document_id, graph_svc) -> str | None`**:
- Calls `graph_svc.match_entity_by_name(name, document_id)`
- Returns entity_id or None if not found

**`enrich(document_id) -> int`**:
- Load all `SectionSummaryModel` for the document (skip if 0)
- For each summary:
  - Call `extract(summary.content, summary.section_id, document_id)`
  - For each `(requires, required_by, confidence)` pair:
    - Resolve both concept names to Entity IDs via `match_entity_by_name`
    - If both resolve: call `graph_svc.add_prerequisite_with_section(dependent_id,
      prerequisite_id, document_id, confidence, summary.section_id or "")`
- After all edges written, compute chain depths and update `SectionModel.difficulty_estimate`
- Return count of edges written

**`_compute_and_store_difficulty(document_id)`**:
- Calls `graph_svc.get_learning_path`... Actually this is circular. Use a direct BFS on the
  adjacency list returned by `get_prerequisite_edges_for_document`:
  - Build a DAG: for each entity, max hop count from any root (Kahn-style BFS from roots).
  - For each SectionSummaryModel, look up all entities mentioned in that section via Kuzu
    `(e:Entity)-[:MENTIONED_IN]->(d:Document)` -- but we need section-level granularity.
  - SIMPLER: compute chain depth per entity (global across all sections of the document).
    Then for each SectionModel: difficulty_estimate = max(chain_depth of any entity whose
    PREREQUISITE_OF source_section_id = this section's id). If no prereq edges sourced here,
    depth = 0.
  - Use raw SQLAlchemy `update()` to set `SectionModel.difficulty_estimate`.

**`async def prereq_extract_handler(document_id: str, job_id: str) -> None`**:
- Top-level handler function for registration with the enrichment worker.

```python
async def get_study_path(
    self, document_id: str, concept: str, db_session
) -> StudyPathResponse:
    """Return FSRS-aware ordered study path for a concept in a document.

    Algorithm:
    1. Call KuzuService.get_learning_path(concept, document_id) to get topo-ordered nodes.
    2. For each node in topo order (deepest prereq first):
       a. Find all chunks in the document whose section mentions this concept
          (via Entity.name match, then SectionModel -> ChunkModel -> FlashcardModel).
       b. Compute avg fsrs_stability across all flashcards for those chunks.
       c. skip = avg_stability >= 14 days
       d. mastery = avg(stability / 21.0) capped at 1.0
    3. Return StudyPathResponse with path ordered from earliest unskipped concept.
    """
```

```python
async def get_start_concepts(
    self, document_id: str, db_session
) -> StartConceptsResponse:
    """Return up to 3 entry-point concepts with highest learning ROI.

    Algorithm:
    1. Call KuzuService.get_entry_point_concepts(document_id) to get root concepts.
    2. For each root concept, count its flashcards and its downstream chain length.
    3. Sort by (unskipped_prereq_count asc, flashcard_count asc) -- fewest barriers first.
    4. Return top 3.
    """
```

**Implementation notes for `get_study_path`**:

The concept-to-flashcard linkage is approximate because PREREQUISITE_OF connects entities, not
sections. The linkage path is:
`entity.name matches chunk.text (via FTS or entity name search) -> chunk.section_id ->
SectionModel -> ChunkModel -> FlashcardModel (via chunk_id)`.

Simpler and more reliable: for each concept name in the path, find flashcards where
`document_id = document_id` AND chunk's text contains the concept name (case-insensitive).
Use SQLAlchemy with `ChunkModel.text.ilike(f"%{concept}%")` joined to `FlashcardModel`.

This is an approximation but avoids building a full entity->section->flashcard mapping.
The story does not require exact linkage -- it requires avg_stability_days and mastery.

## Risk Flags

**Risk 1: Kuzu schema migration for source_section_id on PREREQUISITE_OF (HIGH)**

The existing PREREQUISITE_OF table was created in S117 without `source_section_id`. Kuzu
does not support `ALTER TABLE` on edge tables. The `CREATE REL TABLE IF NOT EXISTS` guard
means the new schema is never applied on existing databases.

**Mitigation**: The `add_prerequisite_with_section` method uses a try/except fallback:
first tries the 3-column INSERT, catches the "unknown property" exception, falls back to the
2-column form. This means `source_section_id` will be absent on old databases but the code
does not crash. Tests use `tmp_path` Kuzu instances (always fresh schema) so the new column
is always present in tests.

Note: The `difficulty_estimate` computation in `_compute_and_store_difficulty` uses
`source_section_id` from the edge to map depth back to sections. On old DBs this will
return empty string; section difficulty will remain NULL (acceptable degradation).

**Risk 2: Concept name resolution in study path is approximate (MEDIUM)**

`match_entity_by_name` uses substring matching, not exact canonical name matching. The
path from `get_learning_path` returns entity names; resolving these back to flashcards
uses `ChunkModel.text.ilike(f"%{concept}%")`. This may match unrelated flashcards.

**Mitigation**: The story explicitly says "mastery per concept = average (fsrs_stability /
21.0) for all flashcards whose section maps to a section mentioning that concept." This is
approximate by design. Document the approximation in the service docstring.

**Risk 3: Entry-point concept definition (MEDIUM)**

The AC says "concepts with the shortest prerequisite chains where the user has fewest
flashcards." The implementation defines entry points as root concepts (no outgoing prereq
edges) that ARE referenced as prerequisites by others. This may return zero results if all
prerequisite edges are between concepts not in the entity graph, or if the LLM extracts
prerequisites for leaf concepts only.

**Mitigation**: `GET /study/start` returns an empty `concepts` list gracefully; the
Learning tab panel is hidden (not rendered) when concepts list is empty.

**Risk 4: PREREQUISITE_OF edges not included in existing graph response (HIGH)**

`KuzuService.get_graph_for_document` currently does NOT include PREREQUISITE_OF edges.
Step 4d adds `get_prerequisite_edges_for_graph` and Step 9b calls it. The Viz tab GraphEdge
interface needs the `relation` field added -- otherwise TypeScript will flag an error.

**Mitigation**: Update GraphEdge interface in Viz.tsx and ensure the API returns the
`relation` field in edge dicts. Verify with `tsc --noEmit`.

**Risk 5: Viz graph type must support directed edges for PREREQUISITE_OF (MEDIUM)**

The current `buildGraph` uses `new Graph()` (undirected). Sigma renders all edges as lines
regardless. For actual arrowheads on PREREQUISITE_OF edges, Sigma requires a directed or
mixed graph AND an `arrowhead` edge program. The existing learning-path view uses
`new Graph({ type: "directed" })` with arrowheads working. The knowledge graph view
switches to mixed graph type.

**Mitigation**: Change `buildGraph` to use `new Graph({ type: "mixed" })`. This is backward
compatible -- undirected edges still render as bidirectional lines; PREREQUISITE_OF edges
added with `g.addDirectedEdge()` will show arrowheads if Sigma is configured with the
arrow edge type. Add a flag to `buildGraph` to identify PREREQUISITE_OF edges and use
`g.addDirectedEdge()` for them. If arrowhead rendering is complex, fall back to just the
purple color distinction (still satisfies AC8 functionally).

**Risk 6: Learning tab content_type detection (LOW)**

The Learning tab does not currently have access to the selected document's `content_type` in
the part of the component tree where the document reader is rendered. The `DocumentListItem`
type in the frontend does include `content_type`, so it is available from the fetched
documents list.

**Mitigation**: Pass `content_type` from the selected document's DocumentListItem down to
the `WhereToStartPanel`. Look up the selected document in the documents list by
`activeDocumentId` to get its content_type.

**Risk 7: Cross-story PREREQUISITE_OF edge format (LOW)**

S117 created PREREQUISITE_OF edges via the regex-based `prerequisite_detector.py` during
ingestion. S139 creates additional edges via LLM-based `PrereqExtractorService` as a
post-ingestion enrichment job. Both pipelines write to the same Kuzu edge table. The
existing `add_prerequisite` (S117) and new `add_prerequisite_with_section` (S139) are both
idempotent -- they check for existing edges before creating. No deduplication conflict.
