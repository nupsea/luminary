# S106 — Tech Debt TD-001: section_id on notes model + DocumentReader dot indicators

## Goal

Add a nullable `section_id` column to the `notes` table and wire it through the
API and the DocumentReader section list so that note-presence indicators survive
page reloads.

## Layers Affected

- **Types** — `NoteCreateRequest`, `NoteUpdateRequest`, `NoteResponse` in `app/routers/notes.py`
  (the notes router owns its own Pydantic schemas; no separate types file is involved)
- **Repo/ORM** — `NoteModel` in `backend/app/models.py`
- **Config/migration** — `backend/app/db_init.py` (additive ALTER TABLE)
- **API** — `backend/app/routers/notes.py` (create, patch/put, response helper, list endpoint)
- **Frontend** — `frontend/src/components/reader/DocumentReader.tsx` (fetch notes for doc on mount, dot indicator logic)
- **Tests** — `backend/tests/test_notes.py` (new section_id test cases)
- **Smoke** — `scripts/smoke/S106.sh`

## Files to Create or Modify

- `backend/app/models.py` — add `section_id: Mapped[str | None]` column to `NoteModel`
- `backend/app/db_init.py` — add additive migration `ALTER TABLE notes ADD COLUMN section_id TEXT` to the migration loop
- `backend/app/routers/notes.py` — add `section_id` to `NoteCreateRequest`, `NoteUpdateRequest`, `NoteResponse`, `_to_response()`, `create_note()`, `_apply_note_update()`
- `frontend/src/components/reader/DocumentReader.tsx` — add `useQuery` to fetch `GET /notes?document_id=X`; derive a `Set<string>` of section IDs that have notes; replace `notedSections` local state with the API-sourced set; pass `section_id` when creating a note; invalidate the query after save
- `backend/tests/test_notes.py` — add tests: create note with `section_id`, verify it round-trips; list notes returns `section_id`; PATCH preserves existing `section_id` when not sent
- `scripts/smoke/S106.sh` — curl-based smoke test: POST note with section_id, GET /notes?document_id=X, assert section_id present in response

## Implementation Steps

### 1. [Repo/ORM] Add `section_id` to `NoteModel`

In `backend/app/models.py`, add the following field to `NoteModel` after `chunk_id`:

```python
section_id: Mapped[str | None] = mapped_column(String, nullable=True)
```

No index needed — lookup is always scoped to `document_id` in the list query.
No FK constraint — sections may be deleted independently; the column is advisory.

### 2. [Config/migration] Add ALTER TABLE migration in `db_init.py`

In `backend/app/db_init.py`, add to the `for ddl in [...]` migration loop:

```python
"ALTER TABLE notes ADD COLUMN section_id TEXT",
```

This follows the exact same additive-migration pattern used for `file_hash`,
`chapter_count`, `conversation_metadata`, `source`, `deck`, and `difficulty`.
SQLite will raise an error if the column already exists; the surrounding
`try/except Exception: pass` absorbs it, so the migration is idempotent.

### 3. [API — Types] Extend Pydantic schemas in `notes.py`

`NoteCreateRequest` — add optional field:
```python
section_id: str | None = None
```

`NoteUpdateRequest` — add optional field:
```python
section_id: str | None = None
```

`NoteResponse` — add field:
```python
section_id: str | None
```

### 4. [API — Logic] Wire `section_id` through create and update handlers

In `create_note()`, add `section_id=req.section_id` to the `NoteModel(...)` constructor call.

In `_to_response()`, add `section_id=note.section_id` to the `NoteResponse(...)` constructor call.

In `_apply_note_update()`, add a conditional update block:
```python
if req.section_id is not None:
    note.section_id = req.section_id
```

Note: PATCH semantics mean `section_id=None` in the request body means "field
not sent" (do not update). This is consistent with how `content`, `tags`, and
`group_name` are handled already. If a caller explicitly wants to clear
`section_id`, they would need a sentinel — this edge case is out of scope for
TD-001 which only needs the forward path (set section_id).

### 5. [Frontend] Fetch notes for document and derive noted-section set

**Current state**: `notedSections` is a local `useState<Set<string>>(new Set())` that
gets populated only when a note is saved in the current session — it never
persists across reloads because it reads from component state, not the API.

**Change**:

a. Add a `useQuery` call to fetch notes for the open document:

```typescript
const { data: docNotes } = useQuery({
  queryKey: ["notes-for-doc", documentId],
  queryFn: async () => {
    const res = await fetch(`${API_BASE}/notes?document_id=${documentId}`)
    if (!res.ok) throw new Error("Failed to fetch notes")
    return res.json() as Promise<NoteResponse[]>
  },
  staleTime: 30_000,
})
```

Define a minimal local `NoteResponse` interface (id, section_id) at the top of
the component file — only the fields needed for the indicator logic.

b. Derive the set of noted section IDs from the API response:

```typescript
const notedSections = useMemo(
  () => new Set((docNotes ?? []).map((n) => n.section_id).filter(Boolean) as string[]),
  [docNotes]
)
```

Remove the existing `const [notedSections, setNotedSections] = useState<Set<string>>(new Set())`
declaration.

c. Pass `section_id` when creating a note. Update the `createNote` call in
`NoteEditor` to accept and forward `section_id`:

- Change `NoteEditorProps` to add `sectionId: string`
- Update `createNote(...)` call to include `section_id: sectionId`
- Update `createNote` function signature to accept `section_id: string | undefined`
- Update the fetch body to include `section_id`

d. In `onSaved` callback for `NoteEditor`, remove the `setNotedSections` call
(the state no longer exists). Instead call:
```typescript
void qc.invalidateQueries({ queryKey: ["notes-for-doc", documentId] })
```
This is already present in the current code; simply remove the `setNotedSections` line alongside it.

e. Pass `sectionId={section.id}` when rendering `<NoteEditor>` inside the section map.

### 7. [Tests] Add section_id test cases to `test_notes.py`

Add three new test functions (append to existing `test_notes.py`):

```python
def test_create_note_with_section_id(client):
    """POST /notes with section_id stores and returns it."""
    resp = client.post(
        "/notes",
        json={"content": "Section note", "document_id": "doc-1", "section_id": "sec-42"},
    )
    assert resp.status_code == 201
    data = resp.json()
    assert data["section_id"] == "sec-42"


def test_list_notes_returns_section_id(client):
    """GET /notes?document_id=X returns section_id in each note object."""
    client.post(
        "/notes",
        json={"content": "Note for section", "document_id": "doc-s", "section_id": "sec-7"},
    )
    resp = client.get("/notes?document_id=doc-s")
    assert resp.status_code == 200
    data = resp.json()
    assert any(n["section_id"] == "sec-7" for n in data)


def test_patch_note_updates_section_id(client):
    """PATCH /notes/{id} with section_id updates it; existing rows without it return null."""
    create_resp = client.post("/notes", json={"content": "No section yet"})
    note_id = create_resp.json()["id"]
    assert create_resp.json()["section_id"] is None  # nullable default

    patch_resp = client.patch(f"/notes/{note_id}", json={"section_id": "sec-99"})
    assert patch_resp.status_code == 200
    assert patch_resp.json()["section_id"] == "sec-99"
```

## Risk Flags

- Risk: `NoteUpdateRequest.section_id = None` is ambiguous — it means "field not sent"
  and also "set section_id to null". Given PATCH semantics here (`if req.section_id is not None`),
  callers cannot clear section_id via PATCH. This is acceptable for TD-001 scope; the indicator
  only needs the forward path (set, not clear). Document in code comment.
  (mitigation: add inline comment on the conditional; no additional behavior change needed)

- Risk: The `notedSections` `useMemo` runs after the query resolves. During the initial
  load the set is empty and no indicators show — this is correct loading-state behavior,
  not a visual bug. The `useQuery` load is fast (notes are small rows).
  (mitigation: no skeleton/spinner needed for the indicator specifically; the document's own
  `isLoading` guard already covers the full-page skeleton)

- Risk: `docNotes` query runs on every DocumentReader mount (per-document). With `staleTime: 30_000`
  it will not re-fetch on every window focus. This is consistent with the global `staleTime: 60_000`
  and the pattern used for Notes.tsx (`staleTime: 10_000`).
  (mitigation: 30s is conservative; acceptable for this indicator use case)

- Risk: Existing `test_notes.py` tests do not assert `section_id` in responses — they will
  pass only if `section_id` is present in `NoteResponse` (step 3). If omitted, tests that
  check full response shape would fail. New tests explicitly verify the field.
  (mitigation: add `section_id` to `NoteResponse` before running tests)

- Risk: The `createNote` function in `DocumentReader.tsx` currently does not accept
  `section_id`. Adding it requires threading the value from the section map through
  `NoteEditorProps` to the `createNote` fetch call. This is a two-step prop change
  (interface + usage site) — must be done together to avoid TypeScript errors.
  (mitigation: implement in a single diff; tsc will catch any mismatch)
