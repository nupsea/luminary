## Implementation Steps

### Step 1 — [Types] Add BloomGap, BloomSectionStat, CoverageReport TypedDicts to app/types.py

Add after the existing TypedDict definitions (after MasteryHeatmapResponse at line 252). All three are pure data containers — no I/O.

```python
# ---------------------------------------------------------------------------
# Flashcard coverage audit (S153)
# ---------------------------------------------------------------------------

class BloomGap(TypedDict):
    section_id: str
    section_heading: str
    missing_bloom_levels: list[int]  # levels 1-6 absent from section's cards


class BloomSectionStat(TypedDict):
    section_heading: str
    by_bloom_level: dict[int, int]   # level -> count (only levels present)
    has_level_3_plus: bool


class CoverageReport(TypedDict):
    total_cards: int
    by_bloom_level: dict[int, int]   # global level -> count across all levels 1-6
    by_section: dict[str, BloomSectionStat]  # keyed by section_id
    coverage_score: float             # fraction of sections with >= 1 card at bloom_level >= 3
    gaps: list[BloomGap]
```

### Step 2 — [Service] Create app/services/flashcard_audit.py

This service depends on FlashcardModel (db query), SectionModel (db query), and FlashcardService (generate call). It must NOT import from the API layer.

Key design decisions:

- `analyze_coverage` is a pure DB-read with no LLM call. It queries FlashcardModel filtered by document_id, joins through ChunkModel to get section_id, then queries SectionModel for headings. Compute stats in Python. Do NOT do N+1 queries: use a single JOIN query (FlashcardModel -> ChunkModel -> SectionModel) to get all (card, section_id, section_heading) tuples in one round-trip.

- `fill_gaps` generates cards for each gap. For each missing bloom_level in each gap section, it calls FlashcardService with a level-specific system prompt. Use `asyncio.Semaphore(3)` to bound concurrency. Each level call generates 1-2 cards scoped to the gap section.

- Level-specific prompt additions (extend the existing TECH_FLASHCARD_SYSTEM):
  - Level 3 (Apply): "Write an APPLICATION question asking how to USE this concept in a realistic scenario."
  - Level 4 (Analyze): "Write an ANALYSIS question asking the reader to COMPARE, CONTRAST, or EVALUATE..."
  - Level 5 (Evaluate): "Write an EVALUATION question asking the reader to justify or critique a design decision."
  - Level 6 (Create): "Write a SYNTHESIS question asking the reader to design, construct, or propose."

- Return from `fill_gaps`: total integer count of created cards (sum over all section/level pairs).

- `get_flashcard_audit_service()` follows the existing module-level singleton pattern (`_service: FlashcardAuditService | None`).

Query strategy for analyze_coverage:

```python
# One JOIN query: flashcard -> chunk -> section
stmt = (
    select(FlashcardModel.bloom_level, ChunkModel.section_id, SectionModel.heading)
    .join(ChunkModel, FlashcardModel.chunk_id == ChunkModel.id)
    .join(SectionModel, ChunkModel.section_id == SectionModel.id)
    .where(FlashcardModel.document_id == document_id)
    .where(FlashcardModel.bloom_level.is_not(None))
)
```

Cards with null bloom_level are counted in total_cards but not in by_bloom_level or gap computation. Cards with null chunk_id (source='note', 'gap', 'feynman') are excluded from section analysis via the join (they will drop out naturally since chunk_id IS NULL fails the join).

coverage_score formula:
- sections_with_l3_plus = number of distinct section_ids with at least one card where bloom_level >= 3
- total_sections = number of distinct section_ids that have at least one card with a non-null bloom_level
- coverage_score = sections_with_l3_plus / total_sections if total_sections > 0 else 0.0

gaps computation:
- For each section_id in the analysis result, compute which levels 1-6 are absent.
- Include a section in gaps only if it has zero cards at bloom_level >= 3 (this is the "needs attention" signal).

### Step 3 — [API] Add audit endpoints to app/routers/flashcards.py

Add two new Pydantic response schemas:

```python
class CoverageReportResponse(BaseModel):
    total_cards: int
    by_bloom_level: dict[int, int]
    by_section: dict[str, dict]   # BloomSectionStat as dict -- avoid TypedDict in Pydantic
    coverage_score: float
    gaps: list[dict]              # BloomGap as dict

class FillGapsResponse(BaseModel):
    created: int
```

Add two endpoints after the existing `generate-technical` endpoint (before `create-trace`) to maintain logical grouping:

```
GET  /flashcards/audit/{document_id}       -> CoverageReportResponse
POST /flashcards/audit/{document_id}/fill  -> FillGapsResponse
```

The fill endpoint must:
1. Call `audit_service.analyze_coverage(document_id, db)` to get current gaps (fresh analysis).
2. Call `audit_service.fill_gaps(document_id, report["gaps"], db)`.
3. Wrap LiteLLM ServiceUnavailableError -> HTTP 503.
4. Return FillGapsResponse(created=count).

Route conflict caution: FastAPI matches routes in registration order. The new `/audit/{document_id}` path uses `audit` as the first path segment; this cannot conflict with existing `/{document_id}` (already at the bottom) because `/audit/...` will be registered BEFORE `/{document_id}`. Verify by checking the order: the existing `/{document_id}` GET is at line 401. Place `/audit/{document_id}` before it.

Use `Depends(get_flashcard_audit_service)` for the new endpoints to follow the existing service dependency pattern.

- **Empty state** (total_cards === 0): "No flashcards yet -- generate some to see Bloom distribution"
- **Chart**: Recharts BarChart lazy-loaded (same pattern as ProgressDashboard -- wrap in `lazy(() => import(...))` and `<Suspense>`). ## Risk Flags

- **Risk: Route ordering conflict in flashcards.py** -- the new `GET /flashcards/audit/{document_id}` must be registered before the existing `GET /flashcards/{document_id}` route or FastAPI will match `audit` as a document_id. Mitigation: place both audit routes immediately before the `/{document_id}` route (currently at line 401). Add a comment noting why order matters.

- **Risk: fill_gaps calls FlashcardService.generate_technical per level** -- generate_technical requires real chunk data and calls LiteLLM. For the fill_gaps implementation, do NOT call generate_technical (that method fetches chunks and builds its own text). Instead, write a focused `_generate_bloom_card` helper in flashcard_audit.py that calls `get_llm_service().generate(prompt, system=AUDIT_FILL_SYSTEM, stream=False)` with a per-level prompt containing the section heading, then parses the result via `_parse_llm_response` (import from flashcard.py). This avoids the DB join overhead in generate_technical and lets the audit service control exactly which bloom_level is stored. The generated FlashcardModel rows need a valid chunk_id; look up the first chunk for that section before generating.

- **Risk: FlashcardModel rows with null chunk_id are excluded from section analysis** -- cards from notes/gap/feynman sources have chunk_id=None. The JOIN query (flashcard -> chunk -> section) will silently exclude these. This is correct behavior (they have no section context) but must be documented in a comment in analyze_coverage so future developers know the exclusion is intentional.

- **Risk: Recharts BarChart import in Study.tsx may increase initial chunk size** -- Recharts is already in the vendor bundle via ProgressDashboard. Since DeckHealthPanel is not lazy-loaded itself, the Recharts import will be included in the Study chunk. Mitigation: guard with `{isOpen && <BarChartSection />}` to avoid rendering. The bundle size impact is acceptable because Recharts is already present in the app. See TD-003 for bundle size tracking.

- **Risk: coverage_score division by zero** -- if a document has flashcards but none have a non-null bloom_level (e.g. all generated with the non-technical path), total_sections will be 0. Mitigation: guard as `coverage_score = 0.0 if total_sections == 0 else ...`. Document this in a comment.

- **Risk: TypeScript strict mode** -- `by_bloom_level` is `dict[int, int]` in Python but TypeScript receives it as `Record<string, number>` (JSON object keys are always strings). Use `Number(key)` when iterating chart data. Define the TS interface with `Record<string, number>` not `Record<number, number>` to match JSON wire format.
