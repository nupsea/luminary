# S117 - Concept Prerequisite Mapping: PREREQUISITE_OF Edges in Kuzu with Learning-Path Endpoint

## Goal

Detect prerequisite linguistic markers in chunk text during ingestion, store directed PREREQUISITE_OF edges in Kuzu between dependent and prerequisite concepts, expose a topological learning-path endpoint, and render those edges as directed orange arrows in the Viz tab.

## Layers Affected

- **Types** (`backend/app/types.py`) - new `LearningPathNode` response dataclass
- **Repo** (`backend/app/services/graph.py`) - new Kuzu schema table, new upsert and query methods (note: this codebase uses `services/graph.py` as the Repo layer for Kuzu; there is no `repos/graph.py`)
- **Service** (`backend/app/services/prerequisite_detector.py`) - new pure-function regex marker scanner
- **Runtime** (`backend/app/workflows/ingestion.py`) - call prerequisite detector inside `entity_extract_node`
- **API** (`backend/app/routers/graph.py`) - new `GET /graph/learning-path` endpoint

Frontend:
- `frontend/src/pages/Viz.tsx` - "Learning Path" toggle, directed orange edges, prerequisite breadcrumb in node popover

## Implementation Steps

### Step 1 - [Types] Add response types for the learning-path endpoint

In `backend/app/types.py`, add:

```python
@dataclass
class LearningPathNode:
    entity_id: str
    name: str
    entity_type: str
    depth: int  # 0 = start, increasing = further prerequisites

class LearningPathResponse(TypedDict):
    start_entity: str
    document_id: str
    nodes: list[LearningPathNode]  # topologically sorted, start node first
    edges: list[dict]  # list of {from_entity, to_entity, confidence}
```

### Step 2 - [Repo] Extend Kuzu schema with PREREQUISITE_OF table

In `backend/app/services/graph.py`, in `_create_schema()`, add:

```python
"CREATE REL TABLE IF NOT EXISTS PREREQUISITE_OF("
"FROM Entity TO Entity, document_id STRING, confidence FLOAT)",
```

This creates a directed edge from the DEPENDENT concept TO its PREREQUISITE (e.g. "Natural Selection" -PREREQUISITE_OF-> "Variation"). The direction is: the entity that "requires" points to the entity it requires.

### Step 3 - [Repo] Add `add_prerequisite` method to KuzuService

New method in `KuzuService`:

```python
def add_prerequisite(
    self,
    dependent_id: str,
    prerequisite_id: str,
    document_id: str,
    confidence: float = 1.0,
) -> None:
    """Create a PREREQUISITE_OF edge from dependent to prerequisite.

    Idempotent: if the edge already exists with the same document_id, it is
    left unchanged. Multiple edges for the same pair but different document_ids
    are allowed.
    """
```

Use the same check-then-create pattern as `add_relation()` (check for existing edge before CREATE).

### Step 4 - [Repo] Add `get_prerequisite_edges_for_document` method to KuzuService

```python
def get_prerequisite_edges_for_document(
    self, document_id: str
) -> list[dict]:
    """Return all PREREQUISITE_OF edges for a document.

    Returns list of dicts:
        {from_entity: str (name), to_entity: str (name),
         from_id: str, to_id: str, confidence: float}
    """
```

Query pattern: MATCH on MENTIONED_IN edges to scope to the document, then fetch PREREQUISITE_OF edges.

### Step 5 - [Repo] Add `get_learning_path` method to KuzuService

```python
def get_learning_path(
    self, start_entity_name: str, document_id: str
) -> dict:
    """Return topologically sorted prerequisite chain starting from start_entity_name.

    Algorithm:
    1. Find the Entity node matching start_entity_name (case-insensitive)
       that is MENTIONED_IN the given document_id.
    2. BFS/DFS traversal following PREREQUISITE_OF edges outward from start.
    3. Topological sort on the traversed subgraph (Kahn's algorithm, pure Python).
    4. Return {start_entity, document_id, nodes, edges} matching LearningPathResponse.
    Returns empty nodes/edges if start_entity not found.
    """
```

Important: The topological sort must be a pure Python function (no I/O), consistent with core-belief #19.

### Step 6 - [Service] Create `prerequisite_detector.py`

New file `backend/app/services/prerequisite_detector.py`.

Define marker phrases (these are the linguistic markers the story specifies):

```python
_PREREQUISITE_MARKERS: list[tuple[re.Pattern, str]] = [
    # Each tuple: (compiled pattern, confidence)
    # Pattern captures (dependent, prerequisite) via named groups
    (re.compile(
        r"(?P<dep>[A-Z][a-zA-Z\s]{2,40}?)\s+(?:requires?\s+understanding\s+of|requires?)\s+(?P<prereq>[A-Z][a-zA-Z\s]{2,40})",
        re.IGNORECASE,
    ), "high"),
    (re.compile(
        r"(?P<dep>[A-Z][a-zA-Z\s]{2,40}?)\s+builds?\s+on\s+(?P<prereq>[A-Z][a-zA-Z\s]{2,40})",
        re.IGNORECASE,
    ), "medium"),
    (re.compile(
        r"(?P<dep>[A-Z][a-zA-Z\s]{2,40}?)\s+(?:is\s+)?defined\s+as\s+a\s+type\s+of\s+(?P<prereq>[A-Z][a-zA-Z\s]{2,40})",
        re.IGNORECASE,
    ), "medium"),
    (re.compile(
        r"(?P<dep>[A-Z][a-zA-Z\s]{2,40}?)\s+is\s+a\s+subclass\s+of\s+(?P<prereq>[A-Z][a-zA-Z\s]{2,40})",
        re.IGNORECASE,
    ), "high"),
    (re.compile(
        r"(?P<dep>[A-Z][a-zA-Z\s]{2,40}?)\s+depends?\s+on\s+(?P<prereq>[A-Z][a-zA-Z\s]{2,40})",
        re.IGNORECASE,
    ), "medium"),
    (re.compile(
        r"(?P<dep>[A-Z][a-zA-Z\s]{2,40}?)\s+(?:was\s+)?first\s+introduced\s+as\s+(?P<prereq>[A-Z][a-zA-Z\s]{2,40})",
        re.IGNORECASE,
    ), "low"),
]

_CONFIDENCE_SCORES = {"high": 0.9, "medium": 0.7, "low": 0.5}
```

Public function (pure, no I/O):

```python
def detect_prerequisites(
    chunks: list[dict],
    entity_names: set[str],
) -> list[tuple[str, str, float]]:
    """Scan chunk texts for prerequisite linguistic markers.

    Args:
        chunks: Chunk dicts with at least a 'text' field.
        entity_names: Lowercase canonical entity names already known for this
                      document. Only return pairs where BOTH names appear in
                      this set (exact or substring match), to avoid phantom
                      entity pairs from regex overmatching.

    Returns:
        List of (dependent_name, prerequisite_name, confidence) triples.
        All names are lowercased. Duplicates are deduplicated; if the same
        pair appears multiple times, the highest confidence is kept.
    """
```

The entity_names guard prevents creating PREREQUISITE_OF edges for names that GLiNER did not extract as genuine entities from this document, which is the primary risk of regex over-matching.

### Step 7 - [Runtime] Wire prerequisite detection into `entity_extract_node`

In `backend/app/workflows/ingestion.py`, inside `entity_extract_node`, after the co-occurrence block and before the call-graph block, add:

```python
# Prerequisite detection: scan chunk texts for marker phrases.
# Only creates edges between entities already confirmed by GLiNER.
from app.services.prerequisite_detector import detect_prerequisites  # noqa: PLC0415

canonical_name_to_id: dict[str, str] = {
    ent["name"]: ent["id"] for ent in canonical_entities
}
known_names: set[str] = set(canonical_name_to_id.keys())

prereq_pairs = detect_prerequisites(ner_chunks, known_names)
for dep_name, prereq_name, confidence in prereq_pairs:
    dep_id = canonical_name_to_id.get(dep_name)
    prereq_id = canonical_name_to_id.get(prereq_name)
    if dep_id and prereq_id and dep_id != prereq_id:
        graph.add_prerequisite(dep_id, prereq_id, doc_id, confidence)

logger.info(
    "Prerequisite edges created: %d",
    len(prereq_pairs),
    extra={"doc_id": doc_id},
)
```

Place this inside the outer `try` block (non-fatal - a failure here must not abort ingestion). Log at INFO level.

### Step 8 - [API] Add `GET /graph/learning-path` endpoint

In `backend/app/routers/graph.py`, add a new endpoint before the `GET /{document_id}` route (to avoid path conflicts with `{document_id}` matching "learning-path"):

```python
from app.types import LearningPathResponse  # at top of file

@router.get("/learning-path")
async def get_learning_path(
    start_entity: str = Query(..., description="Entity name to start path from"),
    document_id: str = Query(..., description="Document scope"),
) -> dict:
    """Return topologically sorted prerequisite chain for a concept.

    Returns LearningPathResponse with nodes sorted from start (depth=0)
    toward deepest prerequisite. Returns empty nodes/edges if the entity
    is not found or has no prerequisite edges.
    """
    svc = get_graph_service()
    return svc.get_learning_path(start_entity, document_id)
```

Important: This route MUST be declared before `GET /{document_id}` in the router, because FastAPI/Starlette match routes in order and `/learning-path` would otherwise be captured as `document_id="learning-path"`.

```python
def test_add_prerequisite_creates_edge(graph_svc):
    graph_svc.upsert_entity("e1", "natural selection", "CONCEPT")
    graph_svc.upsert_entity("e2", "variation", "CONCEPT")
    graph_svc.add_prerequisite("e1", "e2", "doc1", 0.9)
    result = graph_svc._conn.execute(
        "MATCH (a:Entity {id: 'e1'})-[r:PREREQUISITE_OF]->(b:Entity {id: 'e2'})"
        " WHERE r.document_id = 'doc1' RETURN r.confidence"
    )
    assert result.has_next()
    assert result.get_next()[0] == pytest.approx(0.9)

def test_add_prerequisite_idempotent(graph_svc):
    # Calling add_prerequisite twice does not create duplicate edges
    graph_svc.upsert_entity("e1", "a", "CONCEPT")
    graph_svc.upsert_entity("e2", "b", "CONCEPT")
    graph_svc.add_prerequisite("e1", "e2", "d1")
    graph_svc.add_prerequisite("e1", "e2", "d1")
    result = graph_svc._conn.execute(
        "MATCH (a:Entity {id: 'e1'})-[r:PREREQUISITE_OF]->(b:Entity {id: 'e2'})"
        " WHERE r.document_id = 'd1' RETURN count(r)"
    )
    assert result.get_next()[0] == 1

def test_get_prerequisite_edges_for_document(graph_svc):
    graph_svc.upsert_entity("e1", "natural selection", "CONCEPT")
    graph_svc.upsert_entity("e2", "variation", "CONCEPT")
    graph_svc.upsert_document("d1", "Biology", "book")
    graph_svc.add_mention("e1", "d1")
    graph_svc.add_mention("e2", "d1")
    graph_svc.add_prerequisite("e1", "e2", "d1", 0.9)
    edges = graph_svc.get_prerequisite_edges_for_document("d1")
    assert len(edges) == 1
    assert edges[0]["from_entity"] == "natural selection"
    assert edges[0]["to_entity"] == "variation"

def test_get_learning_path_topological_order(graph_svc):
    # Chain: C -> B -> A (C requires B which requires A)
    graph_svc.upsert_entity("ea", "A", "CONCEPT")
    graph_svc.upsert_entity("eb", "B", "CONCEPT")
    graph_svc.upsert_entity("ec", "C", "CONCEPT")
    graph_svc.upsert_document("d1", "Doc", "book")
    for eid in ("ea", "eb", "ec"):
        graph_svc.add_mention(eid, "d1")
    graph_svc.add_prerequisite("ec", "eb", "d1")  # C requires B
    graph_svc.add_prerequisite("eb", "ea", "d1")  # B requires A

    result = graph_svc.get_learning_path("C", "d1")
    names = [n["name"] for n in result["nodes"]]
    assert "C" in names
    # A must come before B and B before C in topological order
    assert names.index("A") < names.index("B")
    assert names.index("B") < names.index("C")

def test_get_learning_path_unknown_entity_returns_empty(graph_svc):
    graph_svc.upsert_document("d1", "Doc", "book")
    result = graph_svc.get_learning_path("nonexistent", "d1")
    assert result["nodes"] == []
    assert result["edges"] == []
```

## Risk Flags

- **Risk: Regex overmatching in prose text** - Natural language patterns like "requires" appear in many non-prerequisite contexts ("the user requires authentication"). Mitigation: the `entity_names` guard in `detect_prerequisites` ensures only pairs where both names are known GLiNER-confirmed entities produce edges. This dramatically reduces false positives.

- **Risk: Kuzu schema migration for existing databases** - Databases created before S117 will not have the `PREREQUISITE_OF` table. Mitigation: use `CREATE REL TABLE IF NOT EXISTS` (already the pattern for all tables in `_create_schema`). This is safe for existing databases.

- **Risk: Route order conflict in `/graph/learning-path` vs `/{document_id}`** - FastAPI will match `/learning-path` as a `document_id` if `GET /learning-path` is declared after `GET /{document_id}`. Mitigation: declare `/learning-path` first in the router file (Step 8 explicitly calls this out). Add a test that verifies the correct route is hit.

- **Risk: Cyclic prerequisite graphs causing infinite loops in traversal** - If ingested text creates A -> B -> A cycles (e.g., "A requires B" AND "B requires A" in different passages), `get_learning_path` DFS would loop forever. Mitigation: the BFS/DFS in `get_learning_path` must track visited node IDs with a `seen: set[str]` guard. Topological sort (Kahn's algorithm) also naturally handles cycles by detecting nodes with no remaining in-edges, stopping cleanly.

- **Risk: Topological sort in Python adding latency** - For large documents with many PREREQUISITE_OF edges, BFS + topo-sort over the subgraph happens in Python. Mitigation: the prerequisite subgraph is scoped to a single document, and GLiNER + entity guard will yield few edges (expected O(10-100) per document). Python-level sort is negligible at this scale. No precomputation needed.

- **Risk: Frontend directed graph rendering with Sigma.js** - The existing `buildGraph()` uses `new Graph()` (undirected by default in Graphology). A directed learning-path graph requires `new Graph({ type: "directed" })`. The two graph instances are separate (learning_path vs knowledge_graph view modes have separate graph objects), so there is no conflict.

- **Risk: `entity_names` lower-case matching vs GLiNER output** - All canonical entity names in `canonical_entities` are already lowercased (per `ner.py` line: `name = ent["text"].strip().lower()`). The regex patterns use `re.IGNORECASE`, and the detector lowercases captured group text before lookup. This must be verified and tested explicitly.
