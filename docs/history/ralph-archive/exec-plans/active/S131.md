# S131 — Code-aware chunking: tech_book content type, code block atomicity, and AST signatures

## Goal

Add `tech_book` and `tech_article` ContentTypes, detect and atomically preserve fenced and indented code blocks during chunking, extract AST signatures for Python code blocks, store code metadata in new `CodeSnippetModel` table and new columns on `ChunkModel`, and expose `GET /documents/{id}/code_snippets`.

### Step 4: [Service] Create `app/services/tech_book_chunker.py`

New file with three pure functions:

**`_detect_language(fence_info: str) -> str | None`**
- Reads the fence info string (the part after the triple backtick) — e.g. `"python"`, `"js"`, `""`.
- Returns lowercase language tag or `None`.
- Normalise known aliases: `"js"` -> `"javascript"`, `"ts"` -> `"typescript"`, `"py"` -> `"python"`.

**`_parse_ast_signature(source: str, language: str | None) -> str | None`**
- Pure function; no I/O.
- For language `"python"`: attempt `ast.parse(source)`. Walk the AST for `ast.FunctionDef` and `ast.AsyncFunctionDef` at module-level (depth 0). Return the first found as `"def name(params)"` string. For `ast.ClassDef`, return `"class Name"`.
- Any exception (SyntaxError or otherwise): return `None`.
- For all other languages: return `None`.

**`extract_code_blocks(section_text: str) -> list[dict]`**
- Pure function; no I/O.
- Scans `section_text` for:
  1. **Fenced blocks**: triple-backtick pattern (``` ``` ```). Uses regex: `` r'```([^\n]*)\n(.*?)```' `` with `re.DOTALL`. Capture group 1 is the language hint; group 2 is the body.
  2. **Indented blocks**: consecutive lines with 4+ leading spaces, minimum 3 lines. Uses a line-scan loop with state tracking.
- Returns a list of `{"type": "fenced"|"indented", "language": str|None, "content": str, "start_pos": int, "end_pos": int}` dicts, ordered by position.

**`chunk_mixed_content(section_text: str, section_id: str | None, doc_id: str, chunk_size: int, chunk_overlap: int) -> list[dict]`**
- Pure orchestration; no I/O.
- Calls `extract_code_blocks(section_text)` to get code spans.
- Walks the section text and:
  - For prose segments between code blocks: splits with `RecursiveCharacterTextSplitter(chunk_size, chunk_overlap, separators=["\n\n", "\n", ". ", " ", ""])`. Each prose chunk gets `has_code=False, code_language=None, code_signature=None`.
  - For each code block: emits it as one atomic chunk (never sub-split). Sets `has_code=True`, `code_language` from fence info, `code_signature` from `_parse_ast_signature()`.
- Returns list of `{"text": str, "has_code": bool, "code_language": str|None, "code_signature": str|None, "is_code_block": bool}` dicts.
- Code blocks are never sub-split even if longer than `chunk_size`.

### Step 5: [Service/Workflow] Extend `_classify()` in `ingestion.py`

File: `backend/app/workflows/ingestion.py`

- After the `"paper"` checks (lines 83-87) and before the `"book"` check (line 88), add:

  ```python
  # tech_book: >= 3 fenced code blocks OR >= 2 numbered section headings in first 5000 chars
  first_5k = raw_text[:5000]
  code_fence_count = len(re.findall(r"```", first_5k))
  numbered_section_count = len(re.findall(r"\b\d+\.\d+\b", first_5k))
  if code_fence_count >= 6 or numbered_section_count >= 2:
      return "tech_book"

  Note: triple-backtick count is checked for >= 6 because each fenced block uses two triple-backticks (opening + closing), so 3 blocks = 6 occurrences.

- Add new async function `_chunk_tech_book()` in `ingestion.py` (after `_chunk_conversation`, before `_chunk_audio`). This function:
  1. Reads `DocumentModel.title` from DB (same pattern as `_chunk_book`).
  2. Iterates over `pd["sections"]`, creates `SectionModel` rows (same pattern as `_chunk_book`).
  3. For each section, calls `chunk_mixed_content(section_text, section_id, doc_id, cfg["chunk_size"], cfg["chunk_overlap"])` from `app.services.tech_book_chunker`.
  4. Creates `ChunkModel` rows with `has_code`, `code_language`, `code_signature` fields populated.
  5. Collects code-block chunks and creates `CodeSnippetModel` rows (id=uuid4, document_id, chunk_id, section_id, language, signature, content).
  6. Writes all models to DB in one session. Commits.
  7. Returns `{**state, "chunks": chunks, "status": "embedding"}`.

- Add endpoint after `get_conversation_metadata`:

  ```python
  @router.get("/{document_id}/code_snippets", response_model=list[CodeSnippetItem])
  async def get_code_snippets(document_id: str) -> list[CodeSnippetItem]:
      """Return extracted code snippets for a document.

      Returns [] for non-tech documents or documents with no code blocks.
      """
      async with get_session_factory()() as session:
          result = await session.execute(
              select(DocumentModel.id).where(DocumentModel.id == document_id)
          )
          if result.scalar_one_or_none() is None:
              raise HTTPException(status_code=404, detail="Document not found")

          snippets_result = await session.execute(
              select(CodeSnippetModel)
              .where(CodeSnippetModel.document_id == document_id)
              .order_by(CodeSnippetModel.created_at)
          )
          snippets = snippets_result.scalars().all()

      return [
          CodeSnippetItem(
              id=s.id,
              chunk_id=s.chunk_id,
              section_id=s.section_id,
              language=s.language,
              signature=s.signature,
              content=s.content,
          )
          for s in snippets
      ]
  ```

- Also add `CodeSnippetModel` to the delete loop in both `delete_document` and `bulk_delete_documents` endpoints so orphaned snippets are removed on document deletion.

## Risk Flags

- **Risk**: `ContentType` is a `Literal` defined inside `ingestion.py` (not in `types.py`), but it is also imported by `documents.py` (`from app.workflows.ingestion import ContentType`). Both files reference the same Literal, so adding new values in `ingestion.py` automatically propagates to the `PatchDocumentRequest` model in `documents.py`. Verify that FastAPI validates the new values correctly after the change.
  - Mitigation: Run `uv run ruff check .` and test the PATCH endpoint with `"tech_book"` value after the change.

- **Risk**: The regex `r"```([^\n]*)\n(.*?)```"` with `re.DOTALL` is greedy and can produce giant matches if a document has unclosed fences or prose backticks.
  - Mitigation: Use non-greedy `.*?` (already shown above). Add a max-content-length guard in `extract_code_blocks()`: skip any matched block whose content exceeds 20,000 characters (treat as malformed).

- **Risk**: `_chunk_tech_book()` creates both `ChunkModel` and `CodeSnippetModel` rows in the same session. The `ChunkModel` PKs (UUIDs) must be generated before `CodeSnippetModel` references them as foreign keys. Use `session.flush()` after `session.add_all(chunk_models)` before adding `CodeSnippetModel` rows, consistent with the `_chunk_book` pattern that flushes sections before chunks.
  - Mitigation: Follow the same flush-before-reference pattern used in `_chunk_book` (line 321 in `ingestion.py`).

- **Risk**: The new `has_code`, `code_language`, `code_signature` columns on `ChunkModel` are NOT NULL for `has_code` but nullable for the others. All existing code paths that create `ChunkModel` instances (conversation, audio, generic notes path) do not set these fields and will use the ORM `default=False` for `has_code`. This is correct because only tech_book/tech_article documents produce code chunks.
  - Mitigation: The `default=False` on `has_code` and `nullable=True` on the other two columns ensure backward compatibility with all existing chunk creation paths. No existing tests will break.

- **Risk**: SQLite additive migration for `has_code` uses `INTEGER NOT NULL DEFAULT 0`. SQLite allows adding `NOT NULL` columns with a `DEFAULT` in `ALTER TABLE`. This is safe.
  - Mitigation: Pattern already used in existing migrations (e.g., `"ALTER TABLE flashcards ADD COLUMN source TEXT NOT NULL DEFAULT 'document'"`).

- **Risk**: The `delete_document` and `bulk_delete_documents` endpoints do not currently delete `CodeSnippetModel` rows. Without adding `CodeSnippetModel` to both delete loops, orphaned snippet rows accumulate when documents are deleted (this is existing TD-010 pattern).
  - Mitigation: Step 7 explicitly adds `CodeSnippetModel` to both delete loops.

- **Risk**: The integration test in Step 8 AC6 requires a running async DB session, FTS5 table, and real `chunk_node` execution. Tests that call `run_ingestion()` end-to-end require mocking the embedding service and LiteLLM to avoid Ollama dependency in CI.
  - Mitigation: Follow the existing pattern from `test_book_ingestion.py` (or equivalent) — use in-memory SQLite, stub embedder, mock LiteLLM, skip graph/NER if needed.
