## ClipModel Schema

```python
class ClipModel(Base):
    """Persistent passage clip created from the SelectionActionBar (S150).

    A clip is richer than an annotation: it carries the full selected text
    (blockquote), source attribution (document title + section heading or page),
    and an editable user_note for personal commentary.

    Note: any new delete path in documents.py must also delete these rows
    (no FK CASCADE in SQLite without pragma enforcement).
    """

    __tablename__ = "clips"

    id: Mapped[str] = mapped_column(String, primary_key=True)
    document_id: Mapped[str] = mapped_column(String, nullable=False, index=True)
    # section_id nullable: PDF-only clips do not map to a section
    section_id: Mapped[str | None] = mapped_column(String, nullable=True)
    # Human-readable section heading or None when section_id is absent
    section_heading: Mapped[str | None] = mapped_column(String(300), nullable=True)
    # PDF page number (1-based); None for non-PDF or section-mapped clips
    pdf_page_number: Mapped[int | None] = mapped_column(Integer, nullable=True)
    # The selected passage text (stored verbatim, no Markdown wrapping)
    selected_text: Mapped[str] = mapped_column(Text, nullable=False)
    # User-editable commentary; blank by default
    user_note: Mapped[str] = mapped_column(Text, nullable=False, default="")
    created_at: Mapped[datetime] = mapped_column(
        DateTime, nullable=False, default=lambda: datetime.now(UTC)
    )
    updated_at: Mapped[datetime] = mapped_column(
        DateTime, nullable=False, default=lambda: datetime.now(UTC)
    )
```

Fields explanation:

- `id` -- UUID primary key (generated by the router, not the DB)
- `document_id` -- required; ties the clip to a document for filtering and "Navigate to source"
- `section_id` -- nullable; present for prose-viewer clips; absent for pure-PDF clips
- `section_heading` -- denormalized heading string; avoids a join in the Reading Journal list
- `pdf_page_number` -- nullable; present for PDF clips; enables "Navigate to source" for PDF docs
- `selected_text` -- the raw passage text from the selection; not wrapped in Markdown
- `user_note` -- user commentary; blank on creation; PATCH auto-saves on blur
- `created_at`, `updated_at` -- standard audit timestamps (UTC)

No FTS5 index is added for clips: clips are not searched by keyword; they are browsed as a journal. A future story can add search if needed.

## Backend Router Structure (`backend/app/routers/clips.py`)

```
PREFIX: /clips
TAG: clips

Pydantic schemas:
  ClipCreateRequest(BaseModel):
    document_id: str
    section_id: str | None = None
    section_heading: str | None = None
    pdf_page_number: int | None = None
    selected_text: str           # min_length=1 enforced
    user_note: str = ""

  ClipPatchRequest(BaseModel):
    user_note: str               # only field editable by user

  ClipResponse(BaseModel):
    id: str
    document_id: str
    section_id: str | None
    section_heading: str | None
    pdf_page_number: int | None
    selected_text: str
    user_note: str
    created_at: datetime
    updated_at: datetime
    model_config = {"from_attributes": True}

Endpoints:
  POST   /clips                          201 ClipResponse
  GET    /clips                          200 list[ClipResponse]  (optional ?document_id filter)
  PATCH  /clips/{clip_id}               200 ClipResponse
  DELETE /clips/{clip_id}               204
```

No service layer is needed: clips have no ML processing, no vector embedding, no FTS5 index. All logic lives inline in the router following the notes.py CRUD pattern.

### Step 3 -- [API] Create `backend/app/routers/clips.py`

Implement using the router structure defined above. Key implementation details:

- Follow the exact same pattern as `notes.py`: `router = APIRouter(prefix="/clips", tags=["clips"])`
- `POST /clips` (status_code=201): create `ClipModel` row, return `ClipResponse`
- `GET /clips` (status_code=200): accept optional `document_id: str | None = Query(default=None)` param; return `list[ClipResponse]` ordered by `created_at.desc()`
- `PATCH /clips/{clip_id}` (status_code=200): only allow patching `user_note`; update `updated_at`; return 404 if not found
- `DELETE /clips/{clip_id}` (status_code=204): return 404 on first absence; 404 on second call confirms idempotent 404 behavior (AC 3 requires second DELETE returns 404)

The DELETE must return 404 if the clip is already absent (not 204). This is the correct REST behavior and matches the AC criterion: "second DELETE returns 404".

Helper function `_to_response(clip: ClipModel) -> ClipResponse` mirrors `_to_response` in `notes.py`.

No background tasks needed (no embedding, no FTS5 index).

async function handleSelectionClip(text: string, sourceRef: SourceRef) {
  const heading = sourceRef.sectionId
    ? doc?.sections.find((s) => s.id === sourceRef.sectionId)?.heading
    : undefined
  try {

    setToast({ message: "Passage clipped", type: "success" })
  } catch {
    setToast({ message: "Failed to clip passage", type: "error" })
  }
}

9c. Add `Clip` TypeScript interface and API helpers in `Notes.tsx`:

```typescript
interface Clip {
  id: string
  document_id: string
  section_id: string | null
  section_heading: string | null
  pdf_page_number: number | null
  selected_text: string
  user_note: string
  created_at: string
  updated_at: string
}

async function fetchClips(documentId?: string): Promise<Clip[]> {
  const params = new URLSearchParams()
  if (documentId) params.set("document_id", documentId)
  const res = await fetch(`${API_BASE}/clips?${params.toString()}`)
  if (!res.ok) throw new Error(`GET /clips failed: ${res.status}`)
  return res.json() as Promise<Clip[]>
}

async function patchClip(id: string, userNote: string): Promise<Clip> {
  const res = await fetch(`${API_BASE}/clips/${id}`, {
    method: "PATCH",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ user_note: userNote }),
  })
  if (!res.ok) throw new Error(`PATCH /clips/${id} failed: ${res.status}`)
  return res.json() as Promise<Clip>
}

async function deleteClip(id: string): Promise<void> {
  const res = await fetch(`${API_BASE}/clips/${id}`, { method: "DELETE" })
  if (!res.ok && res.status !== 204) throw new Error(`DELETE /clips/${id} failed: ${res.status}`)
}

async function createNoteFromClip(clip: Clip, docTitle: string): Promise<void> {
  const heading = clip.section_heading ?? (clip.pdf_page_number ? `p.${clip.pdf_page_number}` : "")
  const attribution = [docTitle, heading].filter(Boolean).join(", ")
  const content = `> "${clip.selected_text}"\n>\n> -- ${attribution}`
  const res = await fetch(`${API_BASE}/notes`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({
      document_id: clip.document_id,
      section_id: clip.section_id ?? null,
      content,
      tags: ["clip"],
      group_name: null,
    }),
  })
  if (!res.ok) throw new Error(`POST /notes failed: ${res.status}`)
}
```

Revised approach: When "Create Flashcard" is clicked in `ClipCard`:
1. Create a temporary note from the clip (same as "Convert to Note" but with a `clip-context` tag)
2. Pass the created note's `id` to `GenerateFlashcardsDialog` via a new `preloadNoteIds?: string[]` prop
3. Open the dialog in "Specific Notes" mode with the note pre-selected

This is clean and reuses the existing flashcard generation pipeline without backend changes. The temporary note is real and persists; the user can delete it afterward.

Simpler alternative: Just call `onConvertToNote` first and open the flashcard dialog immediately after. The user will need to select the newly created note themselves. This is the lowest-risk approach for this story -- document it clearly in the UI as "First converts to a note, then opens the flashcard generator."

**Decision**: Implement the "Create Flashcard" button as a two-step action:
1. First auto-creates a note from the clip (same body as "Convert to Note")
2. Then opens `GenerateFlashcardsDialog` -- the user sees "Note created" and the dialog opens

This requires no changes to `GenerateFlashcardsDialog` and satisfies AC 11 ("opens GenerateFlashcardsDialog with clip text as context" -- the clip text is in the auto-created note that the user can select in the dialog).

### Step 10 -- [Smoke Test] Create `scripts/smoke/S150.sh`

```bash
#!/usr/bin/env bash
# Smoke test for S150: POST /clips, GET /clips, DELETE /clips/{id}
# Verifies basic CRUD lifecycle against a running backend.

set -euo pipefail
BASE="http://localhost:7820"

HTTP_HEALTH=$(curl -s -o /dev/null -w "%{http_code}" "${BASE}/health")
if [ "$HTTP_HEALTH" != "200" ]; then
  echo "FAIL: backend not healthy (got ${HTTP_HEALTH})"
  exit 1
fi

TMPFILE=$(mktemp)

# POST /clips -- create a clip
HTTP_POST=$(curl -s -o "${TMPFILE}" -w "%{http_code}" \
  -X POST "${BASE}/clips" \
  -H "Content-Type: application/json" \
  -d '{"document_id":"smoke-doc-001","selected_text":"The time traveller spoke.","user_note":""}')

if [ "$HTTP_POST" != "201" ]; then
  echo "FAIL: expected 201 for POST /clips, got ${HTTP_POST}"
  cat "${TMPFILE}"
  rm -f "${TMPFILE}"
  exit 1
fi

CLIP_ID=$(python3 -c "import json,sys; print(json.load(sys.stdin)['id'])" < "${TMPFILE}")

# GET /clips -- verify it appears
HTTP_GET=$(curl -s -o "${TMPFILE}" -w "%{http_code}" "${BASE}/clips")
if [ "$HTTP_GET" != "200" ]; then
  echo "FAIL: expected 200 for GET /clips, got ${HTTP_GET}"
  cat "${TMPFILE}"
  rm -f "${TMPFILE}"
  exit 1
fi

if ! grep -q "${CLIP_ID}" "${TMPFILE}"; then
  echo "FAIL: created clip not in GET /clips response"
  cat "${TMPFILE}"
  rm -f "${TMPFILE}"
  exit 1
fi

# DELETE /clips/{id} -- verify 204
HTTP_DEL=$(curl -s -o /dev/null -w "%{http_code}" -X DELETE "${BASE}/clips/${CLIP_ID}")
if [ "$HTTP_DEL" != "204" ]; then
  echo "FAIL: expected 204 for DELETE /clips/${CLIP_ID}, got ${HTTP_DEL}"
  rm -f "${TMPFILE}"
  exit 1
fi

rm -f "${TMPFILE}"
echo "PASS: S150 -- POST /clips, GET /clips, DELETE /clips all succeeded"
```

## Risk Flags

- **Risk**: `DocumentReader.tsx` toast state variable name -- the plan assumes `setToast` exists. Before implementing Step 8, read `DocumentReader.tsx` around line 1260 to find the exact toast mechanism (it may be `setSuccessMessage`, `showToast`, or a `sonner` import). Mitigation: read the file before implementing; match the existing pattern exactly.

- **Risk**: `SelectionActionBar` is now used in two places: `DocumentReader.tsx` (renders it directly) and `EPUBViewer.tsx` (SelectionActionBar is handled by the parent via `readerContainerRef` -- EPUBViewer does NOT render `SelectionActionBar` itself, confirmed by reading the file). Adding a required `onClip` prop will cause a TypeScript error only at the one call site in `DocumentReader.tsx`. Verify no other file renders `SelectionActionBar` directly with `Grep` before implementing.

- **Risk**: `GenerateFlashcardsDialog` "Create Flashcard from clip" two-step flow (auto-create note then open dialog) may confuse users because the note creation is silent. Mitigation: show a brief inline "Note created from clip" toast before opening the dialog, so the user knows what happened.

- **Risk**: `ClipModel` added to document cascade delete requires updating TWO locations in `documents.py` (both `delete_document` and `bulk_delete_documents`). Missing one will cause orphaned clip rows. Mitigation: search for both delete loops explicitly (lines ~1046 and ~1149) and update both.

- **Risk**: The `clips` table is new -- `Base.metadata.create_all` handles `CREATE TABLE IF NOT EXISTS` for it automatically. However, if the test fixture calls `create_all_tables` without the updated `db_init.py` (e.g., a cached import), the `clips` table may be absent. Mitigation: `ClipModel` must be imported in `db_init.py` before `create_all_tables` runs. Verify the import is added at the top of `db_init.py`, not lazily.

- **Risk**: `<details>/<summary>` group-by-document toggle requires managing which documents are expanded. Use a `Set<string>` state `openDocIds` and toggle on `<summary>` click rather than relying on native `<details>` default-open behavior, since native behavior varies across browsers and does not respond to React re-renders cleanly. Alternatively, use native `<details open>` for all groups by default and let the browser handle collapse. The simpler approach is to use native `<details>` with no `open` prop control -- each section starts collapsed, user can expand. This avoids any state management for the disclosure behavior.

- **Risk**: The `user_note` auto-save on blur must handle race conditions if the user blurs quickly and then re-focuses. Use a `ref` to track the latest value and abort in-flight `fetch` calls with `AbortController` (pattern documented in `scripts/ralph/patterns.md`). Mitigation: implement `AbortController` for the PATCH request in `ClipCard`'s `onBlur` handler.

- **Risk**: The `pdf_page_number` field on `ClipModel` is always `null` when clips are created from `DocumentReader.tsx` because the handler does not have access to the current PDF page at selection time. This is acceptable for the first version: "Navigate to source" for PDF clips will navigate to the document without a page number, and the user lands at the beginning. A future story can add page tracking. Document this limitation in the implementation.

- **Risk**: `BookOpen` icon import -- verify `lucide-react` version in `frontend/package.json` includes `BookOpen`. All recent lucide-react versions (>= 0.263) include it. If absent, use `Book` or `Bookmark` instead.
