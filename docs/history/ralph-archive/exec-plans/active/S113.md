### Step 3 — [Service] Create `backend/app/services/goal_service.py`

This service has no LLM calls. It reads flashcards for a given document and computes retrievability at a future date using the FSRS formula.

**FSRS retrievability formula:**

FSRS uses `R(t, S) = (1 + FACTOR * t / S) ^ DECAY` where:
- `FACTOR = 19/81` (FSRS-4.5 constant)
- `DECAY = -0.5` (FSRS-4.5 constant)
- `t` = elapsed days from the card's `last_review` to `target_date`
- `S` = `fsrs_stability` (stored on `FlashcardModel`)

This is identical to the formula already used in `study.py` `get_study_stats()` which computes `math.exp(-days_since / c.fsrs_stability)` (the SM-2/exponential approximation). For FSRS-4.5 the power-law formula is more accurate. Use the power-law form.

Cards with `fsrs_stability == 0.0` or `last_review is None` are "new" cards (never reviewed). They get `projected_retention = 0.0` and are always counted as at-risk.

**`GoalService` structure:**

```python
import math
import logging
import uuid
from datetime import UTC, date, datetime

from sqlalchemy import select
from sqlalchemy.ext.asyncio import AsyncSession

from app.models import FlashcardModel, LearningGoalModel

logger = logging.getLogger(__name__)

_FSRS_FACTOR = 19 / 81
_FSRS_DECAY = -0.5
_AT_RISK_THRESHOLD = 0.60  # below 60% projected retention = at risk


def _retrievability(stability: float, elapsed_days: float) -> float:
    """FSRS-4.5 power-law retrievability R(t, S).

    Pure function. Returns 1.0 when elapsed_days <= 0 (card not yet due).
    Returns 0.0 when stability is 0 (card never reviewed).
    """
    if stability <= 0.0:
        return 0.0
    if elapsed_days <= 0:
        return 1.0
    return (1 + _FSRS_FACTOR * elapsed_days / stability) ** _FSRS_DECAY


class GoalService:
    async def create_goal(
        self,
        document_id: str,
        title: str,
        target_date: str,  # YYYY-MM-DD
        session: AsyncSession,
    ) -> LearningGoalModel:
        goal = LearningGoalModel(
            id=str(uuid.uuid4()),
            document_id=document_id,
            title=title,
            target_date=target_date,
        )
        session.add(goal)
        await session.commit()
        await session.refresh(goal)
        return goal

    async def list_goals(self, session: AsyncSession) -> list[LearningGoalModel]:
        result = await session.execute(
            select(LearningGoalModel).order_by(LearningGoalModel.created_at.desc())
        )
        return list(result.scalars().all())

    async def delete_goal(self, goal_id: str, session: AsyncSession) -> bool:
        result = await session.execute(
            select(LearningGoalModel).where(LearningGoalModel.id == goal_id)
        )
        goal = result.scalar_one_or_none()
        if goal is None:
            return False
        await session.delete(goal)
        await session.commit()
        return True

    async def compute_readiness(
        self, goal_id: str, session: AsyncSession
    ) -> dict:
        """Compute readiness for a goal.

        Returns a dict with keys:
          on_track: bool
          projected_retention_pct: float  (0-100)
          at_risk_card_count: int
          at_risk_cards: list[dict]  each {id, question, due_date, projected_retention_pct}
        """
        goal_result = await session.execute(
            select(LearningGoalModel).where(LearningGoalModel.id == goal_id)
        )
        goal = goal_result.scalar_one_or_none()
        if goal is None:
            return None  # caller raises 404

        # Parse target_date as a date object
        target = date.fromisoformat(goal.target_date)
        today = datetime.now(UTC).date()
        days_until_target = (target - today).days

        # Fetch all flashcards for this document
        cards_result = await session.execute(
            select(FlashcardModel).where(FlashcardModel.document_id == goal.document_id)
        )
        cards = list(cards_result.scalars().all())

        if not cards:
            # No flashcards generated yet: 0% retention, all "at risk" (none to list)
            return {
                "on_track": False,
                "projected_retention_pct": 0.0,
                "at_risk_card_count": 0,
                "at_risk_cards": [],
            }

        now = datetime.now(UTC)
        at_risk = []
        retention_sum = 0.0

        for card in cards:
            # Compute elapsed days from last_review to target_date
            # If never reviewed: treat last_review = now (worst case: stability=0 -> ret=0)
            if card.last_review and card.fsrs_stability > 0:
                last_review_aware = card.last_review.replace(tzinfo=UTC)
                elapsed = days_until_target + (now - last_review_aware).total_seconds() / 86400
            else:
                elapsed = 0.0  # _retrievability returns 0.0 when stability==0

            ret = _retrievability(card.fsrs_stability, max(0.0, elapsed))
            retention_sum += ret

            if ret < _AT_RISK_THRESHOLD:
                at_risk.append({
                    "id": card.id,
                    "question": card.question,
                    "due_date": card.due_date.isoformat() if card.due_date else None,
                    "projected_retention_pct": round(ret * 100, 1),
                })

        avg_retention = retention_sum / len(cards)
        projected_pct = round(avg_retention * 100, 1)

        return {
            "on_track": projected_pct >= 80.0,
            "projected_retention_pct": projected_pct,
            "at_risk_card_count": len(at_risk),
            "at_risk_cards": at_risk,
        }


_goal_service: GoalService | None = None


def get_goal_service() -> GoalService:
    global _goal_service  # noqa: PLW0603
    if _goal_service is None:
        _goal_service = GoalService()
    return _goal_service
```

Note: `_retrievability` is a pure function — no I/O, all inputs explicit — satisfying Core Belief #19.

### Step 4 — [API] Create `backend/app/routers/goals.py`

```python
"""Learning goals endpoints.

Routes:
  POST   /goals                    — create a goal
  GET    /goals                    — list all goals
  DELETE /goals/{id}              — remove a goal
  GET    /goals/{id}/readiness    — FSRS-based readiness projection
"""
import logging
from datetime import datetime, date

from fastapi import APIRouter, Depends, HTTPException
from pydantic import BaseModel, field_validator
from sqlalchemy.ext.asyncio import AsyncSession

from app.database import get_db
from app.services.goal_service import get_goal_service

logger = logging.getLogger(__name__)

router = APIRouter(prefix="/goals", tags=["goals"])


class CreateGoalRequest(BaseModel):
    document_id: str
    title: str
    target_date: str  # YYYY-MM-DD

    @field_validator("target_date")
    @classmethod
    def validate_date(cls, v: str) -> str:
        try:
            date.fromisoformat(v)
        except ValueError as exc:
            raise ValueError("target_date must be YYYY-MM-DD") from exc
        return v


class GoalResponse(BaseModel):
    id: str
    document_id: str
    title: str
    target_date: str
    created_at: datetime

    model_config = {"from_attributes": True}


class ReadinessResponse(BaseModel):
    on_track: bool
    projected_retention_pct: float
    at_risk_card_count: int
    at_risk_cards: list[dict]


@router.post("", response_model=GoalResponse, status_code=201)
async def create_goal(
    req: CreateGoalRequest,
    session: AsyncSession = Depends(get_db),
) -> GoalResponse:
    svc = get_goal_service()
    goal = await svc.create_goal(
        document_id=req.document_id,
        title=req.title,
        target_date=req.target_date,
        session=session,
    )
    logger.info("Goal created", extra={"goal_id": goal.id, "document_id": goal.document_id})
    return GoalResponse.model_validate(goal)


@router.get("", response_model=list[GoalResponse])
async def list_goals(
    session: AsyncSession = Depends(get_db),
) -> list[GoalResponse]:
    svc = get_goal_service()
    goals = await svc.list_goals(session)
    return [GoalResponse.model_validate(g) for g in goals]


@router.delete("/{goal_id}", status_code=204)
async def delete_goal(
    goal_id: str,
    session: AsyncSession = Depends(get_db),
) -> None:
    svc = get_goal_service()
    found = await svc.delete_goal(goal_id, session)
    if not found:
        raise HTTPException(status_code=404, detail="Goal not found")


@router.get("/{goal_id}/readiness", response_model=ReadinessResponse)
async def get_readiness(
    goal_id: str,
    session: AsyncSession = Depends(get_db),
) -> ReadinessResponse:
    svc = get_goal_service()
    result = await svc.compute_readiness(goal_id, session)
    if result is None:
        raise HTTPException(status_code=404, detail="Goal not found")
    return ReadinessResponse(**result)
```

## FSRS Readiness Calculation Approach

The FSRS-4.5 power-law retrievability formula is:

```
R(t, S) = (1 + FACTOR * t / S) ^ DECAY
```

where:
- `FACTOR = 19/81 ≈ 0.2346`
- `DECAY = -0.5`
- `t` = elapsed days between `last_review` and `target_date`
- `S` = `fsrs_stability` stored on `FlashcardModel`

**Why not the exponential approximation?** The existing `study.py` `get_study_stats()` uses `math.exp(-t/S)`, which is the SM-2/Ebbinghaus approximation. FSRS-4.5 uses the power-law formula above, which is what the `fsrs` Python library models. For consistency with the algorithm, use the power-law. The difference is noticeable at high stability values where the power-law decays more slowly.

**Elapsed days computation for a future target date:**

```
t = (target_date - today).days + days_since_last_review
```

Where `days_since_last_review = (now - last_review).total_seconds() / 86400`. This is the total time the card will have gone unreviewed by the target date, assuming no review is done between now and then (pessimistic projection). This is intentional: the projection tells the user what will happen if they do nothing.

**New cards (never reviewed):** `fsrs_stability = 0.0` → `_retrievability` returns `0.0` → counted as at-risk. This is correct: a card never reviewed has zero projected retention.

**Cards reviewed today with high stability:** `last_review ≈ now`, `elapsed = days_until_target`. Even at stability=10, a 30-day target gives `R = (1 + 0.2346 * 30 / 10)^(-0.5) ≈ 0.75` (75%). This is intentionally conservative — it reflects real forgetting.

**Aggregation:** `projected_retention_pct = average R across all cards for the document * 100`. This is a simple mean, which treats each card equally. A future improvement could weight by lapse count, but that is out of scope for S113.

## Risk Flags

- **Risk: `target_date` stored as TEXT** — SQLite comparisons on ISO date strings (YYYY-MM-DD) are lexicographically correct, so `ORDER BY target_date` and `WHERE target_date > :today` work. However, if target_date is not ISO-formatted, comparisons break silently. Mitigation: the Pydantic `field_validator` on `CreateGoalRequest.target_date` calls `date.fromisoformat(v)` and raises a 422 on bad input. The `GoalService.compute_readiness()` also calls `date.fromisoformat(goal.target_date)` and would raise a `ValueError` if a bad value were somehow stored; this would surface as a 500. This is acceptable since the only write path is through the validated API.

- **Risk: "Never reviewed" cards producing 0% makes all goals look bad for new users** — A user who has just generated flashcards but not yet reviewed any will see 0% readiness. This is semantically correct (they have not studied yet), but may be surprising. Mitigation: the empty-state message in the Goals panel explains this: "Readiness is based on your FSRS study history. Review your flashcards first." However, this is a UX note, not a code risk.

- **Risk: `compute_readiness` is a synchronous computation loop over all cards** — For a document with 500 cards, the loop is O(N) arithmetic with no I/O inside, running in microseconds. No async concern. The DB fetch is a single `SELECT * FROM flashcards WHERE document_id=?` — one query, not N queries. Acceptable.

- **Risk: `GoalsPanel` triggers N readiness queries on mount if `enabled` starts as `true`** — Mitigated by Step 7: `enabled` starts as `false` per goal; only set to `true` when the user clicks "Check Readiness". This is essential to avoid flooding the backend with N readiness requests when the page loads with a populated goals list.

- **Risk: `DELETE /goals/{id}` in the router has an overlapping route with `GET /goals/{id}/readiness`** — FastAPI resolves `/goals/{goal_id}/readiness` before `/goals/{goal_id}` if registered in the correct order. Verify that `@router.get("/{goal_id}/readiness")` is registered before `@router.delete("/{goal_id}")` in the router. If not, FastAPI will try to match "readiness" as a `goal_id` for the DELETE route. Mitigation: register `GET /{goal_id}/readiness` before `DELETE /{goal_id}` in the goals router.

- **Risk: `ReadinessResponse.at_risk_cards` is `list[dict]`** — Using `list[dict]` instead of a typed Pydantic model is a boundary violation (Core Belief #11). However, defining an inline `AtRiskCardItem` Pydantic model inside the router is cleaner. Mitigation: add `class AtRiskCardItem(BaseModel)` with fields `id`, `question`, `due_date`, `projected_retention_pct` in `goals.py`, and change `ReadinessResponse.at_risk_cards: list[AtRiskCardItem]`. Update `GoalService.compute_readiness()` to return a plain dict (the router converts it to `ReadinessResponse(**result)`). This is the correct approach and must be implemented.

- **Risk: SQLite FK pragma** — `create_all_tables` runs `PRAGMA foreign_keys = ON`, but `learning_goals.document_id` has no explicit FK constraint declared in the ORM model (matching the pattern of `FlashcardModel.document_id` which also has no FK constraint). Manual cascading deletes are therefore required: if a document is deleted via `DELETE /documents/{id}`, orphan `learning_goals` rows will remain. This is existing tech debt (TD-010). For S113, add a note to the delete path in `documents.py` to also delete `learning_goals` rows for the deleted `document_id`. Check `documents.py` for the existing delete handler and add `await session.execute(delete(LearningGoalModel).where(LearningGoalModel.document_id == document_id))` before the document delete. This prevents orphaned goals.

- **Risk: TypeScript `date` input format** — `<input type="date">` yields values in `YYYY-MM-DD` format natively in all modern browsers, matching the backend's expected format. No conversion needed. However, the `value` state must be initialized to `""` (empty string) not `null` to avoid React uncontrolled/controlled component warnings.
