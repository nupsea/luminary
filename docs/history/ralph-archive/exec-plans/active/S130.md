## Implementation Steps

### Step 1 -- [Data] Audit and fix 15 failing context_hints, produce three split JSONL files

Pre-work (analysis already complete):

The 15 failing hints divide into two root causes:

**Cause A -- Typographic (curly) quotes in hint text that do not match ASCII quotes in source files (11 entries):**
All are in Alice in Wonderland and The Odyssey entries. The source `.txt` files use ASCII straight quotes (`"` and `'`). The hints contain Unicode curly quotes (`\u201c`, `\u201d` for `"`, `\u2018`, `\u2019` for `'`).

Affected lines in book.jsonl (1-indexed):
- 32 (Alice): `\u201cDRINK ME,\u201d` -- fix to `"DRINK ME,"`
- 33 (Alice): `\u201cEAT ME\u201d` -- fix to `"EAT ME"`
- 37 (Alice): `It\u2019s a Cheshire cat` -- fix to `It's a Cheshire cat`
- 38 (Alice): `Then it doesn\u2019t matter` -- fix to `Then it doesn't matter`
- 40 (Alice): `\u201cOff with his head!\u201d` -- fix to `"Off with his head!"`
- 44 (Alice): `it\u2019s always tea-time, and we\u2019ve no time` -- fix to `it's always tea-time, and we've no time`
- 48 (Alice): `Alice\u2019s first thought was` -- fix to `Alice's first thought was`
- 49 (Alice): `we\u2019re all mad here. I\u2019m mad. You\u2019re mad.` -- fix to `we're all mad here. I'm mad. You're mad.`
- 55 (Odyssey): `Ulysses\u2019 house` -- fix to `Ulysses' house`
- 58 (Odyssey): `monster\u2019s eye` -- fix to `monster's eye`
- 60 (Odyssey): `men\u2019s ears` -- fix to `men's ears`
- 66 (Odyssey): `father\u2019s house` -- fix to `father's house`

**Cause B -- Hint text does not appear verbatim in source; hint was authored against a slightly different edition or is a paraphrase (3 entries):**
- Line 9 (Time Machine): hint says `"on its career"` -- source says `"on its interminable voyage"`. Fix: truncate hint before divergence: `"Psychologist himself who sent forth the model Time Machine on its"`
- Line 13 (Time Machine): hint says `"There are balloons."` (with trailing period) but source has `"There are balloons"` (no period at that point). Fix: remove trailing period from hint.
- Line 16 (Time Machine): hint `It\'s against reason` uses a backslash-escaped apostrophe in the stored JSON (actually single ASCII quote). Confirmed in source. This is a straight-quote match failure due to curly single quote in the actual stored hint byte: `\u2019`. Fix to ASCII apostrophe.

Action: write `book_time_machine.jsonl` (rows 1-30), `book_alice.jsonl` (rows 31-50), `book_odyssey.jsonl` (rows 51-70), applying the fixes listed above. After writing, run the audit script to verify 70/70 match rate across all three files combined.

### Step 2 -- [Eval runner] Fix ensure_ingested() in evals/run_eval.py

Current behavior: `ensure_ingested()` only checks the in-memory manifest dict. If `manifest.json` is missing or stale, it calls `ingest_document()` unconditionally, creating a new duplicate document row with a fresh `doc_id`. The new document may not be fully indexed when the eval queries it.

Fix:
1. Add a helper `lookup_document_by_filename(backend_url, filename) -> str | None` that calls `GET {backend_url}/documents` (with no filter) and searches the returned `items` list for a document whose `title` matches `Path(source_file).stem` (the filename without extension and without the directory). Return the `id` if found and `stage == "complete"`, else return `None`.
2. In `ensure_ingested()`, after the manifest-dict check, call `lookup_document_by_filename()` before calling `ingest_document()`. If it returns a `doc_id`, store it in the manifest, save the manifest, and return it -- no re-ingest.
3. The `GET /documents` endpoint returns a paginated response: `{"items": [...], "total": N, "page": 1, "page_size": 20}`. Fetch with `page_size=100` to get all documents in one call.

This ensures: re-runs after the first run always find the cached `doc_id` in `manifest.json`; if `manifest.json` was deleted the title-match lookup recovers it without re-ingesting.

## Risk Flags

- **Risk: GET /documents pagination** -- the endpoint defaults to `page_size=20`. A backend with more than 20 documents will not return all titles in one call, potentially missing the target document. **Mitigation**: in `lookup_document_by_filename()`, always pass `page_size=100` to the query; this covers all realistic eval setups. For production with >100 docs, iterate pages -- but this is out of scope for this story since eval pipelines use <=10 documents.

- **Risk: Title matching is fragile** -- `DocumentModel.title` is set from `Path(file.filename).stem` at upload time. For `time_machine.txt`, the title will be `"time_machine"`. `Path("DATA/books/time_machine.txt").stem` is `"time_machine"`. This is a deterministic match. **Mitigation**: use exact stem match (case-insensitive to be safe); document the matching rule in a comment.

- **Risk: book.jsonl entries 32, 33 context_hint fix** -- entries originally have Unicode left/right double-quote characters. The source files (Project Gutenberg plain text) use ASCII `"`. Replacing curly quotes with ASCII quotes in the per-book split files is the correct fix. **Mitigation**: verify with the audit script after writing the files.

- **Risk: Existing tests reference "book" dataset** -- `test_evals_router.py` has `test_post_run_returns_202` which creates `golden/book.jsonl` and tests with `dataset: "book"`. Since we preserve `book.jsonl`, this test continues to pass. **Mitigation**: no action needed.

- **Risk: Monitoring tab per-book row display** -- the EvalPanel table renders rows from `GET /evals/results`. The `key={item.dataset}` prop uses the dataset name as the React key. After splitting, the old `"book"` row may still appear if old `scores_history.jsonl` entries exist alongside new per-book entries. This is cosmetically confusing but not a bug. **Mitigation**: document in the PR that operators should run eval for each of the three new datasets to populate their rows; the old `"book"` row will remain until `scores_history.jsonl` is cleared or the old entry expires (it won't -- it's a JSONL append-log). This is acceptable behavior.

- **Risk: MRR threshold AC** -- AC5 requires HR@5 >= 0.60 and MRR >= 0.45 for at least two of the three datasets after running against a backend with all 3 books ingested. This requires the `/search` endpoint to return 10 or more chunks per query (currently defaults to `limit=20`, so 20 chunks are retrieved). The `search_chunks()` function in `run_eval.py` already fetches the top 5 from those. With the context_hint fixes and correct document scoping (via manifest), the MRR should improve significantly. However, if chunking still splits across passage boundaries, MRR may stay below 0.45 for some datasets. This AC is a stretch goal dependent on retrieval quality. **Mitigation**: the story plan addresses the manifest fix (primary MRR regression cause per TD-015) and hint quality fix. If thresholds are still not met after these fixes, a follow-up story must address chunk boundary strategy.
