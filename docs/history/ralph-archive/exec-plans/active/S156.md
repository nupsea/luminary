# S156 -- Structured rubric scoring for Feynman and Teach-back

## Goal

Replace the single integer score in teach-back and Feynman session evaluations with a structured three-dimension rubric (accuracy, completeness, clarity) backed by source-text evidence, and expose each missed concept as an inline "Create flashcard" button in the results UI.

## Key Implementation Details

### TeachBackRubric TypedDict structure

The rubric has three dimensions matching the story spec exactly:

```python
class TeachBackRubricDimension(TypedDict):
    score: int          # 0-100
    evidence: str       # quoted or paraphrased source text

class TeachBackCompletenessDimension(TypedDict):
    score: int          # 0-100
    missed_points: list[str]  # each a concise concept phrase

class TeachBackRubric(TypedDict):
    accuracy: TeachBackRubricDimension
    completeness: TeachBackCompletenessDimension
    clarity: TeachBackRubricDimension
```

Note: `accuracy` and `clarity` share the same shape (score + evidence string). `completeness` has `missed_points` (list) instead of `evidence` (string) so it is a separate sub-TypedDict. This distinction matches the story's user journey ("missed: 'event replay', ...").

### Rubric LLM prompt template

The prompt must include the source section text to ground the evidence. The existing teachback endpoint already has access to `card.answer` (the flashcard answer text). For teachback the rubric context is the answer text itself. For Feynman, the section context string is already fetched inside `complete_session()` via `_get_section_context()`.

```python
_RUBRIC_SYSTEM = (
    "You are an expert tutor evaluating a student explanation. "
    "Output a JSON object only -- no preamble, no markdown fences."
)

_RUBRIC_USER_TMPL = (
    "Source material:\n{source_context}\n\n"
    "Student explanation:\n{explanation}\n\n"
    "Evaluate on three dimensions. "
    "For accuracy: score 0-100 and quote specific evidence from the source material supporting or contradicting the student. "
    "For completeness: score 0-100 and list missed_points as short concept phrases the student omitted. "
    "For clarity: score 0-100 and give a one-sentence comment on explanation quality. "
    'Output JSON: {{"accuracy": {{"score": int, "evidence": str}}, '
    '"completeness": {{"score": int, "missed_points": [str]}}, '
    '"clarity": {{"score": int, "evidence": str}}}}'
)
```

Per Core Belief #20, the prompt is minimal and states each instruction once. The word "evidence" is reused for clarity's feedback string to keep the schema symmetric (the frontend renders it as a comment string in both cases).

### Fallback on malformed LLM JSON

If `json.loads()` raises `json.JSONDecodeError` or the parsed object is not a dict with the required keys, the code logs a warning and leaves `rubric_json = None`. The existing `score`, `correct_points`, `missing_points`, `misconceptions` fields on `TeachbackResultModel` remain unchanged and continue to hold the legacy values. This guarantees no 500 is raised.

```python
def _parse_rubric(raw: str) -> dict | None:
    """Parse rubric JSON from LLM response. Returns None on any parse failure."""
    cleaned = raw.strip()
    if cleaned.startswith("```"):
        lines = cleaned.splitlines()
        cleaned = "\n".join(lines[1:-1]) if len(lines) > 2 else cleaned
    try:
        parsed = json.loads(cleaned)
    except (json.JSONDecodeError, ValueError):
        logger.warning("Failed to parse rubric JSON: %r", raw[:200])
        return None
    if not isinstance(parsed, dict):
        return None
    required = {"accuracy", "completeness", "clarity"}
    if not required.issubset(parsed.keys()):
        logger.warning("Rubric JSON missing required keys: %s", set(parsed.keys()))
        return None
    return parsed
```

Note: `_parse_rubric` is a pure function (no I/O) -- testable with plain `assert` per Core Belief #19.

### Frontend: RubricCard component

A new standalone file `frontend/src/components/RubricCard.tsx`. It receives a typed `rubric` prop and a `documentId` prop (for pre-filling flashcard creation):

```tsx
interface RubricDimension {
  score: number
  evidence: string
}

interface RubricCompleteness {
  score: number
  missed_points: string[]
}

export interface Rubric {
  accuracy: RubricDimension
  completeness: RubricCompleteness
  clarity: RubricDimension
}

interface RubricCardProps {
  rubric: Rubric
  documentId: string
}
```

Score badge color logic (pure function, no side effects):
```tsx
function scoreBadgeClass(score: number): string {
  if (score >= 80) return "bg-green-100 text-green-700"
  if (score >= 50) return "bg-yellow-100 text-yellow-700"
  return "bg-red-100 text-red-700"
}
```

The component renders three sections:
1. **Accuracy** -- score badge (green/yellow/red) + evidence string
2. **Completeness** -- score badge + list of `missed_points`; each missed_point has an inline "Create flashcard" button
3. **Clarity** -- score badge + evidence string (rendered as comment)

The "Create flashcard" button for each missed_point:
- Opens a simple inline pre-filled create dialog (NOT `GenerateFlashcardsDialog` -- that dialog is notes-scoped and complex; this is a direct card creation)
- Uses `POST /flashcards/create-trace` with `question="What is {missed_point}?"` and empty answer for the user to fill, OR opens a small inline form
- Simpler approach: inline dialog or popover that pre-fills a question based on the missed_point text and calls `POST /flashcards/create-trace`

Note: `GenerateFlashcardsDialog` is note-scoped (tag or note_ids). The story says "pre-filled flashcard dialog opens". The closest existing endpoint that creates a card directly without LLM overhead is `POST /flashcards/create-trace` (used by Predict-then-Run). The "Create flashcard" button opens a small inline form with:
- question pre-filled: `"Explain: {missed_point}"` (editable)
- answer field: empty (user fills in)
- document_id: from prop
- On submit: POST /flashcards/create-trace; on success show "Added" inline

This is simpler than spawning `GenerateFlashcardsDialog` and avoids a modal-within-a-modal UX problem.

## Risk Flags

- **Risk: Two sequential LLM calls in teachback endpoint** -- The endpoint now makes two sequential LLM calls (legacy evaluation + rubric). If the first call succeeds and the second fails, the legacy result is still persisted. The try/except around the rubric call prevents 500. However, response latency will roughly double on the happy path. (Mitigation: the rubric call is wrapped in bare `except Exception` so it never blocks the response; total latency doubles but does not exceed timeout. Document this in the endpoint docstring.)

- **Risk: FeynmanSessionModel rubric evaluated after commit** -- The rubric evaluation in `complete_session()` happens after the first `await db_session.commit()` (which marks status=complete and creates flashcards). A second commit is then needed to persist `rubric_json`. If the second commit fails, the session is still marked complete with `rubric_json = null`. This is acceptable degradation (rubric fallback). (Mitigation: wrap the entire rubric block in try/except; the bare `except Exception` ensures the session is not left in a broken state.)

- **Risk: AsyncSession reuse after commit** -- After the first `await db_session.commit()`, accessing `feynman_session.rubric_json =` on the already-committed object requires the session factory to have `expire_on_commit=False` (which it does per patterns.md). The object remains accessible after commit. (Mitigation: confirmed by existing pattern in feynman_service.py which commits and then accesses the session object again.)

- **Risk: Inline flashcard creation form in RubricCard vs modal nesting** -- The story says "opens GenerateFlashcardsDialog pre-filled". However, `GenerateFlashcardsDialog` is note-scoped (tag or note_ids) and does not support pre-filling a missed concept. Opening a modal inside `FeynmanDialog` (which is itself a modal) would create a modal-within-a-modal stacking issue. The plan uses an inline mini-form instead (using local component state). This deviates from the story's literal wording but achieves the same UX goal without the modal nesting problem. (Mitigation: the AC says "Create flashcard opens GenerateFlashcardsDialog pre-filled" but `DocumentFlashcardDialog` (used in DocumentReader) is a better analog -- or just use `POST /flashcards/create-trace` directly with an inline form. Flag for human review if behavior needs to exactly match `GenerateFlashcardsDialog`.)

- **Risk: _parse_rubric duplication between study.py and feynman_service.py** -- Both files define the same `_parse_rubric()` function. Extracting to a shared util module would require a new `app/utils/` or `app/services/rubric_eval.py` file, which creates a new layer artifact. Since both modules are at the same layer and the function is small (15 lines), duplication is acceptable here. (Mitigation: document the duplication with a comment referencing S156 in both files.)

- **Risk: Layer compliance for rubric prompt constants** -- `_RUBRIC_SYSTEM` and `_RUBRIC_USER_TMPL` are string constants used in both `study.py` (API layer) and `feynman_service.py` (Service layer). Defining them in `types.py` would be wrong (types layer = no logic). Defining them in a new `services/rubric_eval.py` service and importing it from both `study.py` and `feynman_service.py` is correct per layer order (API can import Service). However, the simpler approach -- duplicating the 6-line constants -- avoids adding another service file for what is essentially two string templates. Given story scope, duplication wins here. (Mitigation: mark with `# S156` comment in both files.)

- **Risk: TeachBackResultCard rubric field in chat graph path** -- The chat graph `teach_back_node` produces a `TeachBackCardData` shape via SSE, but does not call `POST /study/teachback`. Adding `rubric?: Rubric | null` as optional to `TeachBackCardData` is backward-compatible (undefined == fallback to legacy UI). The chat graph path is not changed by this story. (Mitigation: the optional field with `| null | undefined` fallback ensures no TS errors; legacy rendering path is preserved when rubric is absent.)

- **Risk: uv only** -- no new Python packages needed; `json` is stdlib, `litellm` and `sqlalchemy` are already installed.

7. **Structured logging** -- all new log calls use `logger.warning(...)` with `%` formatting; no `print()` calls
