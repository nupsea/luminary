# S136 — Architecture Diagram to Knowledge Graph: Diagram-Type Routing and COMPONENT Node Extraction

## Goal

Implement DiagramExtractorService that reads diagram-type ImageModel rows (after image_analyze completes), routes by diagram type, calls an LLM to extract typed nodes and edges, writes them to Kuzu as new node/edge tables, links extracted nodes to existing LIBRARY/TECHNOLOGY entities via DEPICTS edges, and surfaces them in the Viz tab with hexagon shape, Diagrams toggle, and image thumbnail in sidebar.

## Kuzu Schema Additions

All new tables added to `KuzuService._create_schema()` using `CREATE ... IF NOT EXISTS`:

### New Node Tables

```cypher
CREATE NODE TABLE IF NOT EXISTS DiagramNode(
    id STRING PRIMARY KEY,
    name STRING,
    node_type STRING,       -- COMPONENT | ACTOR | ENTITY_DM | STEP
    source_image_id STRING, -- FK to ImageModel.id (informational, not enforced by Kuzu)
    document_id STRING,
    frequency INT64
)
```

Note: A single `DiagramNode` table typed by `node_type` field is used instead of four separate tables. This keeps the schema flat and avoids combinatorial explosion of DEPICTS edge variants. The `node_type` field is the discriminator for Viz rendering. The `get_entities_by_type` query on the Entity table already filters by `type`; diagram nodes need a separate query via `get_diagram_nodes_for_document`.

### New Edge Tables

```cypher
-- architecture_diagram edges
CREATE REL TABLE IF NOT EXISTS CONNECTS_TO(FROM DiagramNode TO DiagramNode, document_id STRING, label STRING)
CREATE REL TABLE IF NOT EXISTS STORES_IN(FROM DiagramNode TO DiagramNode, document_id STRING)

-- sequence_diagram edges (CALLS_SEQ avoids collision with existing CALLS Entity->Entity)
CREATE REL TABLE IF NOT EXISTS CALLS_SEQ(FROM DiagramNode TO DiagramNode, document_id STRING, message STRING)

-- sequence_diagram send edges
CREATE REL TABLE IF NOT EXISTS SENDS_TO(FROM DiagramNode TO DiagramNode, document_id STRING, message STRING)

-- er_diagram edges
CREATE REL TABLE IF NOT EXISTS HAS_FIELD(FROM DiagramNode TO DiagramNode, document_id STRING)
CREATE REL TABLE IF NOT EXISTS REFERENCES_DM(FROM DiagramNode TO DiagramNode, document_id STRING)

-- flowchart edges
CREATE REL TABLE IF NOT EXISTS LEADS_TO(FROM DiagramNode TO DiagramNode, document_id STRING, condition STRING)

-- name-match linkage: DiagramNode -> Entity (e.g. COMPONENT "PostgreSQL" -> LIBRARY Entity "PostgreSQL")
CREATE REL TABLE IF NOT EXISTS DEPICTS(FROM DiagramNode TO Entity, document_id STRING)
```

## DiagramExtractorService Design

### Location

`backend/app/services/diagram_extractor.py`

### Class Interface

```python
_QUALIFYING_TYPES: frozenset[str] = frozenset({
    "architecture_diagram",
    "sequence_diagram",
    "er_diagram",
    "flowchart",
})

_DIAGRAM_PROMPT_TEMPLATE: str  # per-type prompt template (see below)


class DiagramExtractorService:

    async def extract(self, document_id: str) -> int:
        """Process all ImageModel rows with qualifying image_type and non-null description.

        Returns count of images for which Kuzu nodes were successfully written.
        Skips images with description=null (image_analyze has not yet run for them).
        Idempotent: uses DiagramNode id = f"{image_id}:{node_name_lower}" as PK,
        so re-runs do not create duplicate nodes.
        """

    def _build_prompt(self, image_type: str, description: str) -> str:
        """Return a diagram-type-specific extraction prompt.

        Returns a plain-text prompt string; pure function, no I/O.
        """

    def _parse_llm_response(self, raw: str) -> dict:
        """Parse JSON from LLM response.

        Returns {"nodes": [...], "edges": [...]} or raises ValueError on invalid JSON.
        Strips optional markdown fences before parsing.
        Pure function, no I/O.
        """

    async def _write_to_kuzu(
        self,
        document_id: str,
        image_id: str,
        image_type: str,
        nodes: list[dict],
        edges: list[dict],
    ) -> None:
        """Write extracted nodes and edges to Kuzu.

        Node dict shape: {"name": str, "node_type": str}
        Edge dict shape: {"from": str, "to": str, "edge_type": str, "properties": dict}
        Runs name-match linkage after writing nodes (DEPICTS edges to Entity table).
        Pure Kuzu calls; no SQLite or LanceDB I/O.
        """

    def _match_entity_by_name(
        self, node_name: str, document_id: str
    ) -> str | None:
        """Return Entity.id if an Entity node with matching name exists for document_id.

        Case-insensitive substring match: returns the first Entity whose lowercase
        name contains the lowercase node_name, or None.
        Pure Kuzu query; no LLM.
        """
```

### Prompt Design (per diagram type)

The prompt instructs the LLM to extract JSON with `nodes` and `edges`. It is minimal and unambiguous (Core Belief 20).

```
architecture_diagram:
  "From this architecture diagram description, extract a JSON object with:
   nodes: list of {name, node_type} where node_type is always 'COMPONENT'
   edges: list of {from, to, edge_type, label} where edge_type is one of CONNECTS_TO, STORES_IN
   Return only JSON."

sequence_diagram:
  "From this sequence diagram description, extract a JSON object with:
   nodes: list of {name, node_type} where node_type is always 'ACTOR'
   edges: list of {from, to, edge_type, message} where edge_type is always SENDS_TO
   Return only JSON."

er_diagram:
  "From this ER diagram description, extract a JSON object with:
   nodes: list of {name, node_type} where node_type is always 'ENTITY_DM'
   edges: list of {from, to, edge_type} where edge_type is one of HAS_FIELD, REFERENCES_DM
   Return only JSON."

flowchart:
  "From this flowchart description, extract a JSON object with:
   nodes: list of {name, node_type} where node_type is always 'STEP'
   edges: list of {from, to, edge_type, condition} where edge_type is always LEADS_TO
   Return only JSON."
```

### LLM Call Pattern

Uses `litellm.acompletion` with `get_settings().LITELLM_DEFAULT_MODEL` (not vision model — description text is already available from image_analyze; no image data needed here). `temperature=0.0`. Semaphore(3) to rate-limit concurrent calls.

### Handler Function

```python
async def diagram_extract_handler(document_id: str, job_id: str) -> None:
    """Enrichment handler for job_type='diagram_extract'.
    Delegates to DiagramExtractorService.extract().
    ServiceUnavailableError propagates to mark job 'failed'.
    """
```

### Enrichment Job Sequencing

The `diagram_extract` job is enqueued AFTER `image_analyze` completes for a document. This is implemented by enqueueing the job in `image_analyze_handler` after `ImageEnricherService.enrich()` returns. Specifically: at the end of `image_analyze_handler` in `image_enricher.py`, check if any qualifying image types were produced and if so, enqueue a `diagram_extract` job.

This approach:
- Uses the existing `EnrichmentQueueWorker` FIFO-per-document guarantee to ensure ordering (image_analyze must complete before diagram_extract starts)
- Does not require a new orchestration mechanism

The actual job creation code in `image_analyze_handler`:

```python
# After svc.enrich() returns -- check for qualifying diagram images
# and enqueue a diagram_extract job if any exist
from sqlalchemy import select as _select
from app.database import get_session_factory as _get_sf
from app.models import ImageModel as _ImageModel, EnrichmentJobModel as _EJModel
import uuid as _uuid
from datetime import UTC, datetime as _dt

async with _get_sf()() as session:
    result = await session.execute(
        _select(_ImageModel.id).where(
            _ImageModel.document_id == document_id,
            _ImageModel.image_type.in_([
                "architecture_diagram", "sequence_diagram", "er_diagram", "flowchart"
            ]),
            _ImageModel.description.is_not(None),
        ).limit(1)
    )
    if result.scalar_one_or_none() is not None:
        job = _EJModel(
            id=str(_uuid.uuid4()),
            document_id=document_id,
            job_type="diagram_extract",
            status="pending",
            created_at=_dt.now(UTC),
        )
        session.add(job)
        await session.commit()
        logger.info("image_analyze_handler: enqueued diagram_extract for doc=%s", document_id)
```

Create `backend/app/services/diagram_extractor.py` with:
- Module-level `_QUALIFYING_TYPES` frozenset

The `extract` method:
1. Loads all `ImageModel` rows where `image_type IN qualifying_types AND description IS NOT NULL` for the document
2. For each image, calls `_build_prompt`, then `litellm.acompletion` (gated by semaphore)
3. Parses response with `_parse_llm_response`
4. Calls `_write_to_kuzu(document_id, image_id, image_type, nodes, edges)`
5. Returns count of successfully processed images
6. Catches and logs per-image errors without aborting the batch; re-raises `ServiceUnavailableError`

The `_write_to_kuzu` method:
1. Calls `get_graph_service().upsert_diagram_node(...)` for each node
2. Calls `get_graph_service().add_diagram_edge(...)` for each edge
3. Calls `_match_entity_by_name(node_name, document_id)` for each node; if match found, calls `get_graph_service().add_depicts_edge(...)`

## Risk Flags

### Risk 1: Kuzu does not support heterogeneous FROM/TO in a single REL table
Some proposed edges have DiagramNode -> Entity (DEPICTS) or DiagramNode -> DiagramNode. Kuzu requires FROM/TO node types to be declared at table creation time. The proposed DDL uses `FROM DiagramNode TO Entity` for DEPICTS, which requires Entity to already exist when DEPICTS is created. Since Entity is created first in `_create_schema`, this is safe. Verify the Kuzu version in use supports this; existing CALLS edge (FROM Entity TO Entity) confirms directed FROM/TO types work.

Mitigation: The `_create_schema` method runs all CREATE statements in order; Entity table is created before any edge tables that reference it. DiagramNode table must be created before DEPICTS and DiagramNode-to-DiagramNode edge tables. Order the DDL statements accordingly.

### Risk 2: `add_diagram_edge` has many edge types -- branching verbosity
Following the existing `add_tech_relation` pattern, Kuzu does not support parameterized relation types, requiring explicit per-type branches. With 7 edge types this becomes repetitive.

Mitigation: Use a helper that dispatches by `edge_type` string using a lookup dict of Cypher strings with `%s` placeholders. This reduces the method to a loop rather than 7 if-elif branches, while still being explicit.

### Risk 3: image_analyze_handler may be called multiple times (retry on failure)
If `image_analyze` fails and is retried, the diagram_extract enqueue check could create duplicate jobs.

Mitigation: Before enqueuing `diagram_extract`, check `enrichment_jobs` for any existing pending/running job with `job_type='diagram_extract'` for this document. Only enqueue if none exists. This is a `SELECT COUNT ... LIMIT 1` check.

### Risk 4: LLM may return nodes not matching edge `from`/`to` names
If the LLM returns edges referencing node names that are not in the `nodes` list, `add_diagram_edge` will fail because the source/target DiagramNode does not exist.

Mitigation: In `_write_to_kuzu`, build a `known_names` set from the extracted nodes dict. Skip edges where `from` or `to` is not in `known_names`. Log a warning per skipped edge.

### Risk 5: Viz hexagon shape requires custom Sigma node program
Sigma does not natively render hexagons. Implementing a WebGL custom node program involves OpenGL shaders.

Mitigation: Use Sigma's built-in `"square"` type for diagram nodes in the initial implementation (visually distinct from the default circle). Add a code comment noting a custom hexagon program can be added as a follow-up polish story. The AC says "hexagon shape" -- confirm with human reviewer whether `"square"` is acceptable, or create a minimal `sigmaHexagonProgram.ts` stub.

### Risk 6: `get_graph_for_document` response shape change breaks frontend
Adding `source_image_id` to GraphNode shape from the API is a backward-compatible addition (optional field). The frontend GraphNode interface already uses `interface GraphNode { ... }` and adding an optional field will not break existing code paths.

Mitigation: Mark the field optional (`source_image_id?: string`) in both Python response dict and TypeScript interface. Ensure `buildGraph` does not crash if `source_image_id` is absent.

### Risk 7: Circular import between image_enricher.py and enrichment_worker.py
`image_analyze_handler` (in image_enricher.py) needs to access `get_session_factory` and `EnrichmentJobModel` to enqueue the diagram_extract job. These are already imported at module level in image_enricher.py (via lazy imports inside the method body). No new module-level imports needed; use the existing lazy import pattern.

### Risk 8: KuzuService._get_diagram_nodes_and_edges must be called from get_graph_for_documents too
`get_graph_for_documents` (multi-doc view) iterates per document. Add a call to `_get_diagram_nodes_and_edges` inside the per-document loop to include diagram nodes in the library-wide Viz view.
