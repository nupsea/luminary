## Data Flow: Entities -> Sections -> Flashcards -> Mastery

The critical join chain is:

```
Kuzu entity (concept name C, for a given document_id)
  -> MENTIONED_IN -> Document(id=document_id)
  (C is linked to sections via GLiNER extraction; but Kuzu stores Entity->Document,
   NOT Entity->Section)

SQLite:
  ChunkModel.section_id  (chunk belongs to a section)
  FlashcardModel.chunk_id -> ChunkModel.id (card was generated from a chunk)
  FlashcardModel.chunk_id -> ChunkModel.section_id (indirect section link)

So the chain is:
  concept name C (from Kuzu, for doc_id)
  -> find SectionModel rows by matching entity name against section text
     (cheapest: query SectionModel.heading LIKE '%C%' OR section preview contains C)
     Better: query ChunkModel.text LIKE '%C%' to get chunk.section_id values,
             then get FlashcardModel where chunk_id in (chunks with that section_id)
  -> FlashcardModel rows scoped to those section_ids
  -> compute mastery from fsrs_stability + bloom_level

IMPORTANT edge case: FlashcardModel.section_id does NOT exist; the link is
FlashcardModel.chunk_id -> ChunkModel.id -> ChunkModel.section_id.
The join requires two hops through SQLite.
```

### Entity-to-Section mapping strategy

Kuzu stores which entities are MENTIONED_IN a document, not which section they appear in. We need to identify the sections where concept C appears. Two approaches are available:

1. **Text search** (chosen, simpler): for each concept C, query `ChunkModel` where `document_id = doc_id AND text ILIKE '%{C}%'` to get `section_id` values. Then get flashcards with `chunk_id IN (chunks for those sections)`.
2. FTS5 search (more complex, not needed).

Since SQLite does not have ILIKE, use `func.lower(ChunkModel.text).contains(concept_name.lower())` via SQLAlchemy or a raw `LIKE '%concept%'` clause.

For cross-book mastery via SAME_CONCEPT edges: call `graph.get_concept_clusters()` which uses union-find on SAME_CONCEPT edges and returns groups. If C belongs to a cluster, include flashcards from all document_ids in the cluster's `document_ids` set, applying the same formula.

### 2. [Service] Create `MasteryService`

File: `/Users/sethurama/DEV/LM/learning-mate/backend/app/services/mastery_service.py`

The service is entirely composed of pure data-assembly logic; no LLM calls.

```python
"""MasteryService: aggregates FSRS flashcard stability into concept-level mastery.

Mastery formula for concept C:
  1. Find all ChunkModel rows for document_ids where chunk.text contains C's name
     (case-insensitive substring match).
  2. For each chunk, find FlashcardModel rows with chunk_id == chunk.id.
  3. For each card:
       weight = 1.5 if bloom_level in (4, 5) else 1.0
       capped_stability = min(card.fsrs_stability / 21.0, 1.0)
       contribution = capped_stability * weight
  4. weighted_mean = sum(contribution) / sum(weight) across all cards
  5. Prediction error penalty: count PredictionEventModel rows where
       document_id in doc_ids AND correct=False AND
       chunk_id in (chunks containing C)
     penalty = min(count * 0.05, 0.20)
  6. mastery = max(0.0, weighted_mean - penalty)

Cross-book: if SAME_CONCEPT clusters exist, unify mastery across all entity
variants in the cluster before applying the formula.
"""
```

**Key methods to implement:**

```python
class MasteryService:

    async def _get_concept_section_ids(
        self,
        concept_name: str,
        document_ids: list[str],
        session: AsyncSession,
    ) -> dict[str, list[str]]:
        """Return {doc_id: [section_id, ...]} where chunks mention concept_name.

        Uses case-insensitive LIKE search on ChunkModel.text.
        Pure SQL, no Kuzu calls.
        """

    async def _get_flashcards_for_sections(
        self,
        section_ids: list[str],
        session: AsyncSession,
    ) -> list[FlashcardModel]:
        """Fetch all flashcards whose chunk_id maps to a chunk in section_ids.

        Uses a two-hop join: FlashcardModel.chunk_id -> ChunkModel.id -> ChunkModel.section_id.
        Returns [] if section_ids is empty.
        """

    async def _get_prediction_errors(
        self,
        concept_name: str,
        document_ids: list[str],
        session: AsyncSession,
    ) -> int:
        """Return count of PredictionEventModel rows with correct=False for concept's chunks.

        Joins PredictionEventModel.chunk_id -> ChunkModel.id
        where ChunkModel.text contains concept_name (case-insensitive).
        """

    def _compute_weighted_mastery(self, cards: list[FlashcardModel]) -> float:
        """Pure function: compute weighted mean mastery from a list of cards.

        bloom_level 4 or 5 -> weight=1.5; all others -> weight=1.0.
        fsrs_stability is capped at 21.0 days (100% mastery).
        Returns 0.0 if cards is empty.
        """

    async def compute_mastery(
        self,
        concept_name: str,
        document_ids: list[str],
        session: AsyncSession,
    ) -> ConceptMastery:
        """Compute mastery score for concept_name across document_ids.

        Handles cross-book mastery via SAME_CONCEPT cluster lookup.
        Returns ConceptMastery with mastery=0.0 and no_flashcards=True when no cards exist.
        """

    async def get_all_concept_masteries(
        self,
        document_ids: list[str],
        session: AsyncSession,
    ) -> list[ConceptMastery]:
        """Return ConceptMastery for all Kuzu entities in the given documents.

        Steps:
        1. Call graph.get_entities_by_type_for_document(doc_id) for each doc_id
           to get all entity names.
        2. Deduplicate using SAME_CONCEPT clusters so cross-book entities appear once.
        3. Call compute_mastery() for each unique concept.
        4. Sort by mastery ascending (weakest first).
        """

    async def get_heatmap(
        self,
        document_id: str,
        session: AsyncSession,
    ) -> MasteryHeatmapResponse:
        """Build a chapter x concept mastery grid for a single document.

        Steps:
        1. Fetch SectionModel rows for document_id, ordered by section_order.
        2. Fetch Kuzu entities for document_id.
        3. For each (section, concept) pair, find flashcards in that section
           mentioning that concept. Compute mastery or None.
        4. Return full grid as HeatmapCell list.

        The grid may be sparse: most (chapter, concept) cells will have
        no cards (mastery=None). Frontend renders gray for None.
        """


@lru_cache
def get_mastery_service() -> MasteryService:
    return MasteryService()
```

**Important implementation note on the entity-to-section join:**

Kuzu entities have names only linked to documents, not sections. To produce the heatmap, identify which sections each entity concept appears in by:

```python
# For get_heatmap: get all chunks per section, check if concept name appears
# This produces a sparse grid -- only fill cells where chunks exist
result = await session.execute(
    select(ChunkModel.section_id, ChunkModel.id)
    .where(ChunkModel.document_id == document_id)
    .where(func.lower(ChunkModel.text).contains(concept_name.lower()))
)
```

SQLite's LIKE is case-insensitive by default for ASCII characters. Use `ChunkModel.text.ilike(f"%{concept_name}%")` in SQLAlchemy, which maps to `LIKE` in SQLite (SQLite LIKE is case-insensitive for ASCII).

**Performance guard**: For `get_all_concept_masteries`, the number of entities can be large (50-200 per document). Use `asyncio.gather` with a `Semaphore(10)` to parallelize `compute_mastery()` calls per concept. Cap at 100 concepts per call to avoid timeout.

## Risk Flags

- **Risk: FlashcardModel has no direct section_id column.** Mitigation: the join is two hops via `FlashcardModel.chunk_id -> ChunkModel.id -> ChunkModel.section_id`. Use a subquery or JOIN in SQLAlchemy. Performance for large flashcard sets: add `WHERE ChunkModel.document_id IN (doc_ids)` to narrow the scan early.

- **Risk: ILIKE/case-insensitive LIKE on ChunkModel.text for concept name matching can be slow on large corpora.** Mitigation: SQLite LIKE for ASCII text is case-insensitive by default (no COLLATE needed); the match `text LIKE '%concept%'` will use a full-table scan on chunks. For the initial implementation, limit concepts to top-50 by MENTIONED_IN frequency from Kuzu before computing mastery. This is a known acceptable tradeoff; note it in a code comment.

- **Risk: GapReport TypedDict change is a breaking change.** Mitigation: `weak` defaults to `[]` in all fallback return paths. All existing callers only read `gaps`, `covered`, `query_used`; adding `weak` is additive and backward-compatible for existing consumers.

- **Risk: get_concept_clusters() traverses all SAME_CONCEPT edges globally.** Mitigation: the call is fast (union-find on in-memory edge list); but the edge list is fetched from Kuzu. For large installations, this could be slow. Use a cache keyed on the graph service instance (acceptable for a single-user local-first app).

- **Risk: Heatmap grid can be very large** (many sections x many concepts). Mitigation: in `get_heatmap()`, cap at 20 sections (by section_order) and 20 concepts (by MENTIONED_IN frequency). Frontend renders a scrollable grid. Document the cap in a code comment.

- **Risk: `no_flashcards` flag allows frontend empty state to distinguish "concept exists but no cards" from "mastery 0.0".** The color coding must handle `null` mastery (no cards = gray) vs `0.0` mastery (cards exist but stability=0 = dark blue). This is correctly handled by the `HeatmapCell.mastery: float | None` design.

- **Risk: TypeScript strict mode** -- `mastery: number | null` in `HeatmapCell` requires null checks throughout the rendering code. The `masteryColor()` function must handle null explicitly (already shown above).

- **Risk: Quick study button** -- routing to Study tab with a pre-filtered card set requires a cross-tab state mechanism that does not exist yet. The plan uses a toast notification as a first implementation. This satisfies the AC "Quick study weak spots starts session containing only cards for concepts with mastery < 0.4" only partially (user must manually navigate). If the reviewer flags this as insufficient, the fallback is to pass concept names as URL query params to `/` and handle them in `App.tsx`. Document this tradeoff in the commit message.
