# S118 -- Desktop Push Notifications for FSRS Due Cards

### Step 1 -- [API] Add `DueCountResponse` schema and `GET /study/due-count` endpoint to `study.py`

Add `DueCountResponse` after the existing `StrugglingCardItem` / `SectionHeatmapItem` schemas (after line 233 in the current file):

```python
class DueCountResponse(BaseModel):
    due_today: int
```

Add the endpoint after `get_section_heatmap` (at the end of the endpoints block, before the helpers section):

```python
@router.get("/due-count", response_model=DueCountResponse)
async def get_due_count(
    session: AsyncSession = Depends(get_db),
) -> DueCountResponse:
    """Return the count of flashcards whose due_date is today or in the past.

    Lightweight endpoint for frontend notification checks -- returns only
    the integer count, not the full card list.
    """
    now = datetime.now(UTC)
    result = await session.execute(
        select(func.count()).select_from(FlashcardModel).where(
            FlashcardModel.due_date <= now
        )
    )
    count = result.scalar_one()
    logger.debug("due-count: %d cards due", count)
    return DueCountResponse(due_today=count)
```

Key invariant details:
- `func.count()` with `select_from(FlashcardModel)` + a WHERE clause avoids fetching all rows; it lets SQLite execute `SELECT COUNT(*) FROM flashcards WHERE due_date <= ?` -- O(index) not O(N).
- `datetime.now(UTC)` is already imported in this file (line 18). No new imports needed.
- `func` is already imported (line 23). No new imports needed.
- The endpoint path `/study/due-count` is consistent with existing paths in this router (`/study/due`, `/study/section-heatmap`, `/study/session-plan`). FastAPI route ordering: `/due-count` is registered before `/due` route would match since they are distinct strings. No conflict.

### Step 2 -- [Types/Frontend] Add `reviewRemindersEnabled` to Zustand store

Modify `frontend/src/store.ts` to add the new field and action. The toggle state does NOT need cross-session persistence via `zustand/middleware` `persist` (which would require adding a new npm dependency for the storage adapter). Instead, the hook will read/write `localStorage` directly for the 4-hour timestamp throttle key. The toggle preference itself WILL be stored in Zustand + localStorage via a simple pattern used in the hook.

On reflection: the story description says "Settings page includes toggle" -- the toggle state needs to survive page reload. The cleanest approach without adding a new dependency is:

- Store `reviewRemindersEnabled` in Zustand (in-memory, for reactive UI)
- On store init, read from `localStorage.getItem("luminary:reviewReminders")` to hydrate
- On `setReviewRemindersEnabled`, also write to `localStorage`

```typescript
// store.ts additions

interface AppState {
  // ...existing fields...
  reviewRemindersEnabled: boolean
  setReviewRemindersEnabled: (enabled: boolean) => void
}

// In create() initializer:
reviewRemindersEnabled: localStorage.getItem("luminary:reviewReminders") !== "false",

// Action:
setReviewRemindersEnabled: (enabled) => {
  localStorage.setItem("luminary:reviewReminders", String(enabled))
  set({ reviewRemindersEnabled: enabled })
},
```

Default is `true` (opt-out model): if the key is absent (first run), reminders are enabled. Only setting `"false"` disables them.

### Step 3 -- [Frontend] Create `frontend/src/hooks/useReviewNotification.ts`

This hook implements the full notification lifecycle:

```typescript
/**
 * useReviewNotification -- S118
 *
 * Behaviour:
 * 1. On first mount: calls Notification.requestPermission() once.
 *    Guards with localStorage key "luminary:notifPermRequested" so it only
 *    prompts once per browser profile.
 * 2. Registers a visibilitychange listener. When document becomes visible:
 *    a. If Notification.permission !== 'granted', skip.
 *    b. If reviewRemindersEnabled is false, skip.
 *    c. Check last notification time from localStorage key
 *       "luminary:lastReviewNotif". If < 4 hours ago, skip.
 *    d. Fetch GET /study/due-count.
 *    e. If due_today > 0, fire new Notification('Luminary', { body: `X cards due for review today` }).
 *    f. Set onclick on the notification to navigate to the Study tab.
 *    g. Write current timestamp to "luminary:lastReviewNotif".
 * 3. Cleans up the visibilitychange listener on unmount.
 */
import { useEffect } from "react"
import { useAppStore } from "@/store"

const API_BASE = "http://localhost:8000"
const FOUR_HOURS_MS = 4 * 60 * 60 * 1000
const PERM_REQUESTED_KEY = "luminary:notifPermRequested"
const LAST_NOTIF_KEY = "luminary:lastReviewNotif"

export function useReviewNotification() {
  const reviewRemindersEnabled = useAppStore((s) => s.reviewRemindersEnabled)

  useEffect(() => {
    // Request permission once per browser profile
    if (
      "Notification" in window &&
      Notification.permission === "default" &&
      !localStorage.getItem(PERM_REQUESTED_KEY)
    ) {
      localStorage.setItem(PERM_REQUESTED_KEY, "1")
      void Notification.requestPermission()
    }
  }, [])

  useEffect(() => {
    async function handleVisibilityChange() {
      if (document.visibilityState !== "visible") return
      if (!("Notification" in window)) return
      if (Notification.permission !== "granted") return
      if (!reviewRemindersEnabled) return

      const lastNotif = parseInt(localStorage.getItem(LAST_NOTIF_KEY) ?? "0", 10)
      if (Date.now() - lastNotif < FOUR_HOURS_MS) return

      try {
        const res = await fetch(`${API_BASE}/study/due-count`)
        if (!res.ok) return
        const data = (await res.json()) as { due_today: number }
        if (data.due_today <= 0) return

        localStorage.setItem(LAST_NOTIF_KEY, String(Date.now()))
        const notif = new Notification("Luminary", {
          body: `${data.due_today} card${data.due_today !== 1 ? "s" : ""} due for review today`,
        })
        notif.onclick = () => {
          window.focus()
          // Navigate to Study tab by updating location
          window.location.hash = ""
          // Use BrowserRouter history -- the hook cannot access useNavigate
          // directly. Dispatch a popstate to trigger React Router navigation.
          // Simpler: set window.location.pathname directly (forces page reload)
          // Correct approach: publish a custom event that AppShell listens to.
          window.dispatchEvent(new CustomEvent("luminary:navigate", { detail: "/study" }))
        }
      } catch {
        // Network error -- silently skip; notification is best-effort
      }
    }

    document.addEventListener("visibilitychange", handleVisibilityChange)
    return () => {
      document.removeEventListener("visibilitychange", handleVisibilityChange)
    }
  }, [reviewRemindersEnabled])
}
```

**Navigation from notification click detail:**

`useNavigate()` from react-router-dom is only usable inside a component inside `<BrowserRouter>`. The hook is called inside `AppShell` (which is inside `BrowserRouter`), so `useNavigate` IS available. Import it and pass the navigate function into the effect via a ref:

```typescript
import { useNavigate } from "react-router-dom"

export function useReviewNotification() {
  const reviewRemindersEnabled = useAppStore((s) => s.reviewRemindersEnabled)
  const navigate = useNavigate()
  // Use a ref so the effect closure always has the current navigate function
  const navigateRef = useRef(navigate)
  useEffect(() => { navigateRef.current = navigate }, [navigate])

  // permission request effect (runs once) ...

  useEffect(() => {
    async function handleVisibilityChange() {
      // ... (as above, but use navigateRef.current("/study") in onclick)
      notif.onclick = () => {
        window.focus()
        navigateRef.current("/study")
      }
    }
    document.addEventListener("visibilitychange", handleVisibilityChange)
    return () => document.removeEventListener("visibilitychange", handleVisibilityChange)
  }, [reviewRemindersEnabled])
}
```

## Risk Flags

- Risk: `Notification` API is not available in all browser contexts (e.g. `http://` without HTTPS on some mobile browsers, or browser extensions sandboxes). Mitigation: all notification code is guarded with `"Notification" in window` before accessing `Notification.permission` or `Notification.requestPermission`. The hook is a no-op if the API is unavailable.

- Risk: `useNavigate` called inside `useReviewNotification` requires the hook to be used inside a component that is a descendant of `<BrowserRouter>`. `AppShell` is rendered inside `<BrowserRouter>` in `App()` (line 299-302 of App.tsx), so this is safe. If the hook were placed in `App()` directly (outside `BrowserRouter`), `useNavigate` would throw. Mitigation: call `useReviewNotification()` only from `AppShell`, not from `App`.

- Risk: The `visibilitychange` event fires on every tab switch and minimize/restore. The 4-hour throttle via `localStorage` is the primary guard against excessive API calls. Additionally, the permission check (`Notification.permission !== "granted"`) and toggle check run synchronously before any `fetch`. The `fetch` is only triggered when all guards pass. Mitigation: guards are O(1) synchronous checks; `fetch` is conditional.

- Risk: `localStorage.getItem("luminary:reviewReminders") !== "false"` default-to-true means first-time users who deny the notification permission prompt are still "enabled" in the toggle. The toggle accurately reflects user intent (they chose to keep reminders on); the notification simply cannot fire because permission is `"denied"`. The SettingsDrawer warning message informs them to fix the browser permission. This is correct UX.

- Risk: The Zustand store currently has no persistence middleware. Adding localStorage read in the store initializer (for `reviewRemindersEnabled`) is a direct `localStorage.getItem()` call during module load. This is fine for a browser environment but would break in server-side rendering (SSR). Luminary is client-only (Vite SPA + Tauri), so SSR is not a concern. Mitigation: add a comment noting this assumption.

- Risk: `func.count()` with `.select_from(FlashcardModel)` in `study.py` requires SQLAlchemy to produce `SELECT COUNT(*) FROM flashcards WHERE due_date <= ?`. Verify the import: `func` is already imported at line 23 of study.py (`from sqlalchemy import func, or_, select`). No new import required.

- Risk: `FlashcardModel.due_date` is stored as a naive datetime in SQLite (no timezone info). The existing `GET /study/due` endpoint compares with `datetime.now(UTC)` (timezone-aware) and works because SQLAlchemy's SQLite dialect strips timezone info for comparison. The new `get_due_count` endpoint uses the same pattern -- safe to replicate.

- Risk: The story description says "clicking notification activates Study tab". In a browser notification, `notification.onclick` fires in the browser's main thread only when the user clicks the notification. If the app window is minimized, `window.focus()` may fail silently on some OS/browser combinations. `navigateRef.current("/study")` will still update the React Router state so the Study tab is active when the window is next focused. Mitigation: call both `window.focus()` and `navigate("/study")` in onclick; document that focus behaviour is OS-dependent.

- Risk: AC story description mentions "Settings page includes toggle" -- there is currently no standalone Settings page/tab in the navigation. The Settings UI lives in `SettingsDrawer` (opened by the gear icon in the sidebar). The plan adds the toggle there, which matches the existing UX pattern. If a dedicated Settings page is required in future, this can be extracted. Mitigation: add toggle to `SettingsDrawer` as described.

## Notes

- No new npm packages required. `useNavigate`, `useEffect`, `useRef` are all already in use in the codebase.
- No new Python packages required. `func` and `datetime` are already imported in `study.py`.
- The `GET /study/due-count` endpoint is intentionally separate from `GET /study/due` (which returns full `FlashcardResponse` objects). The count endpoint is lightweight -- `SELECT COUNT(*)` vs loading all card fields -- and appropriate for frequent polling on `visibilitychange`.
- Three frontend states for the notification feature:
  - Loading: no visible loading state (background fetch; if slow, the guard simply does not fire a notification for this visibilitychange event)
  - Error: silent no-op on fetch failure; SettingsDrawer shows "browser blocked" warning when `permission === "denied"`
  - Empty: `due_today === 0` -> no notification, no UI change
- The 4-hour throttle key `"luminary:lastReviewNotif"` is reset each time a notification fires. If the user has 0 due cards on first focus, the key is NOT updated (preserving the ability to check again later in the same window session when more cards become due -- though FSRS scheduling means cards rarely become due mid-session).
- `localStorage.setItem("luminary:notifPermRequested", "1")` is written BEFORE calling `Notification.requestPermission()` to prevent a race where the user dismisses the prompt, reloads, and gets prompted again before the async resolution sets the stored flag.
