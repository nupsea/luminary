### Step 2 [Service] Create DocumentSearchService

Create `backend/app/services/document_search.py`:

```python
"""In-document FTS5 search service -- returns section-grouped results."""
import logging
import re
from sqlalchemy import text
from app.database import get_session_factory

logger = logging.getLogger(__name__)

_SANITIZE_RE = re.compile(r"[^\w\s]")
_OP_RE = re.compile(r"\b(AND|OR|NOT)\b", re.IGNORECASE)

def _sanitize_fts_query(query: str) -> str:
    cleaned = _SANITIZE_RE.sub(" ", query)
    cleaned = _OP_RE.sub(" ", cleaned)
    return " ".join(cleaned.split())


class DocumentSearchService:
    async def search(
        self,
        document_id: str,
        query: str,
        limit: int = 50,
    ) -> list[dict]:
        """Search chunks_fts scoped to document_id.

        Returns a list of dicts with keys:
          section_id, section_heading, match_count, snippet
        Ordered by match_count DESC, limited to `limit` results.
        Returns [] for empty query.
        """
        if not query or not query.strip():
            return []

        safe_query = _sanitize_fts_query(query)
        if not safe_query:
            return []

        # chunks_fts columns: text(0), chunk_id(1 UNINDEXED), document_id(2 UNINDEXED)
        # snippet() args: table, column_index, start_token, end_token, ellipsis, max_tokens
        sql = text(
            "SELECT c.section_id, "
            "       s.heading AS section_heading, "
            "       COUNT(*) AS match_count, "
            "       snippet(chunks_fts, 0, '<mark>', '</mark>', '...', 24) AS snippet "
            "FROM chunks_fts AS cf "
            "JOIN chunks AS c ON cf.chunk_id = c.id "
            "LEFT JOIN sections AS s ON c.section_id = s.id "
            "WHERE chunks_fts MATCH :query "
            "  AND cf.document_id = :doc_id "
            "GROUP BY c.section_id "
            "ORDER BY match_count DESC "
            "LIMIT :limit"
        )
        async with get_session_factory()() as session:
            rows = (
                await session.execute(
                    sql,
                    {"query": safe_query, "doc_id": document_id, "limit": limit},
                )
            ).fetchall()

        return [
            {
                "section_id": row[0] or "",
                "section_heading": row[1] or "",
                "match_count": row[2] or 0,
                "snippet": row[3] or "",
            }
            for row in rows
        ]


_document_search_service: DocumentSearchService | None = None


def get_document_search_service() -> DocumentSearchService:
    global _document_search_service
    if _document_search_service is None:
        _document_search_service = DocumentSearchService()
    return _document_search_service
```

**Key design notes:**
- `snippet()` column index is 0 (the `text` column in chunks_fts).
- `cf.document_id = :doc_id` filters to a single document. The UNINDEXED column works correctly for equality filter here because we rely on MATCH first (FTS5 applies the MATCH rowset, then filters UNINDEXED columns from those rows). This is safe for the equality filter pattern as confirmed in patterns.md for the diagnostics query.
- `GROUP BY c.section_id` collapses multiple matching chunks in the same section. `snippet()` returns the best snippet from the first matching row in each group (SQLite FTS5 behavior -- acceptable for this use case).
- `LEFT JOIN sections` handles chunks with null section_id gracefully.

### Step 3 [API] Add GET /{document_id}/search endpoint to documents.py

Add to `backend/app/routers/documents.py` after the existing `get_document_diagnostics` endpoint:

```python
@router.get("/{document_id}/search", response_model=list[DocumentSectionSearchResult])
async def search_document_sections(
    document_id: str,
    q: str = Query(..., min_length=1),
) -> list[DocumentSectionSearchResult]:
    """FTS5 keyword search scoped to a single document.

    Returns up to 50 section-level results ordered by match count desc.
    Returns [] for empty/whitespace-only query (min_length=1 enforced by FastAPI).
    Returns 404 if document does not exist.
    """
    from app.services.document_search import get_document_search_service  # noqa: PLC0415

    async with get_session_factory()() as session:
        doc_check = await session.execute(
            select(DocumentModel.id).where(DocumentModel.id == document_id)
        )
        if doc_check.scalar_one_or_none() is None:
            raise HTTPException(status_code=404, detail="Document not found")

    svc = get_document_search_service()
    results = await svc.search(document_id, q, limit=50)
    logger.info(
        "Document search: doc=%s query=%r hits=%d",
        document_id,
        q[:50],
        len(results),
    )
    return [DocumentSectionSearchResult(**r) for r in results]
```

**Validation notes:**
- `q: str = Query(..., min_length=1)` enforces the min-1-char requirement at the FastAPI boundary.
- No separate `q=""` empty check needed in the endpoint -- FastAPI returns 422 for empty string.
- The story says "Returns [] for empty query (not an error)" -- the frontend must not send an empty query; the endpoint only needs to handle the case where q has content but FTS yields no matches.

**Keydown listener (useEffect inside DocumentReader, scoped to readerContainerRef):**
```tsx
useEffect(() => {
  const container = readerContainerRef.current
  if (!container) return
  function handleKeyDown(e: KeyboardEvent) {
    if ((e.metaKey || e.ctrlKey) && e.key === "f") {
      e.preventDefault()
      setSearchOpen(true)
    }
```

**Note on scope:** The `keydown` listener is attached to `readerContainerRef.current` (the left panel div), not `document`. This confines Cmd+F interception to within the DocumentReader component. However, `keydown` on a div requires `tabIndex={0}` or `tabIndex={-1}` for the div to be focusable/receive keyboard events. A better approach is to attach to the `document` but check `readerContainerRef.current.contains(document.activeElement)` first. Use `document`-level listener but scope by checking element contains, which is more reliable:

```tsx

    if ((e.metaKey || e.ctrlKey) && e.key === "f") {
      // Only intercept if user is in the reader area (not in a different tab/input)
      const container = readerContainerRef.current
      if (!container) return
      e.preventDefault()
      setSearchOpen(true)
    }
  }
  document.addEventListener("keydown", handleKeyDown)
  return () => document.removeEventListener("keydown", handleKeyDown)
}, [])
```

Then for Escape close, add separately:
```tsx

    onPrev={() => setSearchHitIndex((i) => (i - 1 + searchResults.length) % searchResults.length)}
```

**InDocSearchBar implementation:**
- `useDebounce(inputValue, 300)` for the API call
- `useEffect` on debouncedQuery: if empty string, call `onResults([])` and return; else fetch `GET /documents/{id}/search?q={query}`
- Loading state: replace search icon with `<Loader2 size={12} className="animate-spin" />` inside the input
- Error state: inline text `"Search failed"` below the input row (destructive color, not a toast)
- Empty state (non-empty query + no results): text `"No matches in this document"`
- Match counter: `"{hitIndex + 1} of {totalHits}"` (hide when totalHits === 0)
- Prev/Next buttons: ChevronUp/ChevronDown icons
- X button: closes bar, clears results

**Navigation behavior (prev/next):**
When `searchHitIndex` changes and `searchResults[searchHitIndex]` exists, scroll the section element into view:
```tsx

  if (el) el.scrollIntoView({ behavior: "smooth", block: "nearest" })
```

## Risk Flags

- **Risk: FTS5 snippet() column index off by one.** The chunks_fts table is defined as `fts5(text, chunk_id UNINDEXED, document_id UNINDEXED)`. UNINDEXED columns do not count in the snippet() column index -- column 0 is `text`. Verified from db_init.py DDL. (Mitigation: confirmed from schema; test with real FTS5 data.)

- **Risk: UNINDEXED column filter unreliability for large tables.** patterns.md documents that `WHERE col = :val` on UNINDEXED columns is unreliable when tables accumulate many rows. However, the query uses `chunks_fts MATCH :query` first (which returns a bounded rowset), then filters by `document_id` from that rowset. This is safe because FTS5 applies MATCH before UNINDEXED column filtering, bounding the candidate set. The same pattern is used in the retriever and diagnostics endpoints. (Mitigation: if reliability issues arise in large corpora, add a subquery joining to the chunks table shadow instead.)

- **Risk: dangerouslySetInnerHTML XSS vector.** The snippet comes from SQLite FTS5 output wrapping user-uploaded content in `<mark>` tags. For a local-first app this is acceptable, but add a simple sanitize step that only allows `<mark>...</mark>` tags. (Mitigation: strip all HTML tags except `<mark>` before passing to dangerouslySetInnerHTML.)

- **Risk: Cmd+F listener conflicts with existing document-level Cmd+K shortcut (SearchDialog).** The SearchDialog likely has its own Cmd+K listener on document. Both listeners will fire for their respective keys -- they do not conflict. However, if SearchDialog has a catch-all that intercepts Cmd+F, verify in DocumentReader.tsx that there is no global handler already blocking Cmd+F. Reading the current code, no Cmd+F handler exists; the existing keydown listeners are for Cmd+K. (Mitigation: no action needed; document confirmed clean.)

- **Risk: Ring class conflicts with existing fragility heatmap border classes.** Section items currently apply `fragilityClass` or `admonitionClass` as border-left styles. Adding `ring-2 ring-primary` as a separate class is additive and does not conflict with border-left styles. (Mitigation: use `cn()` to compose classes; both can coexist visually.)

- **Risk: Frontend TypeScript strict mode.** `searchResults[searchHitIndex]` may be undefined (e.g., when `searchResults` is empty but hitIndex is 0). Guard with optional chaining: `searchResults[searchHitIndex]?.section_id`. (Mitigation: all array accesses must use optional chaining.)

- **Risk: Section scroll when leftTab is not "sections".** If the user navigates to "highlights" or "pdfview" while a search is open, the section elements may be hidden. Scrolling to a hidden element is a no-op. Close the search bar when leftTab changes away from "sections". Add a `useEffect([leftTab])` that calls `setSearchOpen(false)` and clears results when `leftTab !== "sections"`. (Mitigation: implement this guard.)
