# S144 — Feynman Technique Mode

## Goal

Add a guided Feynman technique learning mode: section-scoped explanation sessions with SSE-streamed tutor turns, automatic gap identification, remediation flashcard generation, and objective coverage updates.

- **API** — new feynman router (backend/app/routers/feynman.py); registered in main.py

```python
    id: Mapped[str] = mapped_column(String, primary_key=True)
```

### 3. [Service] Create feynman_service.py

**File**: `backend/app/services/feynman_service.py`

Key design decisions:
- System prompt includes the full section summary content (fetched from SectionSummaryModel) in the context. If no section summary exists, fall back to the section preview from SectionModel.
- The tutor prompt uses a structured role: "Socratic tutor using the Feynman technique". It asks the learner to explain the concept as if teaching it to a 12-year-old. Each tutor response:
  1. Gives targeted feedback on the learner's explanation
  2. Identifies specific gaps (misunderstandings or missing concepts) as a JSON list in a `gaps:` block
  3. Follows up with a clarifying question
- Gap parsing is a pure function `_parse_gaps(raw: str) -> list[str]` that extracts the `gaps:` JSON block.
- `complete_session()` collects all gaps across all tutor turns, deduplicates, calls `FlashcardService.generate_from_gaps()` with source='feynman' and flashcard_type='concept_explanation', then calls `ObjectiveTrackerService.update_coverage()` for the document.
- Offline degradation: `litellm.ServiceUnavailableError` and `litellm.APIConnectionError` are caught and re-raised as `HTTPException(503)`.
- All LLM calls go through `get_llm_service()`.

**Session lifecycle**:

```
create_session(document_id, section_id, concept, session) -> (FeynmanSessionModel, opening_message: str)
  - fetch section summary (SectionSummaryModel) or section preview (SectionModel.preview)
  - create FeynmanSessionModel row with status='active'
  - call LLM (non-streaming) to generate opening tutor message
  - persist FeynmanTurnModel(role='tutor', turn_index=0, content=opening_message, gaps_identified=[])
  - return (session_row, opening_message)

stream_turn(session_id, learner_content, session) -> AsyncGenerator[str, None]
  - load session; raise 404 if not found
  - persist FeynmanTurnModel(role='learner', turn_index=N, content=learner_content)
  - fetch all prior turns ordered by turn_index to build conversation history
  - fetch section summary for system context
  - call litellm.acompletion(..., stream=True)
  - accumulate full response; parse gaps from accumulated text
  - yield SSE tokens as: data: {"token": "..."}\n\n
  - after streaming done: persist FeynmanTurnModel(role='tutor', turn_index=N+1, gaps_identified=gaps)
  - yield: data: {"done": true}\n\n
  - if ServiceUnavailableError: yield data: {"error": "llm_unavailable", "message": "Ollama is not running. Start it with: ollama serve"}\n\n

complete_session(session_id, session) -> dict
  - set status='complete'
  - collect all gaps_identified lists from tutor turns; flatten and deduplicate
  - if gaps exist: call FlashcardService.generate_from_gaps(gaps, document_id, session)
  - set flashcard source='feynman', flashcard_type='concept_explanation' on each created card
  - call ObjectiveTrackerService.update_coverage(document_id) as fire-and-forget
  - return {"gap_count": N, "flashcard_ids": [...]}

list_sessions(document_id, session) -> list[dict]
  - query all sessions for document_id ordered by created_at desc
  - for each session, count total gaps from all tutor turns
  - return list with gap_count and status
```

**Gap parsing** (pure function):

```python
def _parse_gaps(raw: str) -> list[str]:
    """Extract the gaps JSON list from the end of a tutor response.

    Looks for a line starting with 'gaps:' and parses the JSON array on that line.
    Returns [] if no gaps block is found or parsing fails.
    """
    import json, re  # noqa: PLC0415
    m = re.search(r"gaps:\s*(\[.*?\])", raw, re.DOTALL)
    if not m:
        return []
    try:
        result = json.loads(m.group(1))
        if isinstance(result, list):
            return [str(g) for g in result if g]
    except (json.JSONDecodeError, ValueError):
        pass
    return []
```

**FlashcardModel augmentation for feynman source**: `generate_from_gaps` already accepts `source` and `deck` implicitly through the model constructor. The Feynman service must set `source='feynman'` and `flashcard_type='concept_explanation'` on each created card. Since `generate_from_gaps()` in `FlashcardService` hardcodes `source='gap'`, the Feynman service needs its own method or must post-process the returned cards by updating the rows directly in the session before commit.

**Implementation choice**: Add a new method `generate_from_feynman_gaps(gaps, document_id, session)` to `FlashcardService` that is identical to `generate_from_gaps` but sets `source='feynman'`, `deck='feynman'`, and `flashcard_type='concept_explanation'`. This avoids modifying existing `generate_from_gaps` behavior (which existing tests cover).

**Endpoints**:

```

  Request: FeynmanSessionCreateRequest(document_id, section_id, concept)
```

## Risk Flags

- Risk: `generate_from_gaps()` hardcodes `source='gap'` -- mitigation: add separate `generate_from_feynman_gaps()` method rather than adding a parameter to the existing method, which would require updating the existing test that asserts `source='gap'`.

- Risk: SSE streaming in `stream_turn()` accumulates the full response before persisting the turn row -- this means if the client disconnects mid-stream, the turn is not saved. Mitigation: persist the turn row with the accumulated content in a finally block after the stream loop completes; if the generator is cancelled the turn row write may be skipped, which is acceptable (the session stays active and the user can re-send the message).

- Risk: `_parse_gaps()` must strip the `gaps: [...]` line from the displayed tutor message so users do not see raw JSON. Mitigation: in `stream_turn()`, strip everything from the last `gaps:` occurrence (using rfind) before yielding the `done` event's `answer` field, consistent with the pattern used in `_split_response()` in `qa.py`.

- Risk: Circular import between `feynman_service.py` and `flashcard.py` -- both are in `app/services/`. Mitigation: use a lazy import inside the method body: `from app.services.flashcard import get_flashcard_service  # noqa: PLC0415`.

- Risk: `ObjectiveTrackerService.update_coverage()` opens its own session (it uses `get_session_factory()()`). Calling it inside `complete_session()` which already has a session open is safe -- each call gets an independent session. Mitigation: call it after the main session commits, using `asyncio.create_task()` with a strong reference (per patterns.md background task pattern).

- Risk: Section summary may not exist for many documents (only tech_book/tech_article get section summaries). Mitigation: fall back to `SectionModel.preview[:2000]` when `SectionSummaryModel` has no row for this section_id. If no preview either, use only the concept name in the system prompt.

- Risk: Frontend TypeScript -- `Brain` icon must be imported from `lucide-react`. Verify the version in `package.json` includes this icon before using it. If absent, use `BookOpen` or `GraduationCap` as fallback (both exist in lucide-react 0.263+).

- Risk: ObjectiveTrackerService coverage threshold (avg stability >= 10 days) is FSRS-based and requires actual review events. The Feynman completion path should call `update_coverage()` so the UI reflects the new feynman flashcards, but newly created cards start with stability=0 and will not immediately trigger coverage. This is correct behavior -- the user must review the generated cards to achieve coverage. No mitigation needed; this is by design.
