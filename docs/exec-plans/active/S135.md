# S135 — Tech Entity Types: LIBRARY, DESIGN_PATTERN, ALGORITHM, DATA_STRUCTURE, PROTOCOL with Version-Aware Merge

## Goal

Extend GLiNER entity extraction with 6 tech-specific label types, add a tech noise filter, version-aware merge, code-block confidence boost, 5 new Kuzu edge types (IMPLEMENTS, EXTENDS, USES, REPLACES, DEPENDS_ON, VERSION_OF), relationship extraction from co-occurrence + pattern matching, and a new `GET /graph/entities/{doc_id}?type=LIBRARY` endpoint; update the Viz tab to render the new types with distinct colors.

- Types (no new type file needed; new edge-type constants are strings in graph.py)
- Service (ner.py, entity_disambiguator.py, graph.py)
- Runtime (ingestion.py — entity_extract_node gains relationship extraction step)
- API (routers/graph.py — new entities endpoint)

### Step 3 — [Service] Add code-block confidence boost in ner.py

Change `extract()` signature: add `chunk_has_code_map: dict[str, bool] | None = None` parameter. When provided, entities in a `has_code=True` chunk get a +0.1 boost applied to their score before the threshold check.

The current flow calls `model.batch_predict_entities(texts, active_types, threshold=threshold)`. GLiNER returns entity dicts without a score field exposed; the threshold is applied internally. To implement the boost correctly without hacking GLiNER internals, use a lower threshold for code-containing chunks:

Implementation approach:
- Split batch into two groups: `code_chunks` (has_code=True) and `prose_chunks`
- For code_chunks: call `batch_predict_entities` with `threshold=0.55` (so 0.55 + 0.10 effective = 0.65 minimum equivalent)
- For prose_chunks: call `batch_predict_entities` with `threshold=0.65` (existing)
- Merge results preserving chunk order

Update `entity_extract_node` in `ingestion.py` to build a `chunk_has_code_map` from `ChunkModel.has_code` field and pass it to `extractor.extract()`.

AC5: `has_code=True` chunk with entity at score 0.58 -> code batch uses threshold 0.55 -> passes. Prose chunk at 0.58 -> prose batch uses threshold 0.65 -> rejected.

### Step 4 — [Service] Version-aware merge in entity_disambiguator.py

Add pure function:
```python
_VERSION_PATTERN = re.compile(r'\b(\d+)(?:\.\d+)*\b')

def _extract_version_qualifier(name: str) -> tuple[str, str | None]:
    """Split 'Python 3.13' into base='Python 3', version='3.13'.

    Returns (base_name, version_string | None).
    Base name retains only the major version number.
    'numpy 1.26' -> ('numpy 1', '1.26')
    'Python 3.13' -> ('Python 3', '3.13')
    'React 18' -> ('React 18', None)  -- single-part version stays
    'numpy' -> ('numpy', None)  -- no version
    """
```

Extend `find_canonical()` with a LIBRARY-specific pre-pass: if `entity_type == "LIBRARY"`, call `_extract_version_qualifier` on both `name` and each `existing`. Versioned variants with the same major version are NOT merged (AC4: "Python 3.11" and "Python 3.13" stay separate). They both link to their shared base via a `VERSION_OF` edge written in `entity_extract_node`.

The `canonicalize_batch()` function returns `(canonical, entity_type, original)` triples. Extend it to also return `version_base` info: for LIBRARY entities, if a version qualifier was detected, record `(canonical, entity_type, original, version_base)`. This requires changing the return type.

Decision: To avoid a breaking API change on `canonicalize_batch`, add an optional out parameter or return a parallel list of version bases. Use the parallel list approach: `canonicalize_batch()` gains an optional `return_version_info: bool = False` kwarg that when True returns `list[tuple[str, str, str, str | None]]` (adds version_base as 4th element). Default behavior unchanged (backward compatible).

### Step 6 — [Service] New tech_relation_extractor.py

Create `backend/app/services/tech_relation_extractor.py` as a pure function module:

```python
"""Tech relationship extractor: pattern-match co-occurrence + verb patterns.

Pure functions only — no I/O, no database reads.
"""

import re

# Verb patterns that signal a tech relationship
_IMPLEMENTS_PATTERN = re.compile(
    r'\b(\w+)\s+implements\s+(\w+)', re.I
)
_EXTENDS_PATTERN = re.compile(
    r'\b(\w+)\s+extends\s+(\w+)', re.I
)
_IMPORT_PATTERN = re.compile(
    r'\bimport\s+(\w+)|\bfrom\s+(\w+)\s+import', re.I
)
_USES_PATTERN = re.compile(
    r'\b(\w+)\s+uses\s+(\w+)', re.I
)
_DEPENDS_PATTERN = re.compile(
    r'\b(\w+)\s+requires\s+(\w+)|\b(\w+)\s+depends\s+on\s+(\w+)', re.I
)

def extract_tech_relations(
    chunks: list[dict],
    known_names: set[str],
) -> list[tuple[str, str, str]]:
    """Return (name_a, name_b, relation_label) triples from chunk texts.

    Only returns pairs where BOTH names are in known_names (confirmed by GLiNER).
    relation_label is one of: IMPLEMENTS, EXTENDS, USES, REPLACES, DEPENDS_ON.
    """
```

The `import X` pattern is handled specially: it creates a USES edge between the importing entity (the document's primary library, if extractable) and the imported library. Simpler approach: `import numpy` creates a self-referential signal confirming `numpy` as LIBRARY; it does NOT create an edge between two entities unless the chunk also contains another known entity. When `import X` is the primary signal with a second known entity in the same chunk, create `second_entity USES X`.

class EntityItem(BaseModel):
    id: str
    name: str

    frequency: int

class EntityListResponse(BaseModel):

## Risk Flags

- Risk: `add_tech_relation` with dynamic relation label — Kuzu does NOT support parameterized rel-type names in Cypher. The method must use explicit `if rel_label == "IMPLEMENTS": self._conn.execute("... CREATE (a)-[:IMPLEMENTS {...}]->(b)", ...)` branches. Five branches, one per label. This is verbose but safe. Mitigation: validate `rel_label` against a frozenset at entry and raise ValueError for unknown labels.

- Risk: Route shadowing `/graph/entities/{document_id}` vs `/{document_id}` — FastAPI evaluates routes in declaration order. The new endpoint must be declared BEFORE the existing `/{document_id}` route. The existing file already has this pattern documented (see the `learning-path` comment). Add a similar comment to the new route.

- Risk: GLiNER model was trained with specific label vocabulary — adding "LIBRARY", "DESIGN_PATTERN" etc. works because GLiNER is zero-shot (labels are embedded as natural language). However, precision may be lower than for standard NLP entity types. The `_TECH_NOISE_LIBRARY` blocklist and `has_code` boost mitigate this. Mitigation: acceptance tests use mocked model so GLiNER quality is not tested by unit tests; integration test uses real model but only asserts `>= 1` entity.

- Risk: Version-aware merge changes `canonicalize_batch` return type — adding optional 4th element to tuples or a `return_version_info` kwarg risks breaking the call site in `entity_extract_node`. Mitigation: use a separate helper that returns version info independently, called only inside `entity_extract_node` after `canonicalize_batch`. Keep `canonicalize_batch` signature unchanged.

- Risk: `get_graph_for_document` returning tech edges doubles query count — currently calls `_get_co_occurrence_edges()` once. Adding 5 more edge-type queries could slow down the Viz for large documents. Mitigation: query all 5 tech edge types in a single UNION query or run them in a loop with early exit if `entity_ids` is empty (already guarded).

- Risk: `extract_tech_relations` pattern false positives — English sentences like "Python uses X" may match a verb pattern when `python` is not an entity. Mitigation: the function only creates edges when BOTH names appear in `known_names` (confirmed by GLiNER). False positives from pattern matching cannot produce an edge unless GLiNER also confirmed both entities.

- Risk: `has_code` field is only set for `tech_book` and `tech_article` content types — `chunk_has_code_map` will be all-False for `book`, `code`, etc. The code batch threshold logic must gracefully handle an empty code group. Mitigation: if no code chunks exist, skip the split and use the single prose threshold call (existing behavior preserved).

- Risk: Kuzu schema migration — existing databases do not have the 6 new edge tables. `_create_schema()` uses `CREATE REL TABLE IF NOT EXISTS` so this is forward-safe. Existing deployments pick up the new tables on next startup. No migration needed.

- Risk: Frontend TypeScript — adding new string values to `ALL_ENTITY_TYPES as const` and `TYPE_COLORS` requires the `EntityType` union type to match. Since `EntityType = (typeof ALL_ENTITY_TYPES)[number]`, adding to the array automatically extends the type. `tsc --noEmit` will catch any missed color entries.
