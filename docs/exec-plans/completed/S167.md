# Exec Plan: S167 -- Tag co-occurrence graph in Viz tab: Sigma.js tag network mode with cross-tab note filter

**Phase:** V3 Phase 1 - Note Organization and Knowledge Graph
**Priority:** 7
**Status:** in-progress

## Goal

This story adds a "Tags" graph mode to the Viz tab. The mode renders a co-occurrence network where nodes are canonical tags (from CanonicalTagModel) and edges connect pairs of tags that appear together on the same note (from NoteTagIndexModel). TagGraphService computes the graph with a 200-node / 500-edge cap; GET /tags/graph serves the result with an in-memory cache that is invalidated whenever _sync_tag_index writes tag data. Clicking a tag node in the Sigma canvas dispatches the luminary:navigate DOM event, which App.tsx listens to by calling react-router's navigate("/notes") and setActiveTag in Zustand, so the Notes tab applies the tag as a filter on mount.

## Implementation Notes

- The luminary:navigate event listener belongs in App.tsx AppShell (not in Viz.tsx) because cross-tab navigation must work from any future page that dispatches the event, not just Viz
- Use `useNavigate` from react-router-dom (already imported in App.tsx / Notes.tsx); do NOT dispatch a second custom event for the tab switch -- call navigate("/notes") directly
- The "tags" viewMode does not need document scope or entity type filters -- hide the document picker and entity filter panel when viewMode === "tags"
- ForceAtlas2 should run synchronously (forceAtlas2.assign) with ~50 iterations before Sigma mounts; avoid requestAnimationFrame loop which can block UI
- Cache TTL: 60 seconds is sufficient; the invalidation on write means stale-cache reads only happen if a note is written externally (e.g. via API directly); document this in the service docstring
- Static route ordering: GET /tags/graph must be registered before GET /tags/{tag_id} in tags.py -- confirmed from patterns.md FastAPI static-path ordering rule
