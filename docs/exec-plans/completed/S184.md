# Exec Plan: S184 -- Flashcard search and unified browse replaces deck-first navigation

**Phase:** V3 Phase 4 - Learner Experience Refinement
**Priority:** 24
**Status:** complete

## Goal

S184 replaces the Study tab's deck-first navigation (a document picker followed by per-deck card list) with a unified search-and-filter interface. The backend gains a `flashcards_fts` FTS5 virtual table and a new `GET /flashcards/search` endpoint that accepts FTS query, document, collection, tag, bloom level range, FSRS state, and flashcard type as optional AND-combined filters. The frontend Study tab is rearchitected to show a search bar and filter chip row as the primary content entry point; the `DeckItem` panel from S169 is removed. The deck field is preserved in the model for backward compatibility but is no longer surfaced in the UI.

   ```python
   async def _sync_flashcard_fts(card: FlashcardModel, session: AsyncSession) -> None:
       await session.execute(
           text("INSERT OR REPLACE INTO flashcards_fts(flashcard_id, question, answer) VALUES (:fid, :q, :a)"),
           {"fid": card.id, "q": card.question, "a": card.answer}
       )
   ```

4. **Backend: service -- `FlashcardService.search()`** -- Implement search method with signature:
   ```python
   async def search(
       self,
       session: AsyncSession,
       query: str | None,
       document_id: str | None,
       collection_id: str | None,
       tag: str | None,
       bloom_level_min: int | None,
       bloom_level_max: int | None,
       fsrs_state: str | None,
       flashcard_type: str | None,
       page: int,
       page_size: int,
   ) -> tuple[list[FlashcardModel], int]:
   ```
   Logic:
   - Build base `select(FlashcardModel)` with filters applied as `.where()` clauses
   - If `query`: join against `flashcards_fts` shadow table -- `WHERE flashcards_fts MATCH :q AND flashcard_id = FlashcardModel.id` using a subquery: `FlashcardModel.id.in_(select(text("flashcard_id")).select_from(text("flashcards_fts")).where(text("flashcards_fts MATCH :q")))`
   - If `document_id`: `FlashcardModel.document_id == document_id`
   - If `collection_id`: use `NoteCollectionMemberModel` join -- `FlashcardModel.note_id.in_(select(NoteCollectionMemberModel.note_id).where(NoteCollectionMemberModel.collection_id == collection_id))`
   - If `tag`: use `NoteTagIndexModel` -- `FlashcardModel.note_id.in_(select(NoteTagIndexModel.note_id).where(or_(NoteTagIndexModel.tag_full == tag, NoteTagIndexModel.tag_full.like(tag + "/%"))))`
   - If `bloom_level_min`: `FlashcardModel.bloom_level >= bloom_level_min`
   - If `bloom_level_max`: `FlashcardModel.bloom_level <= bloom_level_max`
   - If `fsrs_state`: `FlashcardModel.fsrs_state == fsrs_state`
   - If `flashcard_type`: `FlashcardModel.flashcard_type == flashcard_type`
   - All filters combine with AND (default SQLAlchemy `.where()` chaining)
   - Return `(cards, total_count)` using count subquery

## Cross-Story Risk

- **I-4 (FTS5 UNINDEXED column reliability):** `flashcard_id` is UNINDEXED in `flashcards_fts`. The search query does NOT filter by `flashcard_id` equality (it uses MATCH on the indexed question/answer columns) so I-4 does not affect search. For FTS DELETE, must use shadow content table approach: query `flashcards_fts_content` for rowid where `c2 = :flashcard_id`, then `DELETE FROM flashcards_fts WHERE rowid = :rowid`. This is explicitly required.
- **I-10 (loading/error/empty states):** Frontend search results panel requires all three states. Plan explicitly calls them out in step 10.
- **I-13 (quality gate order):** ruff -> pytest -> tsc. Steps 8 and 12 enforce this order.
- **S185 dependency:** S185 "depends on this (search/filter is the primary content area)" per story description. The `GET /flashcards/search` endpoint shape and the filter chip row must be stable before S185 begins. No S185 code exists yet -- no regression risk at this stage.
- **S169 replacement:** The "All Decks" DeckItem panel from S169 is being removed (per story description: "S169 deck list UI is replaced"). The `/flashcards/decks` endpoint itself is NOT removed (backward compat) but the UI panel that consumed it is gone. If any other component references `fetchDecks()` or `DeckItem`, those must be cleaned up -- use Grep to verify before removing.
- **FastAPI route ordering (patterns.md -- dynamic path parameter route ordering):** `GET /search` must be registered BEFORE `GET /{document_id}`. The existing comment block in `flashcards.py` already documents this constraint for `/audit`, `/cloze`, `/health`, `/decks`. The new `/search` route joins this group.
- **Backward compat:** Existing `GET /{document_id}` (list cards by doc) and `GET /decks` remain functional. The Study tab removes its consumption of `/decks` but the endpoint stays for any external caller.
