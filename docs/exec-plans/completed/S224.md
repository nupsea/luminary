# Exec Plan: S224 -- Entity injection at index time -- chunk vocabulary bridge

**Phase:** V3 Phase 8 - Evaluation and Correctness
**Priority:** 52
**Status:** in-progress

## Goal

Inject canonical entity names (derived from the existing GLiNER + EntityDisambiguator pipeline)
into each chunk's stored text so that FTS5 BM25 and LanceDB vector search can match on
surface-form variants. A question referencing "Weena" will find the chunk "her name was Weena"
because "Weena" is now part of the chunk's indexed text. This bridges the vocabulary gap that
caused ~50% of book corpus HR@5 failures in S212 iterations 2-7. No new ML inference is
introduced; this reuses per-chunk entities already extracted during graph building.

## Storage Decision: Option (b) -- sibling `entities_text` column

The story offers two options:
- (a) Append `\n\n[Entities: ...]` directly to `chunk.text` (mutates display content)
- (b) Store entities in a separate `entities_text` column; concatenate with `text` only for
  embedding and FTS indexing; raw `text` is untouched for display

**Decision: option (b)**, per story guidance. This preserves display fidelity and makes the
entity tail a first-class, inspectable column rather than a text appendage that could
accumulate duplicates on re-runs.

## Current State (discovered by reading source)

`entity_extract_node` in `ingestion.py` (lines 1316-1330) already contains a partial entity
injection that appends to `chunk["text"]` in memory AND updates `ChunkModel.text` in the DB.
This is option (a) and does NOT satisfy ACs: no cap at 12, no capitalization, no alphabetical
sort, no idempotency guard, and it mutates the display text.

Pipeline order (confirmed from `_build_graph()`):
```
chunk -> entity_extract -> embed -> keyword_index
```
Entity extraction fires BEFORE embedding and FTS indexing, which is the correct order. The
existing partial implementation flows in the right place; what needs changing is (1) storage
approach (option b), (2) tail format, (3) idempotency, and (4) reindex CLI.

The LanceDB chunk table is `chunk_vectors_v3` (not `chunk_vectors` -- memory entry is stale).

3. **Update `db_init.py`** -- two changes:
   a. Add `"ALTER TABLE chunks ADD COLUMN entities_text TEXT"` to the idempotent list.
   b. The `chunks_fts` virtual table already exists in production DBs; FTS rebuild is handled

4. **Implement `build_entity_tail(canonical_names: set[str]) -> str`** -- pure function,
   suitable for unit testing. Rules: dedupe (set), sort alphabetically (case-insensitive),
   capitalize each label, cap at 12 entries, format as `"[Entities: A, B, C]"`. Place in
   `ingestion.py` as a module-level helper (or extract to a small helper in `services/`).

5. **Replace injection block in `entity_extract_node`** (lines 1316-1330):
   - Build `entities_text` string per chunk via `build_entity_tail`.
   - Update `ChunkModel.entities_text` in DB (single session, Semaphore-safe per I-1).
   - Set `chunk["entities_text"]` on the in-memory dict for downstream nodes.
   - Do NOT modify `chunk["text"]` (preserves option b).
   - Idempotency: overwrite `ChunkModel.entities_text` unconditionally (not append).

6. **Update `embed_node`** -- concatenate for embedding:
   ```python
   texts = [
       c["text"] + ("\n" + c["entities_text"] if c.get("entities_text") else "")
       for c in chunks
   ]
   ```
   The `lancedb_rows` `"text"` field should also store the concatenated form so vector
   search returns the enriched text.

7. **Update `keyword_index_node`** -- change the INSERT SQL to:
   ```sql
   INSERT INTO chunks_fts(rowid, text, chunk_id, document_id)
   SELECT rowid,
          text || CASE WHEN entities_text IS NOT NULL AND entities_text != ''
                       THEN ' ' || entities_text ELSE '' END,
          id, document_id
   FROM chunks WHERE document_id = :doc_id
   ```
   This keeps `chunks.text` clean and computes the indexed text at FTS insert time.

8. **Create `backend/app/scripts/__init__.py`** (empty).

9. **Implement `backend/app/scripts/reindex_entities.py`**:
   - Parse args: `--document-id <id>` or `--all` (mutually exclusive).
   - Load all chunks for the given document(s) from SQLite.
   - For each chunk, extract entities from the Kuzu graph
     (use `graph.get_entities_for_chunk(chunk_id)` -- or query Kuzu directly for entities
     mentioning the document and appearing in a chunk; see note below on Kuzu query).
   - Rebuild `entities_text` via `build_entity_tail`.
   - Update `ChunkModel.entities_text` in SQLite.
   - Re-embed the chunk (concatenated text) and upsert to `chunk_vectors_v3` via
     `asyncio.to_thread(lancedb_svc.upsert_chunks, [row])` per I-2.
   - Update FTS5: delete existing row (`DELETE FROM chunks_fts WHERE rowid = :rowid`)
     then insert new row with concatenated text (rowid-based per I-4 pattern).
   - Use a single AsyncSession (not asyncio.gather) per I-1.
   - Structured log per document: `{doc_id, chunks_updated, entities_injected_total,
     p95_entities_per_chunk}`.

   **Kuzu entity lookup for reindex CLI**: The existing `chunk_to_entities` dict is built
   in-memory during ingest. For the reindex CLI we need a different path. Options:
   a. Re-run NER on the chunk text and re-disambiguate (slow, re-runs GLiNER).
   b. Query Kuzu for entities that MENTION the document, then look up which chunks they
      came from via the `ner.py` `extract()` return value stored in `entities` dict.
   c. Query Kuzu for Entity nodes with a MENTIONED_IN edge to the document, then use
      the entity name directly (no per-chunk granularity, but sufficient for vocabulary
      bridging at document level, injected into all chunks).

   **Recommended approach for reindex CLI:** Re-run NER on chunk texts using the existing
   `extractor.extract(chunks, content_type)` call (same as `entity_extract_node`), then
   apply `canonicalize_batch` and `build_entity_tail`. This avoids Kuzu query complexity and
   produces the same entity tail as a fresh ingest. Wrap in `asyncio.to_thread` since GLiNER
   is CPU-bound. Cap chunk count at NER_CHUNK_LIMIT=5000 as in the ingest pipeline.

## Cross-Story Risk

**S212 (retrieval baseline):** This story is the primary unblock for S212 closure. If entity
injection does not lift TM HR@5 above 0.50 or Odyssey above 0.30, S212 will remain open.
Invariants I-1, I-2, I-4 all apply:
- I-1: reindex CLI uses a single AsyncSession (never asyncio.gather with shared session)
- I-2: all LanceDB upsert calls in reindex CLI wrapped in `asyncio.to_thread`
- I-4: FTS5 row deletion uses rowid-based `DELETE FROM chunks_fts WHERE rowid = :rowid`
  (not UNINDEXED column equality filter)

**S225 (graph-augmented query expansion):** S225 adds a query-time graph expansion lever.
S224 is index-time only. Both stories are orthogonal and can stack. Do not conflate.

**Existing entity injection code (ingestion.py lines 1316-1330):** The partial option-(a)
injection already in `entity_extract_node` will be replaced. Any production database that was
ingested with the old code will have entity text appended directly to `chunks.text`. The reindex
CLI is the migration path: running `--all` replaces the old in-text injection with the new
`entities_text` column approach. Document this in the CLI's `--help` output.

**FTS5 `chunks_fts` schema:** The existing virtual table has columns `(text, chunk_id UNINDEXED,
document_id UNINDEXED)`. The entity tail is concatenated into the single `text` column at
insert time; no schema change to the virtual table is needed, which avoids a destructive drop.
